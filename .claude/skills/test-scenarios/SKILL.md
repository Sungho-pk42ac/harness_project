# Test Scenarios

GitHub 이슈 하나를 입력받아, **① 시그니처(계약)를 확정하고 → ② 그 시그니처를 기반으로 테스트 시나리오를 도출**합니다. 산출물은 `docs/features/<기능>/issue-{N}.md` 한 파일(상단=시그니처, 하단=시나리오)이며, 이후 **TDD로 테스트 코드를 작성하기 직전 단계**까지만 책임집니다.

> **`<기능>` 결정 규칙(하드):** 경로의 `<기능>`은 추측하지 않습니다. 이슈의 라벨/마일스톤 또는 작업 브랜치명 `feature/<slug>`의 **`<slug>`** 로 정합니다(`gh issue view $ARGUMENTS`로 라벨 확인). 정하지 못하면 멈추고 사용자에게 물어봅니다. 이 프로젝트는 `tag`로 시작했지만 이후 `login`·`pin`·`trash`·`sort`·`search`·`markdown-preview`·`duplicate`·`export` 등으로 확장됐으므로 **`tag`를 기본값으로 가정하지 마십시오.**

왜 시그니처가 먼저인가: TDD에서 `expect(fn(...)).toThrow()` 같은 테스트는 함수 이름·파라미터·반환·에러 조건이 정해져 있어야 쓸 수 있습니다. 시나리오를 시그니처 **뒤에** 도출하면 `should...when...` 한 줄이 곧 `it(...)` 한 줄로 떨어져 코드 전환이 매끄럽습니다.

## 하드 규칙 (반드시 지킬 것)

- **구현 코드 작성 금지.** 시그니처는 타입·계약만 정합니다(함수 본문 없음).
- **테스트 코드 작성 금지.** 시나리오는 `should...when...` 자연어 문장까지만. `describe/it/expect`는 다음(TDD) 단계의 몫입니다.
- **기존 코드 패턴을 따를 것.** `src/api/notes.ts`(동사+명사·`async`+`throw`), `src/context/NotesContext.tsx`(변경함수 시그니처), 컴포넌트 `XxxProps` 인터페이스 패턴에 맞춥니다.
- **각 승인 게이트 전에 멈출 것.** 사용자 승인 없이 다음 단계로 넘어가지 않습니다.

## 입력

- 이슈 번호는 **`$ARGUMENTS`** 로 전달됩니다 (예: `/test-scenarios 1`).
- 자연어로 불러도(예: "이슈 6 시나리오") 번호를 추출해 같은 절차를 따릅니다. 번호를 특정 못 하면 추측하지 말고 물어봅니다.

## 절차

### ① 시그니처 확정

다음을 **모두 참고**해 이 이슈가 추가/변경할 인터페이스를 정합니다.

- `gh issue view $ARGUMENTS`로 읽은 **이슈 본문(범위·AC)**
- **`docs/features/<기능>/PRD.md`** (요구사항·데이터 모델). 관련 ADR(`docs/features/<기능>/adr/`)이 있으면 그 결정도 따릅니다.
- **코드베이스**: `src/api/notes.ts`, `src/context/NotesContext.tsx`, 해당 컴포넌트의 Props 패턴.

확정할 항목:

- **함수 시그니처** — 이름, 파라미터 타입, 반환 타입 (예: `addNote(title: string, content: string, tags: string[]): Promise<void>`).
- **에러 케이스** — 어떤 상황에서 무엇을 throw/거부하는지 (예: `createNote` — `res.ok`가 false면 `Error('Failed to create note')`).
- **컴포넌트 Props 타입** — 새/변경되는 `XxxProps`의 필드와 콜백 시그니처.

> 다른 이슈가 책임지는 함수·검증은 여기서 만들지 않습니다(이슈 범위·의존을 근거로 판단).

### ② 시그니처 승인

확정한 시그니처를 사용자에게 보여주고 **검토·승인**을 받습니다. **승인 전에는 ③으로 넘어가지 않습니다.**

### ③ 시그니처 기록

승인된 시그니처를 **`docs/features/<기능>/issue-{N}.md` 상단**에 기록합니다(파일 없으면 생성). 코드 블록으로 타입을 적고, 에러 케이스·Props를 함께 둡니다.

### ④ 시나리오 도출

확정 시그니처를 기준으로 테스트 시나리오를 도출합니다.

- **분류**: `정상` / `경계` / `예외`
- **형식**: `[정상/경계/예외] 함수명 — should [기대동작] when [조건]`
  - 예: `[예외] addNote — should throw when 서버가 실패 응답을 준다`
  - 예: `[경계] addNote — should 빈 배열로 저장 when tags가 비어 있다`
- 함수/계약별로 묶어 적습니다. **테스트 코드는 쓰지 않습니다.**

### ⑤ 시나리오 기록

도출한 시나리오를 **같은 파일 `issue-{N}.md` 하단**에 추가합니다.

### ⑥ AC 대조

`gh issue view $ARGUMENTS`로 이슈의 **AC 목록**을 다시 읽어 시나리오와 대조합니다.

- AC 항목마다 대응하는 시나리오가 최소 1개 있는지 확인합니다.
- 빠진 AC가 있으면 시나리오를 **보강**해 모든 AC가 커버되게 합니다.

### ⑦ 시나리오 승인

완성된 시나리오를 사용자에게 보여주고 **검토·승인**을 받습니다. 승인 전까지 멈춥니다(이후 TDD 단계로 인계).

## 산출물 구조 (`docs/features/<기능>/issue-{N}.md`)

````markdown
# 이슈 #{N} — {제목}

## 확정 시그니처

\```ts
// 함수 시그니처 / 에러 케이스 / Props 타입
\```

## 테스트 시나리오

- [정상] 함수명 — should ... when ...
- [경계] 함수명 — should ... when ...
- [예외] 함수명 — should ... when ...

### AC 커버리지

- AC 시나리오 A → [정상] ...
````

> 경로의 `<기능>`은 위 **`<기능>` 결정 규칙**(이슈 라벨/`feature/<slug>`의 slug)으로 정합니다 — `tag`로 고정하지 마세요.
