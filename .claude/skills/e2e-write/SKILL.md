# E2E WRITE (PRD → Playwright)

기능의 **`PRD.md`**(목표·유스케이스·기능요구사항(FR)+수용기준(AC)·UX 흐름)를 입력받아, **사용자 여정 단위의 E2E 시나리오**를 도출하고 그것을 **실제 브라우저를 구동하는 Playwright 테스트**로 작성합니다. 목표는 단 하나 — **"사용자가 PRD가 약속한 흐름을 진짜 화면에서 끝까지 해낼 수 있는가"를 검증**하는 것입니다. 내부 함수·분기는 Vitest(`test-scenarios`→`tdd-red`)의 몫이고, 이 스킬은 **UI를 통해 들어가 서버까지 갔다 오는 한 바퀴**를 봅니다.

왜 PRD에서 출발하나: 단위 테스트는 "부품이 맞는가"를, E2E는 "조립된 제품이 사용자 손에서 동작하는가"를 봅니다. PRD의 **유스케이스(U)**와 **FR의 AC**가 곧 "사용자가 기대하는 결과"이므로, E2E 한 시나리오는 PRD 한 흐름(또는 묶음)에 1:1로 추적 가능해야 합니다. 화면 뒤 구현이 어떻든, 사용자가 보는 결과가 PRD와 같으면 통과입니다.

## 실행 순서 (절차)

### ① PRD 로드 & 여정 추출

`docs/features/<기능>/PRD.md`를 읽어 **E2E로 검증할 사용자 여정**을 추립니다.

- **유스케이스(§5 U1·U2·…)**와 **UX 흐름(§8)**을 한 여정의 골격으로, **FR+AC(§6)**를 각 단계의 기대 결과로 매핑합니다.
- **Non-Goals(§3 N1·…)와 범위 제외(§10)는 E2E를 쓰지 않습니다**(예: 필터/검색·자동완성). PRD가 약속한 것만 검증합니다.
- 각 여정을 `[여정] 사용자가 …하면 …된다 (U#, FR-#)` 형태로 적어 **PRD ID로 추적**되게 합니다. (단위 분기가 아니라 사용자에게 의미 있는 흐름 단위로 묶습니다.)

### ② Playwright 설치/구성 (없으면)

이 레포는 아직 Playwright가 없습니다. 최초 1회 셋업합니다.

- `npm i -D @playwright/test` 후 `npx playwright install`(브라우저 바이너리).
- `playwright.config.ts`: `testDir: 'e2e'`, `use.baseURL: 'http://localhost:5173'`, 그리고 **`webServer`로 앱을 자동 기동**(`command: 'npm run dev'`, `url`, `reuseExistingServer`)해 vite(5173)+json-server(3001)가 함께 뜨게 합니다.
- `package.json`에 `"e2e": "playwright test"` 스크립트를 추가합니다. (단위 테스트 `vitest`와 디렉터리를 분리: E2E는 `e2e/`, Vitest는 `src/` co-locate.)

### ③ 테스트 데이터 격리 (필수 설계)

json-server는 요청을 **실제 `db.json`에 영속화**합니다. E2E가 노트를 만들고 지우면 개발 데이터가 오염됩니다. 반드시 격리합니다.

- **별도 테스트 DB**(예: `db.e2e.json`)를 시드로 두고, E2E용 서버를 그 파일로 띄우거나(권장), 테스트 전후 `beforeEach`/`afterEach`에서 **생성한 노트를 정리**(API로 삭제)합니다.
- 각 테스트는 **자기 데이터를 스스로 만들고 치우는** 독립 실행이어야 합니다(테스트 간 순서 의존 금지).

### ④ 시나리오 → Playwright 작성

①의 여정 한 줄을 `test()` 한 개로 옮깁니다.

- `e2e/<기능>.spec.ts`에 `test.describe('<기능>')` → `test('[여정] … (U#, FR-#)')`. 문구·ID를 ①과 똑같이 적어 1:1 추적.
- **사용자 관점 셀렉터**를 씁니다: `getByRole`·`getByLabel`·`getByPlaceholder`·`getByText`. 깨지기 쉬운 CSS/nth-child·클래스 셀렉터는 피합니다(PRD §7 접근성과도 맞물림).
- 실제 사용자 행동으로: `page.goto('/')` → 입력·클릭·키 입력(`fill`·`press('Enter')`·`click`) → `expect(locator).toBeVisible()` 등 **눈에 보이는 결과**를 단언. 자동 대기를 활용하고 고정 `waitForTimeout`은 피합니다.

### ⑤ 실행 & 보고

`npm run e2e`(`npx playwright test`)로 실제 브라우저에서 돌립니다.

- 통과/실패를 여정·PRD ID로 보고합니다. 실패가 **버그(구현이 PRD와 다름)인지, 아직 미구현(해당 FR이 다른 이슈 소관)인지** 구분합니다 — 미구현 흐름은 `test.fixme`/`test.skip`에 사유와 담당 이슈를 달아 **"빨간불의 의미"를 명확히** 합니다.
- 실패한 사용자 여정은 **구현 사이클로 인계**합니다(E2E에서 앱 코드를 직접 고치지 않음). 리포트(`playwright-report/`)는 `.gitignore`에 추가합니다.

## 제약 (하드 규칙, 반드시 지킬 것)

- **PRD에서만 도출한다.** 모든 E2E 시나리오는 PRD의 U/FR/UX로 추적 가능해야 합니다. PRD에 없는 요구를 지어내지 않고, **Non-Goals·범위 제외는 작성하지 않습니다**.
- **사용자 여정 단위로.** E2E는 "UI로 들어가 결과를 본다"입니다. 내부 함수·세세한 분기(정규화 규칙 하나하나 등)는 Vitest 소관이니 E2E로 중복 검증하지 않습니다 — 사용자에게 의미 있는 흐름만.
- **앱 코드를 고치지 않는다.** 이 스킬은 시나리오 도출 + 테스트 작성 + 실행·보고까지입니다. 실패(버그)는 메모해 구현 단계(`tdd-*`)나 새 사이클로 인계합니다.
- **실제 데이터를 오염시키지 않는다.** 개발용 `db.json`에 의존하거나 흔적을 남기지 않습니다. 테스트는 자기 데이터를 만들고 치우거나 별도 테스트 DB를 씁니다.
- **셀렉터는 사용자 관점으로.** role/label/text 우선. 구현 세부(클래스명·DOM 구조)에 묶인 셀렉터로 테스트를 깨지기 쉽게 만들지 않습니다.
- **실제 브라우저로 끝까지.** 단위 테스트 흉내(모킹된 컴포넌트 렌더)가 아니라, `webServer`로 띄운 실제 앱에 대고 vite↔json-server 왕복까지 포함해 검증합니다.
- **빨간불의 의미를 분명히.** 미구현으로 인한 실패와 회귀 버그로 인한 실패를 섞지 않습니다. 미구현은 `fixme`/`skip` + 사유·담당 이슈를 답니다.
- **레포 분리를 지킨다.** E2E는 `e2e/`(`*.spec.ts`), Vitest 단위/컴포넌트는 `src/` co-locate(`*.test.tsx`). 서로 섞지 않습니다.

> E2E는 "사용자가 PRD의 약속을 실제로 누리는가"를 묻는 마지막 관문입니다. 부품 검증을 반복하지 말고, 화면을 통해 들어가 결과로 말하게 하세요 — 그리고 실패하면 정확히 어떤 사용자 여정이, 왜(버그/미구현) 막혔는지로 보고하세요.
