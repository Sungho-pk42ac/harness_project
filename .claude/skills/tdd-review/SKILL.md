# TDD REVIEW (커밋 전 점검 게이트)

`tdd-refactor`를 마친 변경분(테스트 전부 초록 + 정리 완료)을 입력받아, **커밋·병합하기 직전에 타입 오류와 보안 취약점을 점검하고 결과를 분류**합니다. 목표는 단 하나 — **"이대로 커밋해도 되는가"를 근거와 함께 판정**하는 것입니다. 이 스킬은 _수정하지 않습니다_. 게이트(통과/차단)를 세우고, 차단 사유가 있으면 적절한 단계로 돌려보냅니다.

왜 커밋 직전에 게이트를 두나: 테스트(`vitest`)는 행위를 지키지만 **타입 안전성·보안**까지 보장하진 않습니다. 초록불이어도 `tsc`가 깨지거나(빌드 불가), 변경 코드에 취약점(주입·하드코딩 비밀·안전하지 않은 DOM 조작)이 섞일 수 있습니다. 커밋은 히스토리에 남는 경계선이므로, 넘기 전에 이 둘을 한 번 더 거릅니다. 단, **사전 존재 이슈(이번 변경이 만든 게 아닌 것)와 이번 변경이 유발한 이슈를 반드시 구분**해, 남의 잘못으로 내 커밋을 막지 않습니다.

## 실행 순서 (절차)

### ① 전제 확인 (출발점 점검)

`npx vitest run`으로 **지금 전부 초록불인지** 먼저 확인합니다. 빨간 게 있으면 리뷰를 시작하지 말고 `tdd-green`/`tdd-refactor`로 돌려보냅니다 — 깨진 테스트 위에서 타입·보안을 논하는 건 의미가 없습니다. 동시에 `git status`/`git diff`로 **이번에 무엇이 바뀌었는지**(점검 범위) 파악합니다.

### ② 타입 점검

프로젝트 타입체크를 실행합니다: `npx tsc -b`(또는 `npm run build`의 tsc 단계 / `npx tsc --noEmit`).

- 각 오류를 **위치(file:line)·메시지**로 수집합니다.
- 핵심은 **귀속 판정**: 이번 변경분(②의 diff)에서 비롯된 오류인지, 아니면 원래 있던 설정/코드 문제인지 가릅니다. (예: 테스트 파일의 `Cannot find name 'expect'`는 보통 변경과 무관한 `tsconfig`의 vitest 전역 타입 설정 문제 — 사전 존재 이슈일 가능성이 큼. 확신이 안 서면 `git stash` 전후 비교로 확인.)

### ③ 보안 취약점 점검

이번 변경분을 중심으로 코드 레벨 취약점과 의존성 경보를 봅니다. 이 레포에서 흔한 점검 포인트:

- **주입/렌더**: `dangerouslySetInnerHTML`·`innerHTML`·`eval`·동적 `Function` 등으로 사용자 입력이 그대로 흐르는가(XSS). (태그·노트 본문은 사용자 입력이므로 렌더 경로 확인.)
- **하드코딩 비밀**: API 키·토큰·비밀번호 리터럴이 코드에 들어갔는가.
- **안전하지 않은 네트워크/URL**: 사용자 입력이 `fetch` URL/경로에 검증 없이 끼어드는가(SSRF/경로 조작). ID를 URL에 넣는 `notes/${id}` 류는 출처·인코딩 확인.
- **의존성**: 필요 시 `npm audit`로 알려진 취약점 확인(고위험만).

> 맥락을 반영합니다: 이 프로젝트는 **로컬 JSON Server(localhost:3001) 강의용**입니다. 인증 부재·localhost 평문 같은 건 설계상 범위로 보고, 과잉 경보(false positive)를 만들지 않습니다 — 실제 악용 가능성 기준으로 판단합니다.

### ④ 결과 분류

②③에서 모은 항목을 세 등급으로 나눕니다. 각 항목에 **근거(file:line)와 귀속(이번 변경/사전 존재)**을 답니다.

- **Blocker (커밋 차단)**: 이번 변경이 유발한 타입 오류(빌드 실패), 실제 악용 가능한 보안 결함. → 커밋하면 안 됨.
- **Warning (기록 후 진행 가능)**: 사전 존재 설정 이슈, 저위험·맥락상 수용 가능한 항목, 후속 이슈로 미룰 사안.
- **Pass**: 문제 없음.

### ⑤ 판정 & 인계

분류 결과를 요약 보고합니다: 타입 `오류 N건(이번 변경 유발 M건)`, 보안 `Blocker K / Warning L`, 그리고 **최종 판정**.

- **Blocker 0** → 커밋해도 됨. `/commit`(또는 `/merge`·PR)으로 인계하고, Warning은 메모로 남겨 후속(새 이슈/RED)으로 넘깁니다.
- **Blocker ≥ 1** → 커밋 보류. 사유를 들고 적절한 단계로 돌려보냅니다(타입/동작 오류 → `tdd-green`·`tdd-refactor`, 새 행위가 필요한 보안 보강 → `tdd-red`부터).

## 제약 (하드 규칙, 반드시 지킬 것)

- **코드를 고치지 않는다.** 이 스킬은 점검·분류·인계만 합니다. 발견을 메모로 남기고, 수정은 해당 단계(green/refactor) 또는 새 사이클(red)에서 합니다 — 게이트가 곧 수리공이 되면 "커밋 전 상태"를 객관적으로 판정할 수 없습니다.
- **테스트 초록을 전제한다.** 빨간불이면 리뷰를 멈추고 앞 단계로 돌려보냅니다. 깨진 테스트 위에서의 타입·보안 판정은 무효입니다.
- **귀속을 분명히 한다.** 모든 발견을 "이번 변경 유발 / 사전 존재"로 구분합니다. 사전 존재 이슈를 Blocker로 올려 남의 잘못으로 이 커밋을 막지 않습니다(반대로, 이번 변경이 만든 결함을 사전 존재로 떠넘기지도 않습니다).
- **분류 기준을 일관되게.** Blocker는 "빌드 실패 또는 실제 악용 가능"으로 한정합니다. 스타일·취향·과잉 추정은 Warning 이하로 — 오탐으로 게이트를 무력화하지 않습니다.
- **맥락을 존중한다.** 강의용 로컬 JSON Server라는 전제를 무시한 일반론적 보안 경보(인증 없음·CORS·평문 등 설계상 범위)는 Blocker로 올리지 않습니다.
- **범위를 지킨다.** 이번 이슈 변경분이 점검의 중심입니다. 전체 레포 보안 감사로 번지지 않습니다(필요하면 별도 작업으로 제안).
- **커밋을 대신 수행하지 않는다.** 이 스킬의 산출물은 "판정"까지입니다. 실제 커밋은 사용자 확인 후 `/commit` 등 별도 단계에서 합니다.

> REVIEW는 "통과시키기"가 목적이 아니라 "통과해도 되는지 판정"이 목적입니다. 깨끗하면 자신 있게 커밋으로 보내고, 막을 사유가 있으면 근거를 들어 정확한 단계로 되돌리세요 — 모호한 "괜찮아 보임"으로 게이트를 넘기지 않습니다.
