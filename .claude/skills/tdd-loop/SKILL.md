# TDD LOOP (풀사이클 오케스트레이터)

GitHub 이슈 번호 하나를 입력받아, CLAUDE.md "새 이슈 작업 사이클"의 **7단계를 정해진 순서로** 진행합니다. 이 스킬의 역할은 단 하나 — **순서 보장과 위임**입니다. 각 단계의 실제 작업(시나리오 도출·테스트·구현·검증·정리·게이트·PR)은 **그 단계의 기존 스킬/에이전트가** 수행하며, 그 안에 있는 **인간 승인 게이트도 그대로 유지**됩니다. 컨테이너는 일을 대신하지 않고, 단계를 건너뛰거나 순서를 바꾸지 않으며, 게이트를 무력화하지 않습니다.

왜 별도 컨테이너인가: 단계마다 매번 어떤 스킬을 어떤 순서로 부를지 기억할 필요 없이, "이슈 N 풀사이클" 한 번으로 RED→…→PR까지의 **레일**을 깔아 줍니다. 단, 레일일 뿐이라 **각 정거장(게이트)에서는 반드시 멈춥니다.**

## 입력

- 이슈 번호는 **`$ARGUMENTS`** 로 전달됩니다 (예: `/tdd-loop 6`).
- 자연어로 불러도(예: "이슈 6 풀사이클") 번호를 추출해 같은 절차를 따릅니다. 번호를 특정 못 하면 추측하지 말고 물어봅니다.
- 시작 전, 해당 이슈의 산출물(`docs/features/<기능>/issue-{N}.md` 등) 또는 GitHub 이슈가 존재하는지 가볍게 확인합니다. 없으면 1단계(`test-scenarios`)부터 정상 진입합니다.

## 실행 순서 (각 단계 = 기존 스킬/에이전트 호출)

각 단계를 **그 단계의 스킬/에이전트로 호출**하고, 그 스킬이 **자기 게이트에서 멈추면 거기서 함께 멈춥니다.** 사용자가 명시적으로 승인("다음"·"진행"·"ok")하기 전에는 다음 단계로 넘어가지 않습니다. 모든 호출에 같은 이슈 번호(`$ARGUMENTS`)를 넘깁니다.

1. **시나리오** — `test-scenarios {N}` 스킬 호출 → 시그니처 확정 + 테스트 시나리오. (게이트: 시나리오 승인)
2. **RED** — `tdd-red {N}` 스킬 호출 → 실패하는 Vitest 테스트 + RED 확인. (게이트: RED 확인)
3. **GREEN** — `tdd-green {N}` 스킬 호출 → 최소 구현으로 전부 통과. (게이트: GREEN 확인)
4. **AC 검증** — `ac-fulfillment-auditor` 에이전트 호출 → 각 AC 의도가 코드에 반영됐는지 검증. (게이트: 결과 검토)
5. **REFACTOR** — `tdd-refactor {N}` 스킬 호출 → 초록불 유지한 채 구조 개선. (게이트: 변경 검토)
6. **REVIEW** — `tdd-review {N}` 스킬 호출 → 타입·보안 점검, Blocker/Warning/Pass 분류. (게이트: 판정 승인, **Blocker 0이어야 7로**)
7. **통합** — `/commit` 으로 커밋 후 `create-pr` 스킬 호출 → push + PR(base `feature/<spec>`). 머지·이슈 클로즈는 사용자 승인 후 별도. (게이트: 각 외부 액션)

> 진행 표시: 각 단계 시작 시 "[k/7] <단계> — <스킬/에이전트> 호출" 형태로 한 줄 알리고, 그 스킬의 산출/게이트 결과를 받은 뒤 다음으로 넘어갈지 사용자에게 묻습니다.

## 제약 (하드 규칙, 반드시 지킬 것)

- **순서만 보장한다 — 일을 대신하지 않는다.** 각 단계의 구현/테스트/판정은 해당 스킬·에이전트가 합니다. 이 컨테이너가 직접 코드를 짜거나 테스트를 만들거나 AC를 판정하지 않습니다(중복·우회 금지).
- **게이트를 유지한다.** 각 스킬 내부의 인간 승인 게이트를 그대로 둡니다. 한 번의 호출로 여러 단계를 몰아서 자동 진행하지 않습니다 — 매 단계 사용자의 승인을 받고 넘어갑니다.
- **순서를 바꾸거나 건너뛰지 않는다.** 1→2→…→7을 지킵니다. 앞 단계 산출물이 없으면 그 단계부터 정상 진입하고, 임의로 점프하지 않습니다.
- **실패 시 환송한다.** 어느 단계가 실패/미충족이면(예: REVIEW Blocker, RED가 안 빨갛다 등) 다음으로 진행하지 말고 **해당 스킬이 지정하는 이전 단계로 되돌립니다.**
- **같은 이슈 번호로 일관되게.** 모든 하위 호출에 `$ARGUMENTS`의 이슈 번호를 그대로 전달합니다. 도중에 대상 이슈를 바꾸지 않습니다(다른 이슈는 새 루프).
- **상태를 정직하게 보고한다.** 각 단계의 실제 결과(통과/실패·게이트 판정)를 그대로 전합니다. 하위 스킬이 멈춘 것을 "완료"로 포장하지 않습니다.
- **한 사이클 = 한 이슈.** 인접 기능/다른 슬라이스로 범위를 넓히지 않습니다(그건 각 단계 스킬의 제약이기도 함). 다른 이슈는 별도 `tdd-loop`.

> tdd-loop는 지휘자가 아니라 **선로**입니다. 연주(작업)는 각 단계의 스킬·에이전트가 하고, 정거장(게이트)마다 사람이 출발 신호를 줍니다. 컨테이너는 그저 "다음 칸이 어디인지"를 틀리지 않게 할 뿐입니다.
