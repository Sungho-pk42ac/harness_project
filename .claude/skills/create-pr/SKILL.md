# CREATE PR (push + PR 자동화)

`tdd-review`를 통과한(테스트 green·Blocker 0) **커밋된 변경**을 입력받아, 현재 feature 브랜치를 원격에 push하고 **gh CLI로 PR을 생성**합니다. 이 프로젝트의 통합 규칙(CLAUDE.md "새 이슈 작업 사이클 ⑦단계")을 그대로 따릅니다 — **base는 `feature/<spec>`**, 제목은 conventional commit, 본문에 검증 결과와 `Closes #N`. 목표는 단 하나 — **리뷰어가 바로 볼 수 있는, 추적 가능한 PR을 정확한 base로 올리는 것**입니다.

PR 생성은 외부로 나가는 행위입니다. 그래서 이 스킬은 **각 외부 액션(push, PR 생성) 전에 멈춰 사용자 승인**을 받고, **머지·이슈 클로즈는 절대 자동으로 하지 않습니다**(보고만).

## 실행 순서 (절차)

### ① 사전 점검 (게이트 확인)

PR을 올리기 전 전제를 확인합니다. 하나라도 어긋나면 멈추고 알맞은 단계로 돌려보냅니다.

- **브랜치**: 현재 브랜치가 `feature/*`인지 확인. `main`에서 직접 PR을 만들지 않습니다(브랜치부터 파라고 안내).
- **리뷰 게이트**: `tdd-review`를 통과(테스트 `npx vitest run` green, 타입/보안 Blocker 0)했는지 확인. 미통과면 `tdd-review`로 환송.
- **미커밋 변경**: `git status`로 working tree가 깨끗한지 확인. 커밋 안 된 변경이 있으면 PR에 안 들어가므로 멈추고 `/commit`으로 안내(이 스킬은 커밋하지 않음).

### ② base 결정

이 프로젝트의 기본 base는 **`feature/<spec>`**(스펙/기능 통합 브랜치)입니다. main으로 바로 보내지 않습니다.

- 현재 작업이 어느 spec 브랜치에서 갈라졌는지 확인(`git merge-base`/브랜치 네이밍)하여 base를 정합니다.
- base 브랜치가 원격에 없으면 먼저 push가 필요함을 알립니다. **main을 base로 하려면 사용자의 명시적 지시가 있을 때만** 진행합니다.

### ③ 현재 브랜치 push (승인 후)

`git push -u origin <현재브랜치>`로 원격에 올립니다(upstream 설정). 이미 추적 중이면 새 커밋만 올라갑니다. **push 전 사용자 승인**을 받습니다.

- husky/lint-staged·commitlint는 커밋 시점에 이미 통과한 상태를 전제합니다(이 스킬은 커밋을 만들지 않음).

### ④ PR 생성 (승인 후)

`gh pr create --base <base> --head <현재브랜치>`로 생성합니다. **생성 전 제목·본문을 사용자에게 보여주고 승인**을 받습니다.

- **제목**: conventional commit 형식(`feat(scope): …`, `fix: …`). commitlint 규칙상 **subject를 대문자로 시작하지 않습니다**(예: 한글/소문자 시작).
- **본문(템플릿)**:
  - `## 개요` — 무엇을/왜 (이슈·PRD 연결)
  - `## 변경` — 핵심 변경 요약(파일/계층)
  - `## 검증` — `npx vitest run` 결과, 타입(`tsc`)/보안(`tdd-review`) 판정, E2E(`npm run e2e`) 있으면 함께
  - `## 후속` — 미구현/다음 이슈로 미룬 항목
  - `Closes #N` — 머지 시 해당 이슈 자동 클로즈(이슈 번호는 `issue-{N}.md`/브랜치명에서 확인)
  - 마지막 줄: `🤖 Generated with [Claude Code](https://claude.com/claude-code)`

### ⑤ 보고 & 인계

생성된 **PR URL**과 base←head, 연결된 이슈를 보고합니다. 이어서 통합의 남은 단계를 **사용자 승인 게이트**로 안내합니다: 리뷰 → **squash merge**(커밋 1개로 압축) → 머지되면 **이슈 #N 클로즈**(`Closes`로 자동). 이 머지·클로즈는 **이 스킬이 직접 수행하지 않습니다.**

## 제약 (하드 규칙, 반드시 지킬 것)

- **외부 액션은 승인 후에만.** push와 PR 생성은 각각 사용자 승인을 받고 실행합니다. 한 번의 승인으로 머지까지 몰아가지 않습니다(CLAUDE.md ⑦단계의 인간 승인 게이트와 일치).
- **base는 `feature/<spec>` 기본.** main 직접 PR은 사용자의 명시적 지시가 있을 때만. 기본은 spec 통합 브랜치로 보냅니다.
- **리뷰 게이트 우선.** `tdd-review` 미통과(테스트 red 또는 Blocker ≥1) 상태에서 PR을 만들지 않습니다. 게이트를 건너뛰지 않습니다.
- **커밋하지 않는다.** 이 스킬은 push + PR까지입니다. 미커밋 변경이 있으면 PR을 만들지 말고 `/commit`으로 안내합니다(커밋은 commitlint·husky를 거치는 별도 단계).
- **머지·이슈 클로즈를 자동 수행하지 않는다.** 산출물은 "PR 생성"까지. squash merge와 이슈 클로즈는 리뷰 후 사용자 승인 하에 별도로.
- **제목은 commitlint를 지킨다.** conventional 형식 + subject 대문자 시작 금지(`subject-case`). 위반하면 PR 제목도 후속 머지 커밋에서 문제됩니다.
- **본문은 추적 가능하게.** 이슈(`Closes #N`)·PRD·검증 결과를 반드시 연결합니다. 빈약한 본문(제목만)으로 올리지 않습니다.
- **gh CLI·인증 전제.** `gh`로 생성합니다(origin: 이 레포의 GitHub 원격). 인증이 안 되어 있으면 `gh auth login`을 안내하고 멈춥니다.
- **자기 PR을 임의 승인·머지하지 않는다.** 리뷰는 사람이 합니다. 자동 approve/merge 금지.

> CREATE PR은 "내 변경을 올바른 base로, 리뷰어가 읽을 수 있게 내보내는" 관문입니다. 게이트를 통과한 커밋만, 정확한 base로, 추적 가능한 본문과 함께 — 그리고 머지의 방아쇠는 사람에게 남겨두세요.
