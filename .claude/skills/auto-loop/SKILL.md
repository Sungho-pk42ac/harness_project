# AUTO-LOOP (목표 → 기능 발굴 → 기획 → 구현 → 통합, 완전 자율)

막연한 목표 한 줄을 입력받아, **무엇을 만들지부터 멀티에이전트가 스스로 정하고**, CLAUDE.md "새 이슈 작업 사이클"과 `feature-planner` 기획 절차를 **사람 승인 없이 연속으로** 끝까지 돌립니다. `tdd-auto-loop`이 "주어진 이슈 1개"를 자율 구현하는 스킬이라면, AUTO-LOOP은 그 **한 단계 위** — "기능을 발굴하고, 기획하고, 그 기능의 모든 이슈를 `tdd-auto-loop`으로 구현하고, main까지 통합"하는 자율 주행입니다.

핵심 원칙 — **"사람 게이트는 빼되, 품질 게이트와 비가역 천장은 남긴다."** 단계 사이에서 승인을 기다리지 않지만(완전 자율), 각 단계의 **객관적 성공 기준**(아래 게이트)을 자동으로 확인하고 충족할 때만 다음으로 갑니다. 불충족이면 **즉시 정지(HALT)하고 보고**합니다. 깨진 것을 자율이라는 이유로 밀어붙이지 않습니다.

> **조립만 한다 — 일을 대신하지 않는다.** 기획·시나리오·테스트·구현·검증·PR·머지는 모두 프로젝트 내 기존 스킬/에이전트(`feature-planner` 절차, `test-scenarios`, `tdd-auto-loop`, `finish-pr`, `ac-fulfillment-auditor`, `design-system`)가 수행합니다. 이 스킬이 새로 더하는 것은 **① 멀티에이전트 기능 발굴**과 **② spec-fixed 인터뷰의 멀티에이전트 합의 대체**, 그리고 **③ 전체를 잇는 자율 레일**뿐입니다. 외부 플러그인(prd-mas 등)은 쓰지 않습니다.

## 입력

- 막연한 목표는 **`$ARGUMENTS`** 로 전달됩니다 (예: `/auto-loop 이 노트 앱을 더 쓸만하게`).
- 비어 있으면 **추측하지 말고**, 현재 코드베이스 상태와 열린 GitHub 이슈를 스캔해 후보 목표를 정리한 뒤 Phase 1로 진입합니다(그래도 못 정하면 한 번만 물어봅니다 — 그 외에는 멈추지 않습니다).
- 언어는 입력에서 감지하되, 감지 불가 시 한국어를 기본값으로 합니다(이 레포 컨벤션: 한국어 주석·문서).

## 실행 순서 (사람 승인 없이 자동 진행)

각 단계 진입 시 `[Phase k] <이름>` 한 줄을 로그로 남깁니다(자율이라 사후 추적이 중요).

| Phase | 단계          | 호출/주체                         | **자동 진행 조건(게이트)**                                                                                                                         |
| ----- | ------------- | --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 0     | 컨텍스트 스캔 | (이 스킬)                         | `git remote -v`·`gh auth status`로 환경 확인, 기존 `docs/features/`·열린 이슈 파악, 작업 base(`main`) 최신화. gh 미인증/리모트 없음이면 정지·보고  |
| 1     | **기능 발굴** | `Agent`(역할 4종)                 | 목표+코드베이스에서 후보 기능을 제안→비판→수렴해 **이번 런의 단일 기능 `<slug>` + 정당화** 확정. slug 충돌(이미 있는 기능)·범위 과대면 축소 재수렴 |
| 2     | **자율 기획** | `feature-planner` 절차 + `Agent`  | `spec.md`→**`spec-fixed.md`(멀티에이전트 합의로 결정)**→`PRD.md`+`adr/`→`issues.md`→`gh issue create`. 모든 결정에 근거 기록, AC↔시나리오 커버     |
| 3     | **구현 루프** | `tdd-auto-loop {N}` (이슈마다)    | 등록 이슈를 **walking skeleton부터 순서대로** 각각 풀사이클. 각 이슈가 자기 품질 게이트(RED/GREEN/AC/REVIEW Blocker 0/CI)를 통과해야 다음 이슈로   |
| 4     | **main 통합** | `finish-pr <spec PR> --base main` | 기능의 **모든 이슈가 닫힌** 뒤, `feature/<slug>`를 main으로 통합(CI 그린 게이트). 머지 천장(아래)에 걸리면 정지·인계                               |

> 게이트 **성공이면 즉시 다음으로**(묻지 않음). **불충족이면 그 자리에서 정지(HALT)** 하고 무엇이/왜 막혔는지와 권장 복구를 보고합니다.

### Phase 1 — 멀티에이전트 기능 발굴 (상세)

목표가 막연하므로 "무엇을 만들지"를 먼저 자율 결정합니다.

1. **제안(병렬)** — `Agent` 툴로 역할 에이전트를 띄워 각자 후보 기능을 1~3개씩 제안하게 합니다(아래 "역할" 참고). 각 후보는 `한 줄 가치 + 대략 범위 + 근거`.
2. **비판** — Devil's Advocate 역할이 후보들의 가정·리스크·중복(이미 구현된 기능과 겹치는지)을 점검해 거릅니다.
3. **수렴** — 아래 합의 규칙으로 **이번 런에 만들 단일 기능 1개**를 고릅니다. 기준: 사용자 가치 × 작은 수직 슬라이스로 쪼개짐 × 기존 아키텍처와 정합. slug는 `feature/<slug>` 라벨·`docs/features/<slug>/`에 쓸 kebab-case로 확정.
4. **로그** — 고른 기능과 탈락 후보(이유)를 로그로 남깁니다. **한 런 = 한 기능** — 여러 기능을 동시에 발굴/구현하지 않습니다.

### Phase 2 — 자율 기획 (인터뷰만 합의로 대체)

`feature-planner`의 절차·산출물 포맷을 **그대로** 따릅니다 — 단 하나, **② spec-fixed의 "사람에게 질문" 게이트를 멀티에이전트 합의로 대체**합니다.

- ① `docs/features/<slug>/spec.md` — 개요·요구사항·비고를 자동 작성.
- ② `docs/features/<slug>/spec-fixed.md` — 데이터모델·입력·정규화/중복/제한·표시·저장 흐름·범위·테스트·영향 파일의 **각 결정점**을 역할 에이전트들이 토론·수렴해 확정. **각 결정에 근거 한 줄**을 답니다(사람 인터뷰의 "추천안+이유"를 합의가 대신).
- ③ `docs/features/<slug>/PRD.md` + `adr/NNNN-*.md`(+`adr/README.md`) — 제품(무엇/왜)과 기술 결정(어떻게/왜 이 선택)을 분리. 중요한 기술 선택 하나당 ADR 하나.
- ④ `docs/features/<slug>/issues.md` — **수직 슬라이싱**(계층 아님). walking skeleton 먼저. 각 이슈에 `AC(Given/When/Then) ≠ DoD(체크리스트)` 분리.
- ⑤ `gh issue create` — 라벨(`feature/<slug>`, `P0/P1/P2`, `slice:*`) 없으면 `gh label create`로 만들고, 한글 본문은 `--body-file`로 등록. 등록 이슈 번호를 `issues.md`에 역링크.

## 멀티에이전트 합의 (사람 인터뷰 대체 — 이 스킬의 핵심 신규 부분)

Phase 1의 발굴과 Phase 2-②의 spec 결정에 같은 메커니즘을 씁니다. `Agent` 툴로 **프로젝트 내에서** 역할 에이전트를 띄웁니다(외부 plugin·신규 에이전트 파일 없이, 역할별 프롬프트로 일반 에이전트를 호출).

- **역할 4종**
  - **PM** — 목표·우선순위·범위·성공지표 관점. "이게 사용자 가치를 주나, 지금 할 일인가?"
  - **Tech Lead** — 제약·아키텍처 정합·NFR·구현 난이도. 이 레포 패턴(`api`→`Context`→컴포넌트 단방향, 응답기반 상태갱신, 순수 함수는 `lib`/`utils`)과의 정합을 본다.
  - **User Advocate** — 페르소나·UX·실제 사용 흐름. "사용자가 실제로 이렇게 쓰나?"
  - **Devil's Advocate** — 가정 검증·리스크·범위 축소. **항상 마지막에 발언**하고, 과대 범위·근거 없는 결정을 깎는다.
- **결정 메커니즘** — 결정점마다: 각 역할이 `추천안 + 근거` 제시 → Devil's Advocate 반박 → **수렴 규칙**으로 1안 확정:
  - 다수가 같은 안 + DA의 치명 반박 없음 → 채택.
  - 충돌 시 **기술적 사안은 Tech Lead, 제품적 사안은 PM**이 타이브레이크.
  - 확정안과 근거(누가 왜)를 `spec-fixed.md`/해당 `adr/`에 기록.
- **완전 자율 폴백** — 합의 불가/정보 부족/모순이면 **사람에게 묻지 않습니다.** 가장 보수적인 추천 기본값(기존 패턴·최소 범위)을 채택하고, 그 사실을 **Open Question으로 PRD에 기록**한 뒤 진행합니다. 멈춤은 "품질 게이트 실패" 때만입니다.

## ⛔ 비가역 천장 (환경 제약 — 정직하게)

이 하네스의 auto-mode 분류기는 가장 비가역적인 **머지(`gh pr merge`)** 를 런타임에서 **하드 차단**한 이력이 있습니다. 천장 처리는 `tdd-auto-loop`/`finish-pr`과 동일합니다:

- 머지·PR 생성·이슈 등록 등 외부/비가역 동작은 `.claude/settings.local.json` 허용 목록 또는 **bypass 권한 모드**가 있어야 무인 실행됩니다.
- **머지가 차단되면 그 자리에서 정지·보고하고, 같은 머지 명령을 절대 재시도하지 마십시오**(우회 시도로 격상되어 더 강하게 막힘). 이때 직전 단계(PR 생성)까지는 정상 완료된 상태이므로 "PR까지 끝, 머지는 권한/모드 확인 후 `finish-pr`로"라고 인계합니다.
- 권한이 풀려 있으면(현재 환경: 허용 목록에 `gh pr merge:*` 포함 + bypass) Phase 3·4의 머지까지 무인으로 진행합니다.

## 하우스키핑 (시작 시 1회, 안전 청소)

- `git fetch --prune`으로 원격에서 사라진 추적 ref 정리.
- `git branch --merged <main>`로 뽑은 **머지 완료 로컬 브랜치만** `git branch -d`(미머지는 거부 → 안전). 현재·`main`·통합(`feature/<slug>`)·`backup-*`·보호 브랜치 제외.
- 이 청소는 **머지를 하지 않습니다** — 죽은 가지만 칩니다. 현재 기능의 머지·클로즈·삭제는 Phase 3·4에서 `tdd-auto-loop`/`finish-pr`이 CI 그린 게이트 위에서 수행합니다.

## 제약 (하드 규칙, 반드시 지킬 것)

- **사람 게이트만 제거 — 품질 게이트는 유지.** 단계마다 멈춰 묻지 않되, 위 표의 객관적 성공 기준을 **반드시** 확인하고 충족할 때만 진행합니다. 확인 없이 다음으로 건너뛰지 않습니다.
- **실패 시 즉시 정지(HALT).** RED가 안 빨갛다 / GREEN이 안 초록이다 / AC 미충족 / REVIEW Blocker / 커밋·push·CI 실패 등 어느 하나라도 어긋나면 그 자리에서 멈추고 보고합니다. 깨진 채로 밀어붙이지 않습니다.
- **한 런 = 한 기능.** Phase 1에서 **단일 기능**만 고르고, 그 기능의 이슈들만 구현합니다. 도중에 다른 기능으로 범위를 넓히지 않습니다(다른 기능은 새 `/auto-loop`).
- **조립만 한다 — 일을 대신하지 않는다.** 기획/구현/테스트/판정은 프로젝트 내 스킬·에이전트가 합니다. 이 스킬이 직접 코드를 짜거나 게이트를 우회 판정하지 않습니다.
- **기존 스킬·포맷을 그대로 쓴다.** Phase 2는 `feature-planner` 절차와 `docs/features/<slug>/` 포맷을, Phase 3은 `tdd-auto-loop`을, Phase 4는 `finish-pr`을 그대로 호출합니다. 새 포맷을 발명하지 않습니다.
- **멀티에이전트는 프로젝트 내에서.** 역할 에이전트는 `Agent` 툴로 띄우고, 외부 plugin(prd-mas 등)을 끌어오지 않습니다. 합의 결과는 반드시 근거와 함께 산출물에 기록합니다.
- **자율 폴백은 "기본값+Open Question"**, 멈춤이 아니다. 결정이 안 서면 보수적 기본값을 택해 진행하고 PRD에 남깁니다. 사람에게 되묻지 않습니다(단 Phase 0의 환경 부재·Phase 4의 머지 천장은 정지 사유).
- **디자인 시스템은 UI 기능의 게이트.** 발굴된 기능이 UI를 만들/바꾸면 Phase 3에서 `design-system`을 적용하고 REVIEW에서 준수를 확인합니다(핵심 액션은 `bg-primary` 인디고, 토큰만, 임의 hex/그림자/인라인 style 금지). 회색 토큰만 쓰고 브랜드가 빠지면 Blocker.
- **머지는 CI 그린 + 권한 둘 다일 때만.** 천장에 걸리면 정지·재시도 금지(위 ⛔ 참고). 머지 성공이 이슈 클로즈의 전제.
- **모든 단계 결과를 정직하게 로그로.** 발굴한 기능·탈락 후보·합의 근거·각 이슈 게이트 결과·머지 여부를 그대로 기록합니다. 실패를 "완료"로 포장하지 않습니다.
- **언제든 중단 가능.** 사용자가 멈추라면 즉시 멈춥니다. 자율은 강제가 아니라 기본 모드일 뿐입니다.

> AUTO-LOOP은 "목표만 던지면 기능 하나를 발굴해 main까지 얹는" 무인 주행이되, **브레이크가 달린** 주행입니다. 품질 게이트·CI 그린이 초록이면 발굴→기획→구현→통합까지 멈추지 않지만, 빨강이거나 비가역 천장(머지 권한)에 닿으면 반드시 섭니다. 종착역은 "main에 통합된, 닫힌 이슈들로 이뤄진 한 기능"입니다.
