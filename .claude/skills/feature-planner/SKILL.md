# Feature Planner

거친 기능 아이디어를 **GitHub Issue로 등록된 구현 작업 묶음**까지 끌고 가는 5단계 기획 절차를 진행합니다.

## 먼저 할 일 — 권위 출처를 읽는다

작업 시작 전 **[`docs/features/feature-planning-workflow.md`](../../../docs/features/feature-planning-workflow.md)를 반드시 읽고 그대로 따르세요.** 그 문서가 이 워크플로의 단일 출처(source of truth)이고, 아래 본문은 빠르게 따르기 위한 요약입니다. 문서가 갱신되면 문서를 우선합니다.

## 진행 규칙

- **단계는 순서대로**, 각 단계 산출물이 다음 단계의 입력입니다.
- **단계 사이마다 사용자 확인을 받고** 다음으로 넘어갑니다. 한 번에 4단계를 몰아치지 마세요 — 각 단계의 결정이 뒤 단계를 좌우합니다.
- **이미 존재하는 산출물은 재작성하지 말고** 그 지점부터 이어갑니다(예: `spec.md`가 이미 있으면 ②부터).
- 산출물은 **`docs/features/<기능-slug>/`** 아래에 둡니다. 새 기능이면 폴더부터 만듭니다.
- 이 스킬은 **기획부터 GitHub Issue 등록(⑤)까지** 책임집니다. 코드 구현은 이슈 등록 후 별도로 진행합니다.

## 4단계 절차

### ① 초안 정의서 — `spec.md`

거친 요구를 **최소 형태**로 적습니다. 완벽할 필요 없습니다.

- **활동**: 기능 개요 + 핵심 요구사항 + 비고(제약)만.
- **산출물**: `<기능>/spec.md`.
- **다음으로**: "실제로 만들려니 결정 안 된 게 많다" 싶을 때 ②로.

### ② 인터뷰로 구체화 — `spec-fixed.md`

초안의 **모호함을 질문으로 없앱니다.** 이 단계가 워크플로의 핵심입니다.

- **활동**: 데이터 형태 / 입력 방식 / 정규화·중복·제한 / 표시 / 저장 흐름 / 범위 / 테스트를 질문으로 결정.
  - **쉬운 기술 질문을 한 번에 하나씩** 던집니다(AskUserQuestion 활용).
  - 각 질문에 **추천안과 그 이유**를 붙여 사용자의 결정을 돕습니다(추천을 첫 번째 옵션으로).
  - 전문용어는 풀어서 설명합니다 — 사용자가 비개발자일 수 있습니다.
- **산출물**: `<기능>/spec-fixed.md` (확정본: 데이터 모델·입력·표시·저장·범위·테스트·영향 파일).
- **다음으로**: 구현에 필요한 결정이 빠짐없이 채워졌을 때 ③으로.

### ③ 제품/기술 분리 — `PRD.md` + `adr/`

확정 내용을 **두 관점으로 나눕니다. 한 문서에 섞지 마세요.**

- **PRD** = 무엇을·왜. 배경, 목표/비목표, 요구사항(FR, 우선순위+수용기준), 성공지표, 범위, 리스크. 기획 중 진화합니다.
- **ADR** = 어떻게·왜 이 선택. **중요한 기술 결정 하나당 한 파일**. `Status / Context / Decision / Consequences / Alternatives Considered`. 한 번 Accepted 되면 불변(번복 시 새 ADR로 Superseded).
- **산출물**: `<기능>/PRD.md`, `<기능>/adr/NNNN-*.md` + `<기능>/adr/README.md`(인덱스). 서로 링크.
- **다음으로**: 요구사항과 핵심 기술 결정이 합의됐을 때 ④로.

### ④ 수직 슬라이싱 — `issues.md`

구현 작업을 **계층(수평)이 아니라 사용자 가치 단위(수직)** 로 쪼갭니다.

- 각 이슈는 데이터→API→UI를 관통해 **그 자체로 동작·관찰 가능**해야 합니다.
- **walking skeleton 먼저** — 가장 얇게 전 계층을 관통하는 happy-path 한 조각. 이후 가치를 덧붙입니다.
- 각 이슈는 `제목 / 설명 / 범위 / 인수기준(AC) / 완료정의(DoD)`.
- **AC ≠ DoD를 반드시 구분**합니다:

|      | 인수 기준 (AC)             | 완료 정의 (DoD)                 |
| ---- | -------------------------- | ------------------------------- |
| 질문 | "**올바르게 동작**하는가?" | "**정말 끝났는가?**"            |
| 범위 | 이슈별 행위                | 공통 품질·프로세스 게이트       |
| 포맷 | **Given / When / Then**    | 체크리스트(테스트·회귀·리뷰 등) |

- **산출물**: `<기능>/issues.md`.
- **다음으로**: 각 이슈가 독립 착수 가능할 만큼 명확하면 ⑤로.

### ⑤ GitHub Issue 등록 — `gh issue create`

`issues.md`의 각 수직 슬라이스를 **실제 GitHub Issue로 등록**해 추적 가능하게 만듭니다. (`issues.md`는 버전관리되는 기록으로 남기고, 등록은 그 사본을 GitHub로 올리는 것)

- **외부에 게시하는 작업이므로 먼저 사용자 확인을 받습니다.** `issues.md`가 확정된 뒤에만 진행.
- **활동**:
  1. 환경 확인: `git remote -v`로 리모트, `gh auth status`로 `gh` 로그인·`repo` 권한 확인. (gh 미인증/리모트 없음이면 사용자에게 알리고 중단)
  2. 라벨 준비: `gh label list`로 기존 라벨 확인 후, `issues.md`가 쓰는 라벨(`feature/<slug>`, `P0/P1/P2`, `slice:*` 등) 중 없는 것을 `gh label create`로 생성. (없는 라벨로 생성하면 `gh issue create`가 실패)
  3. 이슈 생성: 이슈마다 본문(설명·범위·AC·DoD)을 임시 파일로 쓰고 `gh issue create --title ... --body-file ... --label ...`. 한글 본문은 셸 이스케이프 깨짐을 피해 **`--body-file`** 을 씁니다. 본문 머리에 `docs/features/<slug>/issues.md`·PRD·의존 이슈 링크를 적어 추적성을 줍니다.
  4. 등록된 이슈 URL 목록을 사용자에게 보여주고 임시 파일을 정리합니다.
- **산출물**: GitHub Issues(저장소) + (변경 없으면) `issues.md`. 이슈 번호를 `issues.md`에 역링크로 적어두면 추적이 쉬워집니다.
- **다음으로**: 등록 완료 후 walking skeleton 이슈부터 구현 단계로 인계.

> 리모트가 없거나 `gh`가 없으면 ⑤는 건너뛰고 `issues.md`까지가 산출물입니다. 그 경우 사용자에게 "GitHub 등록은 환경이 갖춰지면 가능"하다고 안내하세요.

## 핵심 원칙

1. **모호함은 앞 단계에서 제거** — 인터뷰(②)에서 결정을 끝내고 코드로 넘어간다.
2. **제품 ↔ 기술 분리** — PRD(무엇/왜)와 ADR(어떻게/왜 이 선택)을 나눈다.
3. **수직 슬라이싱** — 계층이 아니라 사용자 가치로 쪼갠다. walking skeleton 먼저.
4. **AC ≠ DoD** — 행위는 AC(G/W/T), 품질은 DoD(체크리스트).

## 단계별 체크리스트

- [ ] ① `spec.md` — 개요·요구사항·비고를 러프하게 적었다.
- [ ] ② `spec-fixed.md` — 쉬운 질문 + 추천안으로 모든 모호함을 결정했다.
- [ ] ③ `PRD.md` — 배경·목표/비목표·FR·지표·범위·리스크를 정리했다.
- [ ] ③ `adr/` — 중요한 기술 결정마다 ADR(맥락·결정·결과·기각 대안)을 남겼다.
- [ ] ④ `issues.md` — 수직 슬라이스로 쪼개고, 각 이슈에 AC(G/W/T) + DoD를 분리했다.
- [ ] ⑤ GitHub Issue — 사용자 확인 후 `gh issue create`로 각 슬라이스를 등록했다(라벨 포함).
- [ ] 구현 시작 전, 각 이슈가 독립 착수 가능할 만큼 명확하다.

> 실제 적용 예시(태그 기능)는 `docs/features/tag/`의 `spec.md → spec-fixed.md → PRD.md → adr/ → issues.md`를 참고하세요.
