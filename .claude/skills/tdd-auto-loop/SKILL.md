# TDD AUTO-LOOP (완전 자율 풀사이클)

GitHub 이슈 번호 하나를 입력받아, CLAUDE.md "새 이슈 작업 사이클"을 **사람 승인 없이 연속으로** 진행합니다. `tdd-loop`와 동일하게 **각 단계는 그 단계의 기존 스킬/에이전트가 수행**하지만, 단계 사이의 **인간 승인 게이트를 제거**해 한 번의 호출로 시나리오→…→**머지·이슈 클로즈(그리고 spec의 마지막 이슈면 main 통합)**까지 자율 주행합니다. 단 가장 비가역적인 **머지는 "CI 그린 게이트"와 "머지 권한(bypass/allowlist)"이 둘 다 충족될 때만** 실행하고, 권한이 없으면 PR까지만 하고 멈춥니다.

핵심 원칙 — **"사람 게이트는 빼되, 품질 게이트는 남긴다."** 승인을 기다리지 않는다는 것이지 아무거나 통과시킨다는 뜻이 아닙니다. 각 단계의 **객관적 성공 기준**(RED는 실제로 빨강, GREEN은 전부 초록, AC 충족, REFACTOR 후에도 초록 유지, REVIEW Blocker 0)을 자동으로 확인하고, **충족하면 다음 단계로 자동 진행**, **불충족이면 즉시 정지(HALT)하고 보고**합니다. 깨진 것을 자율이라는 이유로 밀어붙이지 않습니다.

## 입력

- 이슈 번호는 **`$ARGUMENTS`** 로 전달됩니다 (예: `/tdd-auto-loop 7`).
- 번호를 특정 못 하면 추측하지 말고 한 번만 물어봅니다(그 외에는 멈추지 않습니다).
- 시작 전 작업 브랜치(`feature/<slug>-N-...`)가 적절한 base(`feature/<spec>`) 위에 있는지 확인하고, 없으면 만든 뒤 자율 주행을 시작합니다.

## 실행 순서 (사람 승인 없이 자동 진행)

각 단계를 해당 스킬/에이전트로 호출하고, **그 단계의 인간 승인 게이트는 자동 통과**시킵니다(아래 자동 게이트가 성공일 때만). 같은 이슈 번호(`$ARGUMENTS`)를 모든 호출에 넘깁니다. 각 단계 진입 시 `[k/9] <단계>` 한 줄을 로그로 남깁니다(자율이라 사후 추적이 중요).

| #   | 단계        | 호출                                     | **자동 진행 조건(품질 게이트)**                                                                                                                                                                                                                           |
| --- | ----------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | 시나리오    | `test-scenarios {N}`                     | `issue-{N}.md`에 시그니처+시나리오가 작성되고 AC가 모두 커버됨. **UI 이슈면 디자인 AC(브랜드 적용)도 시나리오에 포함**                                                                                                                                    |
| 2   | RED         | `tdd-red {N}`                            | 새 테스트가 **전부 실패(올바른 이유)**, 기존 테스트 회귀 없음                                                                                                                                                                                             |
| 3   | GREEN       | `tdd-green {N}` (+ UI면 `design-system`) | 대상 테스트 **전부 통과**, 회귀 없음. **UI를 만들/바꾸면 `design-system` 스킬 적용** — 핵심 액션은 `bg-primary`(인디고), 로고/라틴은 `font-display`, 색은 토큰만(임의 hex/그림자/인라인 style 금지). 필요한 토큰(예: `--color-primary`)이 없으면 정의부터 |
| 4   | AC 검증     | `ac-fulfillment-auditor`                 | 모든 AC가 **충족/부분충족**이고 **미충족(blocker) 0**                                                                                                                                                                                                     |
| 5   | REFACTOR    | `tdd-refactor {N}`                       | 변경 후에도 **전부 초록 유지**(동작 불변)                                                                                                                                                                                                                 |
| 6   | REVIEW      | `tdd-review {N}`                         | 타입/보안 **Blocker 0** + **디자인 시스템 준수**(신규/변경 UI가 브랜드 토큰을 쓰고 임의값이 없는가) Blocker 0. (Warning은 기록 후 진행)                                                                                                                   |
| 7   | PR          | `/commit` → `create-pr`                  | 커밋(husky/commitlint 통과) → push → **PR 생성**까지                                                                                                                                                                                                      |
| 8   | 머지·클로즈 | `finish-pr {PR}`                         | **CI 전부 SUCCESS** + 충돌/차단 없음 → `gh pr merge --squash` → 머지 성공 확인 → **이슈 #N 클로즈**. ⚠️ 머지 권한 없으면(하네스 차단) 여기서 정지·보고                                                                                                    |
| 9   | main 통합   | `finish-pr <spec PR> --base main`        | 이 이슈가 **해당 `feature/<spec>`의 마지막 열린 이슈**였으면, spec 브랜치를 **main으로** 통합(CI 그린 게이트 동일). 아니면 이 단계는 건너뜀                                                                                                               |

> 각 단계의 자동 게이트가 **성공이면 즉시 다음 단계로** 넘어갑니다(사용자에게 묻지 않음). **실패면 그 자리에서 정지**하고, 무엇이/왜 막혔는지와 권장 복구 단계를 보고합니다.

> **7단계 자율 처리:** `create-pr`는 내부에 push·PR 생성 *사람 승인 게이트*를 갖지만, 무인 오케스트레이터인 이 루프가 부르면 그 게이트는 **자동 통과**합니다(`create-pr`의 "🤖 자율 호출 예외" 조항에 명문화). 단 품질 게이트(tdd-review Blocker 0·working tree 클린·올바른 base·gh 인증)는 그대로 확인하며, 하나라도 어긋나면 PR을 만들지 않고 정지합니다.

> **8·9단계 머지 자율 처리:** 머지는 `finish-pr`에 위임합니다 — **CI 그린 게이트**(`gh pr checks --watch` 전부 SUCCESS)를 통과해야만 `gh pr merge --squash`를 실행하고, 머지 성공(`MERGED`) 확인 후에만 이슈를 닫습니다(`finish-pr`의 체인과 동일). CI 그린이 사람 승인을 대신하는 객관적 게이트입니다.

> **⛔ 머지 권한 의존(환경 제약):** 이 하네스의 auto-mode 분류기는 기본적으로 `gh pr merge`를 **하드 차단**합니다. 사용자가 (a) `.claude/settings.local.json`에 `Bash(gh pr merge:*)`를 추가하거나 (b) **bypass 권한 모드**로 세션을 띄운 경우에만 8·9단계가 실제로 실행됩니다. **권한이 없어 머지가 차단되면 그 자리에서 정지·보고하고, 같은 머지 명령을 절대 재시도하지 마십시오**(우회 시도로 격상되어 더 강하게 막힘). 이때 7단계(PR 생성)까지는 정상 완료된 상태이므로, "PR까지 끝, 머지는 권한 해제 후 `finish-pr`로"라고 인계합니다.

## 하우스키핑 (안전 브랜치 청소)

브랜치가 쌓이는 걸 줄이기 위해, 루프 **시작 시 1회** 안전 청소를 합니다 — 단 **이미 머지된 것만**:

- `git fetch --prune` — 원격에서 사라진 추적 ref 정리.
- `git branch --merged <통합base>`로 뽑은 **머지 완료 로컬 브랜치만** `git branch -d` (미머지는 `-d`가 거부 → 안전). 현재 브랜치·`main`·통합 브랜치(`feature/<spec>`)·`backup-*`·보호 브랜치는 제외.

이 청소(시작 시 1회)는 **머지를 하지 않습니다** — 더는 살아 있을 이유가 없는 가지만 칩니다. 현재 이슈 PR의 **머지·이슈 클로즈·브랜치 삭제**는 8·9단계에서 `finish-pr`이 CI 그린 게이트 위에서 수행합니다(머지 권한이 있을 때).

## 제약 (하드 규칙, 반드시 지킬 것)

- **사람 승인 게이트만 제거한다 — 품질 게이트는 유지한다.** 단계마다 멈춰 묻지 않되, 위 표의 객관적 성공 기준을 **반드시** 확인하고 충족할 때만 진행합니다. 확인 없이 다음 단계로 건너뛰지 않습니다.
- **실패 시 즉시 정지(HALT).** RED가 안 빨갛다 / GREEN이 안 초록이다 / AC 미충족 / REFACTOR가 초록을 깬다 / REVIEW Blocker / 커밋·push 실패 등 어느 하나라도 어긋나면 **그 단계에서 멈추고** 보고합니다. 깨진 채로 밀어붙이거나 다음 단계로 넘어가지 않습니다.
- **순서·범위·시그니처를 지킨다.** 1→…→9 고정, 한 사이클=한 이슈(구현 슬라이스는 그 이슈만 — 9단계 main 통합만 spec 단위), 확정 시그니처 준수, 인접 슬라이스로 확장 금지. 자율이라고 임의 점프·범위 확대 금지.
- **하위 스킬을 그대로 호출한다 — 일을 대신하지 않는다.** 구현/테스트/판정은 각 스킬·에이전트가 합니다. 컨테이너가 직접 코드를 짜거나 게이트를 우회 판정하지 않습니다.
- **머지는 CI 그린 + 권한이 둘 다 있을 때만 자율 실행.** 자율 범위는 **머지·이슈 클로즈·(마지막 이슈면) main 통합까지**로 확장됩니다 — 단 `finish-pr`의 **CI 그린 게이트**(모든 필수 체크 SUCCESS·충돌/차단 없음)를 통과하고 **머지 권한**(bypass/allowlist)이 있을 때만. CI 레드면 머지하지 않고 정지, 권한 없어 차단되면 PR까지 하고 정지·재시도 금지. **머지 성공이 이슈 클로즈의 전제**입니다(머지 안 됐는데 이슈 닫지 않음).
- **디자인 시스템은 UI 이슈의 게이트다.** UI를 새로 만들거나 바꾸는 이슈는 GREEN/REFACTOR에서 `design-system` 스킬을 적용하고, REVIEW에서 준수를 확인합니다 — **핵심 액션 버튼은 `bg-primary`(인디고)**, 로고/라틴은 `font-display`, 색·그림자·radius는 토큰만(임의값 금지). 신규 UI가 회색 토큰만 쓰고 브랜드(primary)가 빠지면 **Blocker로 보고 정지**합니다. "테스트만 통과하면 OK"로 넘기지 않습니다.
- **모든 단계 결과를 로그로 남긴다.** 자율이라 사후 추적이 중요합니다. 각 단계의 실제 결과(통과/실패·수치·판정)를 정직하게 기록하고, 실패를 "완료"로 포장하지 않습니다.
- **테스트를 고쳐 통과시키지 않는다.** GREEN/REVIEW에서 막히면 구현을 고치거나 정지합니다. 통과를 위해 테스트·기대값을 무르지 않습니다(각 단계 스킬의 제약과 동일).
- **언제든 중단 가능.** 사용자가 중간에 멈추라고 하면 즉시 멈춥니다. 자율은 강제가 아니라 기본 모드일 뿐입니다.

> tdd-auto-loop는 무인 주행이되 **브레이크가 달린** 주행입니다. 신호(품질 게이트·CI 그린)가 초록이면 멈추지 않고 **머지·이슈 클로즈·main 통합까지** 달리지만, 빨강이면 반드시 섭니다. 종착역은 "main에 통합된 닫힌 이슈"이되, 마지막 비가역 문(머지)은 **CI 그린과 머지 권한이라는 두 열쇠**가 다 있을 때만 열립니다 — 둘 중 하나라도 없으면 PR 앞에서 정지합니다.
