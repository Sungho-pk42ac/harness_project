# FINISH PR (통합 마감 + 브랜치 정리)

`create-pr`이 만든 PR을 입력받아, CLAUDE.md "새 이슈 작업 사이클" 7단계의 **꼬리(머지 → 이슈 클로즈 → 브랜치 정리)**를 마무리합니다. `create-pr`·`tdd-loop`·`tdd-auto-loop`은 모두 **PR 생성까지**가 경계라, 머지 이후의 정리를 담당하는 주인이 없었습니다 — 이 스킬이 그 공백을 메웁니다.

다루는 행위(머지·이슈 클로즈·브랜치 삭제)는 **가장 비가역적**입니다. 그래서 이 스킬은 머지를 **객관적 게이트(CI 그린 + 충돌 없음)** 위에서만 하고, **머지된 브랜치만** 지우며, 현재/보호/통합 브랜치는 건드리지 않습니다.

> **🔗 전체 체인 (이 스킬이 보장하는 순서):** `① 대상 PR 확인 → ② CI 그린 대기 → ③ squash merge → ④ 머지 성공 시 이슈 자동 클로즈 → ⑤⑥ 브랜치 정리`. 각 칸은 **앞 칸이 성공해야** 다음으로 갑니다 — **CI가 레드면 머지하지 않고**, **머지가 실패/차단되면 이슈를 닫지 않습니다**(머지 성공이 클로즈의 전제). 머지 트리거는 둘: 대화형이면 사용자 승인, 체인 자율 호출이면 **CI 그린이 승인을 대신**합니다.

> **⛔ 이 환경의 하드 천장:** 이 하네스의 auto-mode 분류기는 `gh pr merge`를 **런타임에서 하드 차단**할 수 있습니다. 그 경우 체인은 ③에서 막히며 — **재시도하지 말고**(우회 시도로 격상됨) "사용자가 `.claude/settings.local.json`에 머지 권한을 직접 추가하거나 bypass 모드로 재시작 후 재실행 필요"로 보고하고 정지합니다. 즉 **로직은 준비되지만, 무인 실행은 사용자가 권한을 풀어야 작동**합니다.

## 입력

- PR 번호(또는 현재 브랜치의 PR)를 대상으로 합니다. `/finish-pr 11` 또는 인자 없이 현재 브랜치 기준.
- 대상 PR을 특정 못 하면 추측하지 말고 물어봅니다.

## 실행 순서

### ① 대상 PR & 상태 확인

`gh pr view <PR> --json number,state,mergeStateStatus,baseRefName,headRefName,body,url`로 상태를 봅니다.

- **이미 MERGED** → ②③을 건너뛰고 **④ 이슈 클로즈**로(머지는 이미 끝났으니 클로즈·정리만).
- **OPEN** → ②(CI 그린 게이트)로.
- **CLOSED(미머지)** → 머지하지 않습니다. 닫힌 사유를 보고하고 정지(이슈 클로즈·브랜치 삭제도 하지 않음).

### ② CI 그린 게이트 (머지 전 객관적 관문)

머지는 가장 비가역적이라, **CI가 전부 초록일 때만** 통과시킵니다.

- `gh pr checks <PR> --watch`로 모든 **필수 체크가 SUCCESS**가 될 때까지 기다립니다(완료된 상태면 `gh pr checks <PR>`로 즉시 확인).
- **하나라도 FAILURE/CANCELLED** → 머지하지 않고 **정지·보고**(CI 레드 위 머지 금지). 사용자가 원인을 고쳐 다시 푸시한 뒤 재실행하도록 안내.
- `mergeStateStatus`가 **DIRTY(충돌)·BLOCKED(리뷰 미승인 등)** 이면 머지하지 않고 그 사유를 보고하고 정지.
- CI 전부 SUCCESS **그리고** 충돌·차단 없음 → ③으로.

### ③ squash merge (게이트 통과 후에만)

`gh pr merge <PR> --squash --delete-branch`로 **squash 머지**(커밋 1개로 압축)하고 원격 head 브랜치를 함께 정리합니다.

- **머지 트리거**: 대화형 호출이면 머지 직전 **사용자 승인**을 받습니다. CI 게이트 자율 호출(체인)이면 **②의 CI 그린이 승인을 대신**합니다.
- ⛔ **환경 차단 시**: `gh pr merge`가 하네스에 막히면 **재시도하지 말고**(우회 시도로 격상) "권한 해제(`settings.local.json` / bypass) 후 재실행 필요"로 보고하고 정지합니다. 이때 ④ 이슈 클로즈로 넘어가지 않습니다 — **머지가 실제로 안 됐기 때문**.
- 머지 후 `gh pr view <PR> --json state`로 **`MERGED` 확정**을 확인합니다(성공 확인 전엔 다음 칸으로 안 감).

### ④ 머지 성공 → 이슈 자동 클로즈

**③에서 머지 성공(`state == MERGED`)을 확인한 경우에만** 연결 이슈를 닫습니다. PR 본문의 `Closes #N`을 파싱해 대상 이슈 번호를 얻습니다.

- **base가 기본 브랜치(main)였으면** GitHub가 `Closes`로 **자동 클로즈** → `gh issue view <N> --json state`로 닫힘만 확인.
- **base가 `feature/<spec>`이면** GitHub가 자동으로 닫지 않습니다 → `gh issue close <N> --comment "PR #<PR> 머지로 완료(<url>)"` 로 **수동 클로즈**.
- **스택 PR**(spec 브랜치가 나중에 main으로 일괄 머지되는 전략)이면, 지금 닫을지 vs spec→main 머지 때 일괄로 닫을지 **사용자 의사를 존중**(자율 호출이면 base 규칙대로 닫고 로그로 남김).
- 머지가 안 됐으면(②③에서 정지) **이슈를 닫지 않습니다** — 머지 성공이 클로즈의 전제입니다.

### ⑤ 원격 브랜치 정리

머지된 head 브랜치가 원격에 남아 있으면 삭제합니다: `git push origin --delete <headRefName>` (또는 머지 시 `--delete-branch`를 썼다면 생략).

> 참고: 레포 설정 _"Automatically delete head branches"_(`delete_branch_on_merge`)를 켜두면 이후 머지부터 원격이 자동 삭제됩니다. 사용자가 원하면 `gh api -X PATCH repos/<owner>/<repo> -f delete_branch_on_merge=true`로 켜는 것을 제안합니다.

### ⑥ 로컬 브랜치 정리

- `git fetch --prune`으로 원격에서 사라진 추적 ref를 정리합니다.
- **머지된 로컬 브랜치만** 삭제합니다: `git branch --merged <통합base>`로 목록을 뽑아 `git branch -d <branch>`(`-d`는 미머지면 거부 → 안전장치). 현재 체크아웃 브랜치·`main`·통합 브랜치(`feature/<spec>`)·보호 브랜치는 제외합니다.
- 삭제 전 **무엇을 지울지 목록을 보여주고** 진행합니다.

### ⑦ 보고

체인 결과를 요약합니다: CI 게이트 통과 여부, 머지 여부(또는 환경 차단으로 정지), 이슈 클로즈 상태(#N), 삭제한 원격·로컬 브랜치, prune 결과. 남은 미머지 브랜치가 있으면 "왜 남겼는지"(미머지/보호/현재)를 함께 알립니다. 체인이 중간(CI 레드·머지 차단)에 멈췄으면 **어느 칸에서 왜 멈췄고, 사용자가 무엇을 하면 재개되는지**를 명확히 적습니다.

## 제약 (하드 규칙, 반드시 지킬 것)

- **머지는 CI 그린 게이트 위에서만.** 머지는 비가역입니다. `gh pr checks`가 **전부 SUCCESS**이고 충돌·차단(DIRTY/BLOCKED)이 없을 때만 머지합니다. **CI 레드/PENDING이면 절대 머지하지 않고 정지**합니다. 머지 트리거는 대화형이면 사용자 승인, 체인 자율 호출이면 CI 그린이 대신합니다.
- **머지 성공이 이슈 클로즈의 전제.** 머지가 실제로 `MERGED`로 확정되기 전엔 **이슈를 닫지 않습니다.** 머지가 실패하거나 환경에 차단되면 클로즈 단계로 넘어가지 않습니다(닫힌 이슈가 "완료"를 거짓 신호하지 않도록).
- **환경 차단 시 재시도 금지.** `gh pr merge`가 하네스에 막히면 같은 명령을 반복하지 않습니다(우회 시도로 격상). 권한 해제 방법을 안내하고 정지합니다.
- **머지된 브랜치만 삭제한다.** `git branch -d`(소문자)만 사용해 미머지 브랜치 삭제를 막습니다. `-D`(강제)는 쓰지 않습니다. "혹시 모를 백업" 성격의 브랜치(예: `backup-*`)는 건드리지 않습니다.
- **보호 대상은 절대 삭제 금지.** 현재 체크아웃 브랜치, `main`, 통합 브랜치(`feature/<spec>`), 원격 기본 브랜치는 정리 대상에서 제외합니다.
- **지우기 전에 목록을 보여준다.** 무엇을 원격/로컬에서 삭제할지 먼저 제시하고 진행합니다(자율 호출이어도 삭제 목록은 로그로 남김).
- **이슈 클로즈는 base를 보고 판단.** base가 main이 아니면 자동 클로즈가 안 된다는 점을 반영해, 수동 클로즈할지 사용자 의사를 확인합니다(스펙 브랜치 통합 시 일괄 클로즈 전략을 존중).
- **범위는 "이 PR의 마감"까지.** 무관한 브랜치 대청소로 번지지 않습니다. 전체 정리가 필요하면 별도로 제안합니다.

> finish-pr은 사이클의 **마지막 칸**입니다. create-pr이 연 문 안으로 들어가 머지를 매듭짓고, 더는 필요 없어진 브랜치만 골라 치웁니다 — 살아 있어야 할 가지는 한 개도 자르지 않으면서.
