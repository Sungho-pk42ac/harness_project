# TDD REFACTOR

`tdd-green` 단계의 산출물(전부 초록불인 테스트 + 통과를 위한 최소 구현)을 입력받아, **테스트를 계속 통과시킨 채 코드의 구조·가독성만 개선**합니다. 목표는 단 하나 — **동작을 1도 바꾸지 않으면서(behavior-preserving) 더 깔끔한 코드로 만드는 것**입니다. 새 기능·새 분기·새 행위는 REFACTOR의 몫이 아니라 다음 사이클(RED)의 몫입니다.

왜 "동작 불변"인가: REFACTOR의 안전망은 GREEN 상태의 테스트입니다. 테스트가 초록불을 유지하는 한, 외형(중복·네이밍·구조)을 바꿔도 행위가 보존됐다는 보장이 생깁니다. 반대로 리팩터 중에 빨간불이 나면 그건 "구조 개선"이 아니라 "동작 변경"이라는 신호입니다 — 즉시 되돌립니다. 매 변경마다 테스트를 돌려 초록을 확인하는 짧은 루프가 이 단계의 핵심입니다.

## 실행 순서 (절차)

### ① GREEN 확인 (출발점 점검)

`npx vitest run`으로 **지금 전부 초록불인지** 먼저 확인합니다. 하나라도 빨간 게 있으면 REFACTOR를 시작하지 말고 `tdd-green`으로 돌아가 초록불을 먼저 만듭니다. 리팩터는 "통과하는 코드"에서만 출발합니다 — 이 초록불이 모든 변경의 기준선(baseline)입니다.

### ② 냄새 식별 (무엇을 정리할지 목록화)

대상 이슈에서 건드린 코드를 읽고, 개선할 지점을 **구체적 목록**으로 적습니다. 흔한 냄새:

- **중복**: 같은 로직·리터럴이 여러 곳에 반복(추출·상수화 대상).
- **가독성**: 긴 함수, 모호한 네이밍, 중첩 분기, 의도가 안 드러나는 코드.
- **매직넘버/하드코딩**: 의미 없는 숫자·문자열(이름 있는 상수로).
- **GREEN에서 거칠게 둔 부분**: `tdd-green`이 통과만을 위해 임시로 둔 메모·하드코딩.
- **AC 검증에서 나온 갭**: `ac-fulfillment-auditor`가 짚은 "테스트는 통과하나 구현이 의도와 어긋나는" 지점(단, 동작 교정이 필요하면 그건 새 RED로 — 아래 제약 참고).

각 항목을 "어디를(file:line) → 어떻게 바꾼다"로 한 줄씩 정리하고, **작고 독립적인 단위**로 쪼갭니다.

### ③ 한 번에 하나씩 리팩터

목록의 항목을 **하나만** 골라 적용합니다. 한 번에 여러 개를 바꾸지 않습니다 — 빨간불이 났을 때 원인을 즉시 좁히기 위해서입니다. 추출(extract)·이름 변경(rename)·인라인(inline)·상수화·분기 정리 등 **외형만** 바꾸고, 입출력·side effect·관찰 가능한 동작은 그대로 둡니다.

### ④ 매 변경마다 GREEN 재확인 (짧은 루프)

변경 하나가 끝날 때마다 `npx vitest run`을 돌려 **여전히 전부 초록불인지** 확인합니다.

- 초록이면 그 변경을 확정하고 ③으로 돌아가 다음 항목으로.
- 빨간불이 나면 그건 "동작이 바뀌었다"는 신호 — 테스트를 고쳐 맞추려 하지 말고 **방금 변경을 되돌려** 초록 기준선으로 복귀한 뒤, 더 작은 단위로 다시 시도합니다.

### ⑤ 회귀·타입체크 & 보고

목록을 다 비우면 전체 테스트(`npx vitest run`)와 타입체크(`npx tsc --noEmit` 또는 `npm run build`의 tsc 단계), 린트(`npm run lint`)로 **이 이슈 밖 테스트가 깨지지 않았고 동작이 보존됐는지** 최종 확인합니다.

보고 내용: `통과 N / 실패 0`(GREEN 때와 동일해야 함), 정리한 냄새 목록과 각 변경 요약, **동작 불변·회귀 없음**. 한 사이클(RED→GREEN→REFACTOR)이 끝났으니, 이어서 **다음 이슈의 `test-scenarios` 또는 커밋(`/commit`)으로 인계**합니다.

## 제약 (하드 규칙, 반드시 지킬 것)

- **동작을 바꾸지 않는다.** REFACTOR는 외형(구조·가독성)만 바꿉니다. 관찰 가능한 입출력·side effect가 달라지면 그건 리팩터가 아니라 기능 변경 — 멈추고 새 RED로 갑니다.
- **테스트를 통과시킨 채 진행한다.** 매 변경 후 전부 초록불이어야 합니다. 빨간불이 나면 테스트를 고치지 말고 **변경을 되돌립니다**(테스트는 안전망이지 조정 대상이 아님).
- **새 기능·새 행위를 넣지 않는다(YAGNI).** 테스트가 없는 새 동작을 추가하지 않습니다. 새 행위가 필요하면 이 단계를 끝내고 `tdd-red`부터 새 사이클을 엽니다.
- **테스트 코드는 원칙적으로 손대지 않는다.** 단언(assertion)·기대값을 바꾸지 않습니다. 예외적으로 **테스트 코드 자체의 구조 리팩터**(중복 setup 추출 등)는 가능하나, 그때도 검증 대상·기대값은 불변이어야 하고 변경 후 전부 초록을 재확인합니다.
- **공개 시그니처는 신중히.** 함수/Props 계약을 바꾸면 테스트가 깨지거나 다른 슬라이스에 영향이 갑니다. 시그니처 변경이 꼭 필요하면 멈추고 알립니다(보통은 새 사이클 소관).
- **한 번에 하나, 작게.** 변경을 작은 단위로 쪼개 매번 초록을 확인합니다. 대규모 일괄 변경은 원인 추적을 어렵게 하므로 금지.
- **범위를 지킨다.** 이번 이슈에서 건드린 코드만 정리합니다. 인접 기능·다른 이슈로 리팩터를 확장하지 않습니다(회귀 위험).
- **레포 컨벤션을 따른다.** api(동사+명사·`async`+`throw`), Context의 응답기반 상태갱신(`add`→append/`edit`→map), 컴포넌트 `XxxProps`·콜백 패턴, 한국어 주석·camelCase, 시맨틱 토큰(`bg-card`·`text-muted-foreground` 등) 같은 기존 패턴을 유지·강화하는 방향으로만 정리합니다.

> REFACTOR는 "완성"이 목적이고 그 수단은 "초록불 유지"입니다. 정리하다 행위를 바꾸고 싶어지면 그 욕구를 메모로 남기고 — 그건 다음 RED가 검증할 새 행위입니다. 이 단계에서 코드는 더 깔끔해지되, 하는 일은 똑같아야 합니다.
