# TDD GREEN

`tdd-red` 단계의 산출물(실패하는 Vitest 테스트 + 최소 스텁)을 입력받아, **그 테스트를 통과시키는 데 필요한 최소한의 구현 코드**를 작성합니다. 목표는 단 하나 — **빨간불(RED)을 초록불(GREEN)로 바꾸는 것**입니다. 깔끔한 설계·중복 제거·추가 기능은 GREEN 이후(REFACTOR)의 몫입니다.

왜 "최소" 구현인가: TDD에서 GREEN은 "테스트가 요구하는 것만" 채우는 단계입니다. 테스트가 검증하지 않는 코드를 미리 넣으면, 그 코드는 어떤 테스트도 지켜주지 않는 채로 늘어나고(=죽은 보장 없는 코드), RED→GREEN의 인과가 흐려집니다. 일단 전부 초록불로 만든 뒤, 다듬는 건 다음 단계에서 합니다.

## 하드 규칙 (반드시 지킬 것)

- **테스트 코드를 고치지 않는다.** 통과가 안 되면 구현을 고칩니다. 테스트가 명백히 틀렸다고 판단되면 임의로 바꾸지 말고 멈춰서 사용자에게 알리고 `tdd-red`/`test-scenarios`로 되돌립니다.
- **최소 구현만.** 통과에 필요한 코드만 씁니다. 테스트가 요구하지 않는 기능·분기·추상화·최적화를 미리 넣지 않습니다(YAGNI).
- **확정 시그니처를 지킨다.** `issue-{N}.md`의 시그니처와 `tdd-red`가 둔 함수/Props 형태를 바꾸지 않습니다(시그니처를 바꿔야 하면 멈추고 앞 단계로).
- **다른 이슈 범위로 새지 않는다.** 지금 이슈의 테스트만 초록불로 만듭니다. 인접 기능(정규화·중복·삭제 등 다른 슬라이스)은 그 테스트가 없으면 구현하지 않습니다.
- **전부 초록불이어야 끝.** 일부만 통과하면 미완입니다. 또한 **이 이슈와 무관한 기존 테스트를 깨뜨리지 않습니다**(회귀 금지).
- **기존 코드 패턴을 따른다.** `src/api/notes.ts`(동사+명사·`async`+`throw`), Context의 응답기반 상태갱신(`add`→append/`edit`→map), 컴포넌트 `XxxProps`·콜백 패턴, 한국어 주석·camelCase 등 레포 컨벤션을 유지합니다.

## 입력

- 이슈 번호는 **`$ARGUMENTS`** 로 전달됩니다 (예: `/tdd-green 6`).
- 자연어로 불러도(예: "이슈 6 통과시켜") 번호를 추출해 같은 절차를 따릅니다. 번호를 특정 못 하면 추측하지 말고 물어봅니다.
- **출발점은 RED 상태**입니다. 시작 전 `npx vitest run`으로 "지금 실패 중"임을 먼저 확인합니다. 실패 테스트가 없으면(이미 통과/테스트 없음) GREEN을 시작하지 말고 `tdd-red`부터 안내합니다.

## 절차

### ① RED 확인 (출발점 점검)

`npx vitest run`을 실행해 **현재 무엇이, 왜 실패하는지** 파악합니다. 실패 메시지(어떤 assertion인지, `Not implemented`인지)가 곧 "무엇을 구현해야 하는지"의 명세입니다. 대상 테스트 파일과 `docs/features/tag/issue-{N}.md`의 확정 시그니처도 함께 읽습니다.

### ② 구현 계획 (테스트 → 필요한 최소 변경)

실패 테스트를 묶어, 통과에 필요한 최소 변경을 계층별로 정리합니다. 보통 RED와 같은 안→밖 순서(데이터/타입 → api → Context → 컴포넌트)로 진행하면 의존 방향과 맞아 한 번에 풀립니다.

- 각 실패 테스트가 **무엇을 기대하는지**(호출 인자, 반환값, throw, 렌더 결과)를 한 줄로 적습니다.
- 그 기대를 만족시키는 **가장 작은 코드 변경**을 정합니다.

### ③ 최소 구현

스텁을 실제 동작으로 채웁니다. 통과에 필요한 만큼만:

- **api**: `fetch` 호출 + `if (!res.ok) throw new Error(...)` + `res.json()` 반환. 타임스탬프는 클라이언트에서 채우는 기존 패턴 유지.
- **Context**: api 호출 후 **응답으로 로컬 상태 갱신**(`add`→append, `edit`→map). 자체 try/catch 없이 에러는 호출부로 전파(기존 패턴).
- **컴포넌트**: 테스트가 기대하는 UI/동작만(예: 태그 입력칸·칩 렌더·저장 시 콜백 호출·실패 시 alert). 빈값/정규화/중복 등 다른 이슈 소관 동작은 넣지 않습니다.

### ④ GREEN 확인 (반복)

`npx vitest run`을 다시 실행합니다.

- 아직 빨간 게 있으면 **그 실패 메시지를 근거로 구현만 고쳐** 다시 돌립니다(테스트는 건드리지 않음). 이 ③↔④를 전부 초록불이 될 때까지 반복합니다.
- 한 번에 다 고치려 하기보다, **한 계층(또는 한 테스트)씩 초록불로** 바꿔가며 진행하면 원인 파악이 쉽습니다.

### ⑤ 회귀 확인 & 보고

전체 테스트(`npx vitest run`)와 타입체크(`npx tsc --noEmit` 또는 `npm run build`의 tsc 단계)로 **이 이슈 밖 테스트가 깨지지 않았는지** 확인합니다. 원하면 `--coverage`로 커버리지가 RED 때(≈0%)보다 올라간 것을 함께 보고합니다.

보고 내용: `통과 N / 실패 0`, 변경한 파일과 각 파일에서 채운 최소 구현 요약, 회귀 없음. 이어서 **REFACTOR(정리) 단계로 인계**합니다(동작을 바꾸지 않고 중복·가독성 개선 — 테스트는 계속 초록불 유지).

## 구현 시 참고 (이 레포 패턴)

- **api 계층**(`src/api/notes.ts`): 모든 네트워크 호출은 여기에만. 각 함수 `async` + `fetch` + `if (!res.ok) throw new Error('...')` + `res.json()`. 에러 메시지는 기존 영어 문구(`'Failed to create note'` 등) 유지.
- **Context**(`src/context/NotesContext.tsx`): 낙관적 업데이트 아님 — **API 응답으로** 로컬 `notes` 갱신. 변경 함수는 자체 catch 없이 호출부로 throw 전파.
- **컴포넌트**(`src/components/`): 데이터 변경은 `useNotes()`로, 자체 fetch 금지. 폼은 로컬 `useState`. 실패 처리는 호출부 try/catch + `alert` 패턴.
- **타입**(`src/types/note.ts`): 읽기 시 `note.tags ?? []`로 방어(구버전 노트 호환, ADR-0001).
- **스타일**: UI를 새로 만들 땐 시맨틱 토큰(`bg-card`, `text-muted-foreground`, `border-border`, `text-destructive`)을 쓰고 임의 색상값을 피합니다.

> GREEN은 "통과"가 목적이지 "완성도"가 아닙니다. 통과를 위해 잠시 거칠게 둔 부분(중복·하드코딩)은 메모해 두고 REFACTOR 단계에서 정리하세요 — 단, 그때도 테스트는 계속 초록불이어야 합니다.
