# ADR-0002 — 백업 JSON 형태: 노트 배열 그대로 직렬화

- **상태(Status)**: Accepted
- **날짜**: 2026-05-31
- **관련**: [PRD §6 FR-2](../PRD.md#6-기능-요구사항-functional-requirements), [ADR-0001](0001-client-export-pure-fn.md)

## Context

전체 백업 `.json`을 "어떤 형태"로 담을지 정해야 한다. 제약·맥락은 다음과 같다.

- 백업의 일차 목적은 **유실 대비·이동(다른 곳에 보관)** 이다. 이번 범위에는 **가져오기(import)가 없다**(Non-Goal).
- 데이터 소스는 `NotesContext.notes: Note[]` — 이미 `id/title/content/createdAt/updatedAt/tags`를 가진 완전한 객체 배열이다.
- 파일은 사람이 열어볼 수도(점검), 나중에 도구로 다시 읽을 수도 있어야 하므로 **표준적이고 단순한** 형태가 좋다.

## Decision

백업을 **노트 배열 그대로** 직렬화한다.

```ts
export function notesToJson(notes: Note[]): string {
  return JSON.stringify(notes, null, 2); // 들여쓰기 2칸, 사람이 읽을 수 있게
}
```

- 결과는 `db.json`의 `notes` 컬렉션과 **같은 형태의 배열**이라 직관적이고, 필요하면 그대로 비교·복원하기 쉽다.
- `{ version, exportedAt, notes }` 같은 **메타 래퍼는 두지 않는다**(import이 없어 버전 협상이 불필요).

## Consequences

**긍정**

- 가장 단순하고, 노트의 모든 필드를 **손실 없이** 담는다(라운드트립 시 파싱 결과가 원본과 동일).
- `db.json`과 형태가 같아 사람이 점검하거나 수동 복원하기 쉽다.
- 변환이 `JSON.stringify` 한 줄이라 테스트·유지보수가 쉽다.

**부정 / 트레이드오프**

- 메타데이터(내보낸 시각·스키마 버전·앱 버전)가 파일 안에 없다 → 식별 정보는 **파일명의 날짜**로 대체(FR-5).
- 훗날 import을 도입하며 스키마가 바뀌면 버전 구분이 필요해질 수 있다 → 그때 **메타 래퍼를 새 ADR로 도입**하고, 구버전(배열) 파일도 읽도록 마이그레이션 규칙을 둔다(YAGNI: 지금은 불필요).

## Alternatives Considered

1. **메타 래퍼로 감싸기** (`{ version: 1, exportedAt, notes: [...] }`)
   - 기각 이유: import이 없는 현 범위에선 버전 협상이 불필요한 복잡성. 식별은 파일명 날짜로 충분. 필요해지는 시점에 새 ADR로 도입하는 편이 깔끔하다.
2. **노트마다 개별 .md 파일을 zip으로 묶어 백업**
   - 기각 이유: zip 라이브러리 의존·다중 파일 처리로 범위가 과도. 무손실 복원엔 단일 JSON이 더 적합. (다중 .md는 별도 기능으로 분리)
3. **민감/내부 필드를 제거한 슬림 JSON**(예: `id` 제외)
   - 기각 이유: 백업은 무손실이 핵심이라 `id/타임스탬프`까지 보존해야 한다. 슬림화는 공유용 별도 옵션의 몫.
