# ADR-0001: 인증 데이터 모델 — db.json `users` + 평문 검증

- **Status**: Accepted
- **Date**: 2026-05-30
- 관련: [PRD](../PRD.md) FR-1, [spec-fixed §1·§2](../spec-fixed.md)

## Context

로그인하려면 사용자 계정을 어딘가에 저장하고, 입력한 자격증명을 검증해야 한다.
그러나 현재 백엔드는 JSON Server(`db.json`)뿐이라 **실제 인증 서버·해싱·토큰 발급
수단이 없다.** 이 프로젝트는 강의용 실습으로, 목표는 실서비스 보안이 아니라 인증
UX/상태관리 패턴 학습이다.

선택지:

1. `db.json`에 `users` 컬렉션을 두고 JSON Server로 조회·검증 (평문).
2. 코드에 고정 계정 1개 하드코딩.
3. 별도 인증 서버/JWT 도입.

## Decision

**`db.json`에 `users` 컬렉션을 신설하고, 비밀번호는 평문으로 저장한다.**
검증은 JSON Server 쿼리로 한다: `GET /users?email=<email>&password=<password>`.
결과 배열이 1건 이상이면 성공(첫 사용자), 0건이면 실패.

- 저장 형태: `{ id, email, password }` (id는 JSON Server 부여 — 기존 노트 규칙과 동일).
- 클라이언트는 비밀번호를 제외한 `User { id, email }`로 다룬다.
- 네트워크 호출은 기존 컨벤션대로 **`src/api/auth.ts` 한 곳**에만 둔다.

## Consequences

**좋은 점**

- 기존 JSON Server 인프라를 그대로 활용 — 추가 서버/의존성 없음.
- 쿼리 파라미터 검증이라 db 전체를 받지 않아 효율적이고, 코드가 단순.
- 회원가입(POST /users)으로 자연스럽게 확장 가능(이번엔 비목표).

**나쁜 점 / 한계 (명시적 수용)**

- ⚠️ **실제 보안이 아니다.** JSON Server는 모든 데이터를 클라이언트에 노출하고,
  비밀번호가 평문이며 검증이 클라이언트에서 일어난다. 누구나 `/users`를 직접 조회 가능.
- 따라서 PRD 리스크·코드 주석에 "실습 한정, 비보안"을 반드시 남긴다.
- 실서비스로 가려면 서버측 인증·해싱·토큰이 필요 → 그때 새 ADR로 대체(Superseded).

## Alternatives Considered

- **고정 계정 하드코딩**: 가장 단순하지만 다중 사용자·회원가입 확장이 막히고, 데이터
  계층 학습 가치가 없어 기각.
- **실제 인증 서버/JWT**: 가장 현실적이고 안전하나, 실습 프로젝트 범위를 크게 벗어나고
  백엔드 신설이 필요해 기각.
- **비밀번호 해싱(클라이언트)**: 클라이언트 해싱은 보안 이득이 제한적이고 라이브러리만
  늘어, 평문과 본질적 차이가 없어 기각(한계는 동일하게 수용).
