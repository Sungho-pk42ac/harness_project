# 로그인 기능 PRD

> 제품 관점(무엇을·왜). 기술적 "어떻게"는 [`adr/`](adr/README.md)에 분리한다.
> 입력: [`spec-fixed.md`](spec-fixed.md). 기획 중 진화하는 살아있는 문서.

## 1. 배경 (Background)

노트 CRUD 앱은 현재 **누구나 모든 노트에 접근**할 수 있다. 학습 흐름상 "사용자가
로그인해야 앱을 쓸 수 있다"는 인증 게이트를 도입해, 인증 상태 관리·세션 유지·보호
라우팅의 기본 패턴을 실습한다.

현재 백엔드는 JSON Server(`db.json`)뿐이라 실제 인증 서버가 없다. 따라서 이번 기능은
**실서비스 보안이 아니라 인증 UX/상태관리 패턴 학습**에 초점을 둔다(평문·클라이언트
검증의 한계는 [ADR-0001](adr/0001-auth-data-model.md)에 명시).

## 2. 목표 / 비목표

### 목표 (Goals)

- 이메일/비밀번호로 로그인하고, 로그인해야만 노트 화면에 들어갈 수 있다.
- 로그인 상태가 새로고침 후에도 유지된다.
- 사용자가 명시적으로 로그아웃할 수 있다.
- 기존 코드 컨벤션(Context 패턴, `api/` 단일 fetch 계층)을 그대로 따라 일관성을 지킨다.

### 비목표 (Non-goals)

- 회원가입 / 비밀번호 찾기 / 변경.
- 비밀번호 해싱·토큰(JWT)·실제 보안.
- 노트의 사용자별 소유/분리(`userId`) — 노트는 공용 유지.
- 권한·역할(role), 세션 자동 만료/갱신.
- react-router 등 라우팅 라이브러리 도입.

## 3. 사용자 / 가치

- **대상**: 앱을 쓰려는 단일 사용자(실습 시드 계정).
- **가치**: "내 앱은 로그인해야 쓴다"는 최소 인증 경험과, 새로고침해도 풀리지 않는
  세션의 편의.

## 4. 기능 요구사항 (FR)

우선순위: **P0**(필수, walking skeleton 포함) · **P1**(중요) · **P2**(선택).

| ID   | 우선순위 | 요구사항                        | 수용 기준(요약)                                    |
| ---- | -------- | ------------------------------- | -------------------------------------------------- |
| FR-1 | P0       | 이메일/비밀번호로 로그인        | 올바른 계정 입력 시 노트 화면으로 전환된다         |
| FR-2 | P0       | 미로그인 시 노트 화면 접근 차단 | 로그인 전에는 `LoginPage`만 보인다                 |
| FR-3 | P0       | 세션 유지(새로고침 복원)        | 로그인 후 새로고침해도 로그인 상태가 유지된다      |
| FR-4 | P1       | 로그아웃                        | 로그아웃 시 세션이 지워지고 `LoginPage`로 돌아간다 |
| FR-5 | P1       | 로그인 실패 인라인 에러         | 불일치 시 폼 안에 에러 메시지가 표시된다           |
| FR-6 | P2       | 현재 사용자 표시                | 헤더에 로그인한 사용자 이메일이 보인다             |

### 수용 기준 (상세)

- **FR-1 로그인**
  - `GET /users?email=&password=` 결과가 1건 이상이면 성공, 해당 user(`{id,email}`)를 상태에 설정.
  - 0건이면 실패로 처리(FR-5).
- **FR-2 접근 차단**
  - `user`가 `null`이면 `App`은 `LoginPage`만 렌더, 노트 화면(`Layout`/`NoteList`/`NoteEditor`)은 렌더하지 않는다.
- **FR-3 세션 유지**
  - 로그인 성공 시 비밀번호를 제외한 user를 localStorage에 저장.
  - 앱 시작 시 localStorage를 읽어 `user`를 복원. 복원 판정 동안의 로딩 처리를 포함.
- **FR-4 로그아웃**
  - 헤더의 로그아웃 버튼 클릭 → localStorage 제거 + `user=null` → `LoginPage` 전환.
- **FR-5 실패 에러**
  - 인라인 텍스트(`text-destructive`)로 "이메일 또는 비밀번호가 올바르지 않습니다." 표시. `alert` 미사용.
  - 네트워크 오류도 동일 영역에 사용자 친화 메시지로 표시.
- **FR-6 사용자 표시**
  - 헤더 우측에 `user.email` + 로그아웃 버튼.

## 5. 성공 지표 (Success Metrics)

실습 프로젝트이므로 정성 기준:

- 올바른 시드 계정으로 로그인 → 노트 화면 진입까지 막힘 없이 동작.
- 로그인 후 새로고침 시 재로그인 요구가 **발생하지 않음**.
- 잘못된 계정 입력 시 화면 전환 없이 인라인 에러만 표시.
- 핵심 로직(성공/실패/복원/로그아웃) Vitest 테스트가 통과.

## 6. 범위 (Scope)

[`spec-fixed.md` §7](spec-fixed.md)과 동일. P0(FR-1~3)이 walking skeleton의 골격이며,
삭제/표시 성격의 가치(FR-4~6)를 뒤이어 덧붙인다.

## 7. 리스크 / 완화

| 리스크                                               | 영향                    | 완화                                                                                                    |
| ---------------------------------------------------- | ----------------------- | ------------------------------------------------------------------------------------------------------- |
| 평문 비밀번호·클라이언트 검증이 실제 보안으로 오해됨 | 잘못된 보안 인식        | PRD·ADR·코드 주석에 "실습 한정, 비보안" 명시([ADR-0001](adr/0001-auth-data-model.md))                   |
| localStorage 토큰 부재로 위변조 가능                 | 누구나 user 흉내 가능   | 실습 범위로 수용. 한계를 ADR에 기록                                                                     |
| 세션 복원 타이밍(초기 로딩) 처리 누락 시 깜빡임      | 로그인 화면이 잠깐 번쩍 | `AuthContext`에 `loading` 상태 두고 복원 완료 전 렌더 가드([ADR-0002](adr/0002-session-persistence.md)) |
| 라우터 없이 조건부 렌더 → URL 구분 불가              | 딥링크/뒤로가기 약함    | 비목표로 선언. 추후 필요 시 react-router 도입([ADR-0003](adr/0003-route-protection.md))                 |
| 노트 공용이라 사용자 전환해도 같은 노트              | 멀티유저 오해           | 비목표 명시. 향후 `userId` 분리는 별도 기능                                                             |
