# 기능 기획 워크플로 (Feature Planning Workflow)

거친 아이디어 하나를 **구현 가능한 이슈 묶음**까지 끌고 가는 반복 가능한 절차입니다.
태그 기능을 이 절차로 기획했고, 각 단계의 실제 산출물을 예시로 함께 적었습니다. 새 기능도 같은 경로를 따르세요.

## 왜 이렇게 하나

- **모호함을 앞 단계에서 제거**한다. 뒤로 갈수록 수정 비용이 커지므로, 코드 한 줄 쓰기 전에 결정을 끝낸다.
- **무엇/왜(제품)** 와 **어떻게/왜(기술)** 를 분리한다. 섞으면 추적·번복이 어렵다.
- 작업을 **사용자 가치 단위(수직 슬라이스)** 로 쪼개 빨리, 끝까지 동작하는 조각을 만든다.

## 한눈에 보기

```
거친 아이디어
   │
   ▼  ① 초안 정의서
spec.md ──────────────┐
   │  ② 인터뷰로 구체화 (쉬운 질문 + 추천안)
   ▼
spec-fixed.md ────────┤
   │  ③ 제품/기술 분리
   ▼
PRD.md  +  adr/ ──────┤
   │  ④ 수직 슬라이싱 (AC + DoD)
   ▼
issues.md ────────────┘
   │
   ▼
구현
```

| 단계              | 산출물            | 답하는 질문                             |
| ----------------- | ----------------- | --------------------------------------- |
| ① 초안 정의서     | `spec.md`         | 무엇을 만들고 싶은가? (러프)            |
| ② 인터뷰로 구체화 | `spec-fixed.md`   | 구체적으로 어떻게 동작해야 하는가?      |
| ③ PRD + ADR       | `PRD.md`, `adr/*` | 무엇/왜(제품) + 어떻게/왜 이 선택(기술) |
| ④ 수직 슬라이싱   | `issues.md`       | 어떤 순서로 쪼개 만들까?                |

---

## ① 초안 정의서 — `spec.md`

거친 요구를 **최소한의 형태**로 적는다. 완벽할 필요 없다. 다음 단계 질문의 출발점이면 된다.

- **입력**: 머릿속 아이디어, 한두 줄 요구.
- **활동**: 기능 개요 + 핵심 요구사항 + 비고(제약)만 적는다.
- **산출물**: `spec.md`.
- **다음 단계 트리거**: "이걸 실제로 만들려니 결정 안 된 게 많다"는 느낌이 드는 순간.

> **태그 예시** — [`tag/spec.md`](tag/spec.md): "노트에 태그를 추가/삭제/조회한다, 문자열로 저장, 중복 불허" 수준. 데이터 형태·입력 UX·검증 규칙은 아직 미정.

## ② 인터뷰로 구체화 — `spec-fixed.md`

초안의 **모호함을 질문으로 없앤다.** 핵심은 "쉬운 기술 질문 + 추천안과 이유"를 한 번에 하나씩 던지는 것. 답이 모이면 확정본을 쓴다.

- **입력**: `spec.md`.
- **활동**: 데이터 형태 / 입력 방식 / 정규화·중복·개수 / 표시 / 저장 흐름 / 범위 / 테스트를 질문으로 결정. 각 질문에 **추천안과 그 이유**를 붙여 결정을 돕는다.
- **산출물**: `spec-fixed.md` (확정본).
- **다음 단계 트리거**: 구현에 필요한 결정이 빠짐없이 채워졌을 때.

> **태그 예시** — [`tag/spec-fixed.md`](tag/spec-fixed.md): 20개 질문으로 `tags: string[]`, Enter/쉼표 입력, 정규화(trim+소문자), 중복 시 alert, 최대 5개, 칩 표시, 필터는 범위 제외 등을 확정.

## ③ 제품/기술 분리 — `PRD.md` + `adr/`

확정된 내용을 **두 관점으로 나눠** 정리한다. 한 문서에 섞지 않는 게 핵심.

- **PRD** = 무엇을·왜. 배경, 목표/비목표, 요구사항(FR), 성공지표, 범위, 리스크. 기획 중 진화한다.
- **ADR** = 어떻게·왜 이 선택. 중요한 기술 결정 **하나당 한 파일**, 맥락·결정·결과·기각한 대안. 한 번 정하면 불변(번복 시 새 ADR로 대체).
- **입력**: `spec-fixed.md`.
- **산출물**: `PRD.md`, `adr/NNNN-*.md` + `adr/README.md`(인덱스). 서로 링크.
- **다음 단계 트리거**: 요구사항과 핵심 기술 결정이 합의됐을 때.

> **태그 예시** — [`tag/PRD.md`](tag/PRD.md)(FR-1~10·지표·범위·리스크) + [`tag/adr/`](tag/adr/README.md): 0001 데이터 모델(string[] 임베드), 0002 정규화·중복·개수 정책, 0003 addNote 시그니처 확장.

## ④ 수직 슬라이싱 — `issues.md`

구현 작업을 **계층(수평)이 아니라 사용자 가치 단위(수직)** 로 쪼갠다. 각 이슈는 데이터→API→UI를 관통해 그 자체로 동작·관찰 가능해야 한다.

- **입력**: `PRD.md`(특히 FR).
- **활동**:
  - **walking skeleton** 먼저 — 가장 얇게 전 계층을 관통하는 happy-path 한 조각.
  - 이후 삭제·검증·표시 등 가치를 덧붙인다.
  - 각 이슈에 **AC(인수 기준)와 DoD(완료 정의)를 분리**해 적는다.
- **산출물**: `issues.md`.
- **다음 단계 트리거**: 각 이슈가 독립적으로 착수 가능할 만큼 명확할 때 → 구현 시작.

### AC ≠ DoD (꼭 구분)

|      | 인수 기준 (AC)             | 완료 정의 (DoD)                   |
| ---- | -------------------------- | --------------------------------- |
| 질문 | "**올바르게 동작**하는가?" | "**정말 끝났는가?**"              |
| 범위 | 이슈별 행위                | 공통 품질·프로세스 게이트         |
| 포맷 | **Given / When / Then**    | 체크리스트(테스트·토큰·회귀·리뷰) |

> **태그 예시** — [`tag/issues.md`](tag/issues.md): TAG-1(추가·영속화 = walking skeleton) → TAG-2(삭제) → TAG-3(검증·정규화) → TAG-4(목록 표시). 각 이슈는 AC를 Given/When/Then으로, DoD를 품질 게이트로 분리.

---

## 핵심 원칙 요약

1. **모호함은 앞 단계에서 제거** — 인터뷰(②)에서 결정을 끝내고 코드로 넘어간다.
2. **제품 ↔ 기술 분리** — PRD(무엇/왜)와 ADR(어떻게/왜 이 선택)을 나눈다.
3. **수직 슬라이싱** — 계층이 아니라 사용자 가치로 쪼갠다. walking skeleton 먼저.
4. **AC ≠ DoD** — 행위는 AC(G/W/T), 품질은 DoD(체크리스트).

## 산출물 맵

이번 태그 기능이 남긴 결과물:

```
docs/features/
├── feature-planning-workflow.md   ← (이 문서)
└── tag/
    ├── spec.md                    ① 초안 정의서
    ├── spec-fixed.md              ② 인터뷰 확정본
    ├── PRD.md                     ③ 제품 요구사항
    ├── adr/                       ③ 기술 결정 기록
    │   ├── README.md
    │   ├── 0001-tag-data-model.md
    │   ├── 0002-tag-normalization-policy.md
    │   └── 0003-addnote-api-extension.md
    └── issues.md                  ④ 수직 슬라이스 이슈
```

## 다음 기능에 적용하는 체크리스트

- [ ] ① `spec.md` — 개요·요구사항·비고를 러프하게 적었다.
- [ ] ② `spec-fixed.md` — 쉬운 질문 + 추천안으로 모든 모호함을 결정했다.
- [ ] ③ `PRD.md` — 배경·목표/비목표·FR·지표·범위·리스크를 정리했다.
- [ ] ③ `adr/` — 중요한 기술 결정마다 ADR(맥락·결정·결과·기각 대안)을 남겼다.
- [ ] ④ `issues.md` — 수직 슬라이스로 쪼개고, 각 이슈에 AC(G/W/T) + DoD를 분리해 적었다.
- [ ] 구현 시작 전, 각 이슈가 독립 착수 가능할 만큼 명확하다.
