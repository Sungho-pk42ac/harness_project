# ADR-0001 — 소프트 삭제 데이터 모델: `Note.deletedAt?` 마킹

- **상태(Status)**: Accepted
- **날짜**: 2026-05-31
- **관련**: [PRD §9](../PRD.md#9-데이터--상태-모델-개요), [ADR-0002](0002-active-trashed-derivation.md)

## Context

노트 삭제를 되돌릴 수 있게 하려면, "삭제됨"을 **데이터에 어떻게 표현**할지 정해야 한다. 제약은 다음과 같다.

- 저장소는 JSON Server(`db.json`)로 스키마가 자유롭고 관계형 조인이 없다.
- 기존 노트에는 삭제 관련 필드가 없다(마이그레이션 부담 최소화 필요).
- 기존 `removeNote`는 HTTP DELETE로 **즉시 영구 제거**한다 — 되돌릴 수 없는 게 문제의 본질.
- 휴지통은 "언제 지워졌는지"를 보여주거나(향후) 보관기간 계산에 쓰일 수 있어, 단순 boolean보다 **시각 정보**가 확장에 유리하다.

## Decision

삭제 여부를 노트 문서 안의 **선택적 타임스탬프 필드 `deletedAt?: string`(ISO 8601)** 로 표현한다.

```ts
interface Note {
  // ...
  deletedAt?: string; // 소프트 삭제 시각. 없으면(undefined) 활성 노트
}
```

- **활성 노트의 기본 표현은 "필드 부재(`undefined`)"** 로 통일한다. 즉, 삭제 안 된 노트는 `deletedAt`을 갖지 않는다.
- 삭제 = `updateNote(id, { deletedAt: new Date().toISOString() })`. 타임스탬프는 **클라이언트**가 채운다(`createdAt/updatedAt`과 동일 철학).
- 복원 = `updateNote(id, { deletedAt: null })`. 읽는 쪽은 **truthy 여부(`!!deletedAt`)** 로만 판정하므로 서버가 `null`을 남기든 키를 지우든 동일하게 활성으로 본다.
- 영구 삭제만 기존 `deleteNote(id)`(HTTP DELETE)로 실제 제거한다.
- 기존 데이터 호환: 읽는 지점에서 `deletedAt`이 falsy면 **활성**으로 취급(서버/기존 문서 일괄 수정 없음).

## Consequences

**긍정**

- 필드 하나 추가로 소프트 삭제가 성립한다. 서버 변경·마이그레이션이 없다.
- 시각을 보존하므로 향후 "삭제 N일 후 자동 비우기"·"휴지통 보관기간 표시"를 데이터 변경 없이 얹을 수 있다.
- 삭제·복원이 모두 기존 `updateNote` 한 경로로 처리되어 API 표면이 늘지 않는다.
- "활성 = 필드 부재" 불변식 덕에 활성 노트 직렬화가 기존과 동일(불필요한 `deletedAt:null` 없음).

**부정 / 트레이드오프**

- 삭제된 노트도 `notes`에 남아 메모리/DB가 누적된다 → 자동 비우기(별도 슬라이스)로 완화.
- `removeNote`의 **의미가 영구삭제에서 소프트삭제로 바뀐다** → 기존 호출부(NoteList)와 회귀 테스트로 보호해야 한다.
- PATCH로 `null`을 보내는 처리가 서버 구현에 따라 키 잔존/제거가 갈릴 수 있어, **읽기는 truthy 판정으로만** 통일해야 한다(불변식).

## Alternatives Considered

1. **boolean `isDeleted` 플래그**
   - 장점: 가장 단순.
   - 기각 이유: 삭제 시각 정보가 없어 자동 비우기·보관기간 등 향후 요구에 다시 스키마를 바꿔야 한다. 시각 한 필드로 boolean의 표현력을 모두 포함하면서(truthy 판정) 확장성을 얻을 수 있다.
2. **별도 `deletedNotes` 컬렉션으로 이동**(`/notes` ↔ `/trash`)
   - 기각 이유: 이동 시 두 컬렉션 간 일관성·ID 관리 부담, 복원 시 역이동 필요. 단일 컬렉션 + 파생 필터가 훨씬 단순(YAGNI).
3. **클라이언트 메모리에서만 숨기고 서버는 즉시 DELETE**
   - 기각 이유: 새로고침/재조회 시 영속성이 없어 "복원"이 불가능 — 안전망이라는 목적 자체를 달성하지 못한다.
