## 1. TL;DR

노트 삭제를 **즉시 영구 제거 대신 `deletedAt` 마킹(소프트 삭제)** 으로 바꾸고, **휴지통 화면**에서 삭제된 노트를 **복원**하거나 **영구 삭제**할 수 있게 한다. 실수로 지운 노트를 되돌릴 안전망을 제공한다. 이번 범위는 **소프트 삭제·휴지통 조회·복원·영구삭제**까지이며, **자동 비우기 스케줄·일괄 처리·Undo 토스트는 다음 릴리스로 분리**한다.

## 2. 배경 & 문제 (Background & Problem)

- 현재 `removeNote`는 **HTTP DELETE로 노트를 즉시 영구 제거**한다. 로컬 단일 사용자 앱이라 실수로 지운 노트를 **되돌릴 수단이 전혀 없다** — 가장 아픈 데이터 손실 구멍.
- 노트가 늘고 삭제가 잦아질수록 "잘못 지웠다"의 비용이 커진다. 한 번의 오클릭이 곧 영구 손실이다.
- 저장소는 JSON Server(`db.json`)로 스키마가 자유로워 **필드 하나(`deletedAt`) 추가만으로** 소프트 삭제를 도입할 수 있다(서버 변경 불필요).
- **왜 지금**: 태그·검색으로 노트 자산이 쌓이는 흐름에서, 그 자산을 **잃지 않게 보호**하는 안전망이 실사용 신뢰도를 가장 크게 올린다.

## 3. 목표 & 비목표 (Goals / Non-Goals)

### Goals

- G1. 노트 삭제가 즉시 영구 제거가 아니라 **되돌릴 수 있는 소프트 삭제**가 된다.
- G2. 삭제된 노트를 **휴지통 화면**에서 모아 보고 **복원**할 수 있다.
- G3. 더 이상 필요 없는 노트는 휴지통에서 **영구 삭제**할 수 있다.
- G4. 일반 목록은 활성 노트만 보여 평소 사용 경험은 그대로다.

### Non-Goals (이번 릴리스 제외)

- N1. **자동 비우기 스케줄**(N일 경과 시 자동 영구삭제).
- N2. **일괄 처리**(다중 선택 복원/영구삭제, "휴지통 비우기" 버튼).
- N3. **Undo 토스트**(삭제 직후 "실행 취소" 스낵바).
- N4. 휴지통 보관 기간 표시·정렬·검색.

## 4. 성공 지표 (Success Metrics)

| 지표        | 정의                                      | 목표                        |
| ----------- | ----------------------------------------- | --------------------------- |
| 복구 가능성 | 삭제한 노트를 휴지통에서 되살릴 수 있는가 | AC로 검증(100%)             |
| 데이터 안전 | 일반 삭제로 DB에서 사라진 노트 수         | 0 (영구삭제 외엔 제거 없음) |
| 목록 정합성 | 활성 목록에 삭제 노트가 새지 않는가       | 단위 테스트 100% 통과       |
| 회귀        | 기존 CRUD·태그·검색 동작 영향             | 회귀 0                      |

> 소규모 학습 프로젝트이므로 정량 트래킹 인프라는 없다. 지표는 **수용 기준(AC)과 단위 테스트·수동 점검**으로 대체한다.

## 5. 사용자 & 유스케이스

- **페르소나**: 노트를 다수 작성·관리하는 단일 사용자(로컬 앱).
- **핵심 유스케이스**
  - U1. 노트를 삭제했다가 "아 방금 그거 필요했는데" → 휴지통에서 **복원**한다.
  - U2. 평소 목록에는 지운 노트가 보이지 않아 깔끔하게 작업한다.
  - U3. 확실히 필요 없는 노트를 휴지통에서 **영구 삭제**해 정리한다.

## 6. 기능 요구사항 (Functional Requirements)

우선순위: **P0**=이번 릴리스 필수, **P1**=가능하면 포함.

| ID    | 요구사항                              | 우선순위 | 수용 기준(AC)                                                            |
| ----- | ------------------------------------- | -------- | ------------------------------------------------------------------------ |
| FR-1  | 삭제를 소프트 삭제로 바꾼다           | P0       | "삭제" 시 DB에서 제거되지 않고 `deletedAt`이 채워진다 (ADR-0001)         |
| FR-2  | 일반 목록은 활성 노트만 표시한다      | P0       | `deletedAt`이 있는 노트는 일반 목록에 보이지 않는다 (ADR-0002)           |
| FR-3  | 휴지통 화면을 연다                    | P0       | 헤더의 휴지통 토글로 삭제된 노트만 모인 화면을 본다 (ADR-0002)           |
| FR-4  | 휴지통의 노트를 복원한다              | P0       | "복원" 시 `deletedAt`이 비워지고 노트가 일반 목록으로 돌아온다           |
| FR-5  | 휴지통의 노트를 영구 삭제한다         | P0       | "영구 삭제"(확인 후) 시 DB에서 실제로 제거되어 어디에도 안 보인다        |
| FR-6  | 기존(필드 없는) 노트와 호환된다       | P0       | `deletedAt`이 없는 기존 노트는 에러 없이 활성으로 취급된다 (ADR-0001)    |
| FR-7  | 휴지통 빈 상태를 전용 문구로 안내한다 | P0       | 휴지통이 비면 **"휴지통이 비어 있습니다"**(활성 0건과 구분)              |
| FR-8  | 변경을 영속화한다                     | P0       | 삭제·복원·영구삭제 결과가 서버에 반영되어 재조회 시 유지된다             |
| FR-9  | 실패를 사용자에게 알린다              | P0       | 실패 시 기존 패턴(`try/catch` + alert)으로 알린다                        |
| FR-10 | 영구 삭제는 오작동을 막는다           | P1       | 영구 삭제 전 **확인(confirm)** 을 거친다(되돌릴 수 없으므로)             |
| FR-11 | 사라진 노트의 편집 상태를 정리한다    | P1       | 편집 중 노트가 삭제/영구삭제되면 선택이 해제되어 편집기가 빈 상태가 된다 |

## 7. 비기능 요구사항 (NFR)

- **아키텍처 정합**: 서버 데이터(활성·삭제 모두)는 `NotesContext`의 `notes` 단일 출처. **휴지통 보기 여부(`view`)는 화면 상태**이므로 `App.tsx`의 `useState`에 둔다(`selectedNoteId`와 같은 경계). 서버 데이터와 섞지 않는다 (ADR-0002).
- **호환성**: 기존 `db.json` 노트는 `deletedAt`이 없다. falsy면 활성으로 취급해 **마이그레이션 없이** 동작한다 (ADR-0001).
- **데이터 불변**: 활성/삭제 분리는 **파생 계산(순수 함수)** 으로만 — 원본 `notes`는 변형하지 않는다 (ADR-0002).
- **성능**: 수십~수백 건 동기 `filter`는 체감 지연 0.
- **접근성**: 휴지통 토글·복원·영구삭제 버튼은 키보드 포커스·클릭 가능, `aria-label` 제공.
- **스타일**: 시맨틱 토큰(`bg-card`, `border-border`, `text-muted-foreground`, `text-destructive`)만 사용. 임의 색/그림자/인라인 style 금지.

## 8. UX / 인터랙션 흐름

```
[삭제]   목록 "삭제" → removeNote → deletedAt=now(PATCH) → 응답 교체 → 활성 목록에서 사라짐(휴지통엔 생김)
[휴지통]  헤더 토글 → view='trash' → trashedNotes만 표시 (비면 "휴지통이 비어 있습니다")
[복원]   휴지통 "복원" → restoreNote → deletedAt=null(PATCH) → 응답 교체 → 활성 목록 복귀
[영구삭제] 휴지통 "영구 삭제" → confirm → purgeNote → DELETE → 로컬에서 제거 → 어디에도 없음
```

빈 상태 우선순위(`NoteList`): `loading` → `error` → 활성 0건("노트가 없습니다") → 목록.
빈 상태(`TrashList`): `loading` → `error` → 휴지통 0건("휴지통이 비어 있습니다") → 목록.

## 9. 데이터 / 상태 모델 (개요)

- 소프트 삭제 표현: **[ADR-0001](adr/0001-soft-delete-data-model.md)** — `Note.deletedAt?: string`, 활성=필드 부재.
- 활성/삭제 파생 + 화면 상태 경계: **[ADR-0002](adr/0002-active-trashed-derivation.md)** — 순수 함수로 파생, 원본 불변, `view`는 `App` 화면 상태.

```ts
// src/types/note.ts
export interface Note {
  id: string;
  title: string;
  content: string;
  createdAt: string;
  updatedAt: string;
  tags: string[];
  deletedAt?: string; // 신규
}
// src/utils/trash.ts
export function isTrashed(note: Note): boolean;
export function activeNotes(notes: Note[]): Note[];
export function trashedNotes(notes: Note[]): Note[];
// App.tsx
const [view, setView] = useState<'notes' | 'trash'>('notes');
```

## 13. 의존성

- `NotesContext`의 `notes`(단일 출처)와 `App.tsx`의 화면 상태 경계.
- 기존 `api/notes.ts`의 `updateNote`/`deleteNote` 재사용(추가 API 없음).
- 태그·검색 기능과 독립이나, 검색은 활성/휴지통 어느 목록을 거르는지에 영향받을 수 있음(현재 검색은 활성 목록 대상 — 충돌 없음).
