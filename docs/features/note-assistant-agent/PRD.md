# note-assistant-agent — PRD (제품 요구사항)

## 배경

SLNOTE는 노트 CRUD·태그·검색·휴지통·핀·정렬·내보내기·자동저장을 갖춘 노트앱이다. 데이터는
Supabase(Postgres + Auth + RLS)에 있고 Vercel로 배포된다. 노트가 쌓일수록 "원하는 노트를 찾고,
빠르게 만들고, 고치는" 마찰이 커진다. **대화형 AI 비서**로 이 마찰을 줄인다 — 클릭·타이핑 흐름 대신
자연어 한 줄로 검색·생성·수정·삭제를 맡긴다.

## 목표 / 비목표

**목표**

- 채팅으로 노트를 **검색·생성·수정·삭제**(삭제는 확인 게이트)할 수 있다.
- 에이전트는 **LangGraph**로 오케스트레이션하고 **OpenAI**를 쓰되, **키는 절대 브라우저에 노출하지 않는다**.
- 모든 동작은 **로그인 사용자 본인 노트**로 제한된다(RLS).
- 에이전트의 변경이 **노트 목록/에디터에 즉시 반영**되어 사용자가 통제감을 갖는다.

**비목표(이번 기능 아님)**

- 응답 스트리밍, 대화 영속화/검색, 멀티 세션, 음성/STT, 모델 선택 UI, 비용 대시보드.
- **Graph DB 시각화** — 별도 기능 `note-graph-viz`로 분리(아래 로드맵).

## 기능 요구사항 (FR)

| ID   | 우선순위 | 요구사항                                                | 수용 기준(요지)                                                          |
| ---- | -------- | ------------------------------------------------------- | ------------------------------------------------------------------------ |
| FR-1 | P0       | 채팅 패널에서 메시지를 주고받는다                       | 입력 전송 시 사용자 메시지 표시, 어시스턴트 응답 렌더                    |
| FR-2 | P0       | 에이전트가 `createNote` 도구로 노트를 만든다            | "~노트 만들어줘" → 노트 1개 생성 + 목록 반영                             |
| FR-3 | P0       | 브라우저↔OpenAI 사이에 인증 프록시를 둔다               | 키는 함수 env에만, 미인증 요청 401                                       |
| FR-4 | P1       | 에이전트가 `searchNotes`/`listNotes`로 내 노트를 읽는다 | "~찾아줘/목록" → RLS 범위 내 결과 요약                                   |
| FR-5 | P1       | 에이전트가 `updateNote`로 노트를 고친다                 | "제목/본문/태그 바꿔줘" → 변경 + 반영                                    |
| FR-6 | P2       | 에이전트가 `deleteNote`로 노트를 휴지통에 보낸다        | **확인 카드 승인 후에만** 소프트 삭제                                    |
| FR-7 | P1       | 결과 양방향 반영 + 로딩/에러 UX                         | 생성/수정/삭제가 채팅+목록에 동기화, 실패 시 안내                        |
| FR-8 | P0       | UI는 디자인 시스템 준수                                 | 시맨틱 토큰만, 핵심 액션 `bg-primary`, 임의 hex/그림자/인라인 style 금지 |

## 성공 지표

- 채팅으로 노트 1개 생성까지 평균 **1 메시지**(happy path).
- 도구 디스패처·그래프·reducer **단위 테스트 커버리지 ≥ 핵심 분기 전부**, LLM 모킹으로 결정적.
- 에이전트 동작 중 **타사용자 노트 노출 0건**(RLS E2E로 확인).

## 리스크 / 완화 (Devil's Advocate 반영)

| 리스크                                        | 완화                                                                     |
| --------------------------------------------- | ------------------------------------------------------------------------ |
| LangGraph.js가 Vite 브라우저 번들에서 안 돈다 | 에이전트 루프를 인터페이스 뒤로 추상화 → 순수 tool-loop 폴백([ADR-0001]) |
| Edge Function 코드가 Vitest 밖                | 함수는 얇은 프록시만, 로직은 `src/lib`로 → 단위 테스트 가능([ADR-0002])  |
| LLM 비결정성 ↔ RED/GREEN                      | LLM 모킹으로 그래프 테스트, 실제 호출은 E2E 스모크([ADR-0005])           |
| service_role로 RLS 우회 위험                  | 도구를 브라우저(사용자 세션)에서 실행 → RLS 자동 준수([ADR-0003])        |
| 삭제 비가역 위험                              | 소프트 삭제 + 실행 전 확인 게이트([ADR-0004])                            |

## Open Questions (보수적 기본값으로 진행 중)

- **OPENAI_API_KEY 보유 미확인**: 코드·테스트는 키 없이 통과하도록 설계. 실제 LLM 호출/배포는
  **사용자가 키를 함수 secret으로 제공한 뒤** 가능(프록시는 키 없으면 503 + 가이드). → 통합 이슈의 DoD에 명시.
- **LangGraph.js 번들 검증**: walking skeleton 직후 스파이크로 확정. 막히면 폴백 루프(테스트 불변).

## 로드맵 (다음 기능 — 이 기능 다음 auto-loop 런)

- **`note-graph-viz`**: 노트를 **그래프(노드=노트, 엣지=공유 태그/링크/에이전트 도출 관계)** 로 시각화.
  **Neo4j**(<https://github.com/neo4j/neo4j>)의 속성 그래프 모델을 **참고**하되, 별도 그래프 DB 서버를
  세우기보다 기존 데이터에서 그래프를 만들어 브라우저로 시각화하는 방향을 우선 검토(기존 `graphify` 자산 활용).
  상세 설계는 해당 기능의 spec에서.
