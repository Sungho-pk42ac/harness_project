# 목록 정렬 토글 기능 정의서 (확정본)

> 원본 `spec.md`를 바탕으로, 설계 질문의 답변을 반영해 구현 가능한 수준으로 구체화한 문서입니다.
> 검색(`search`) 기능과 같은 "읽기·표시 계층" 철학을 따릅니다: 서버 데이터를 바꾸지 않고, 화면 상태로 순서만 파생 계산합니다.

## 기능 개요

노트 목록을 **수정일·생성일·제목** 기준 + **오름/내림차순**으로 정렬해 원하는 노트를 빠르게 찾을 수 있다.

## 기능 요구사항

- 목록 상단의 드롭다운으로 정렬 **기준**(수정일·생성일·제목)을 고를 수 있다
- 정렬 **방향**(오름·내림)을 바꿀 수 있다
- 정렬은 타이핑 없이 선택 즉시 목록에 반영된다
- 검색·핀 등 다른 표시 기능과 충돌 없이 함께 동작한다

## 1. 데이터·정렬 상태 모델

- 정렬 기준·방향은 **서버 데이터가 아니라 화면 상태**다. `selectedNoteId`/`searchQuery`와 같은 경계에 두어 **`App.tsx`의 `useState`** 로 보관한다 (ADR-0001).
  - 근거(의견 일치): 기존 규칙(서버 데이터만 Context, 선택·생성·검색 같은 화면 상태는 App)을 그대로 따라야 상태 책임 경계가 깨지지 않는다. Context에 두면 노트 전환·재진입 시 잔여 상태가 남을 위험이 있다. 검색(`searchQuery`)이 이미 같은 자리에 있어 일관된다.
- 정렬은 **원본 `notes` 배열을 바꾸지 않고**, 렌더링 시점에 **정렬된 사본만 계산**한다(원본 불변, ADR-0002).
  - 근거: `NotesContext`는 "서버 데이터의 단일 출처"다. `notes`를 정렬된 결과로 덮어쓰면 `addNote`(append)·`editNote`(map)·`removeNote`(filter)의 응답 기반 갱신이 정렬된 배열을 망가뜨린다. `Array.prototype.sort`는 제자리 정렬(in-place)이므로 반드시 **복사 후 정렬**(`[...notes].sort(...)`)한다.
- 정렬 로직은 **순수 함수 `sortNotes(notes, sortBy, sortDir)`** 로 분리한다. 입력(노트 배열 + 기준 + 방향) → 출력(정렬된 새 배열)이 명확해 단위 테스트와 재사용이 쉽다.

```ts
// src/utils/sortNotes.ts
export type SortBy = 'updatedAt' | 'createdAt' | 'title';
export type SortDir = 'asc' | 'desc';

/**
 * 노트 목록을 기준·방향으로 정렬한다 (원본 불변, 새 배열 반환).
 * @param notes 원본 노트 목록
 * @param sortBy 정렬 기준 (수정일·생성일·제목)
 * @param sortDir 정렬 방향 (오름·내림)
 * @returns 정렬된 새 노트 배열
 */
export function sortNotes(notes: Note[], sortBy: SortBy, sortDir: SortDir): Note[];
```

```ts
// App.tsx — 화면 상태로 보관 (searchQuery와 같은 자리)
const [sortBy, setSortBy] = useState<SortBy>('updatedAt');
const [sortDir, setSortDir] = useState<SortDir>('desc');
```

- **기본값**: `sortBy='updatedAt'`, `sortDir='desc'`(최근 수정 먼저).
  - 근거: 노트 앱의 가장 흔한 멘탈모델은 "방금 손댄 노트가 위로". 기존에 `NoteItem`이 `updatedAt`을 표시하고 있어 시각적으로도 일관된다.

## 2. 입력 동작 (SortControl)

- 정렬 컨트롤은 **노트 목록 바로 위(NoteList 영역 상단)** 에 둔다. 검색창(`SearchBar`)이 이미 그 위치에 있다면 검색창과 목록 사이(또는 검색창과 같은 줄)에 배치한다.
  - 근거: spec의 "목록 상단에 제공"에 정확히 부합하고, 컨트롤과 결과 목록이 시선상 이어져 "순서가 바뀐다"는 인과가 즉시 이해된다.
- 컨트롤은 **`SortControl` 프레젠테이션 컴포넌트로 분리**하고 `sortBy`/`sortDir`과 `onSortByChange`/`onSortDirChange`만 props로 받는다(stateless).
  - 근거: 프로젝트의 "프레젠테이션 컴포넌트는 콜백 props(`onXxx`)로 받는다" 컨벤션과 일치하고, UI를 독립시켜야 라벨·접근성·방향 토글 디테일을 단위 테스트로 검증하기 쉽다.
- **정렬 기준**: 네이티브 `<select>` 드롭다운으로 `수정일 / 생성일 / 제목` 세 옵션을 제공한다.
  - 근거: 옵션이 3개로 고정이고, 네이티브 select는 키보드·접근성이 기본 보장되며 구현·테스트가 가장 단순하다. 커스텀 드롭다운(팝오버)은 포커스 트랩·외부 클릭 닫기라는 숨은 복잡도를 부른다.
- **정렬 방향**: 오름/내림을 바꾸는 **방향 토글 버튼**(`↑`/`↓` 또는 "오름차순/내림차순" 라벨)을 드롭다운 옆에 둔다.
  - 근거: 기준과 방향은 독립 축이라 분리하면 조합이 직관적이다(예: 제목 오름차순 ↔ 제목 내림차순). 6개 옵션을 한 select에 합치면 항목이 늘고 라벨이 장황해진다.
- 선택 즉시 실시간으로 목록을 재정렬한다(확인 버튼 없음). `aria-label`을 드롭다운·토글 버튼에 부여한다.

```ts
// src/components/SortControl.tsx
interface SortControlProps {
  sortBy: SortBy;
  sortDir: SortDir;
  onSortByChange: (sortBy: SortBy) => void;
  onSortDirChange: (sortDir: SortDir) => void;
}
```

## 3. 정렬 규칙 (매칭·비교)

모든 비교는 순수 함수 `sortNotes` 안에서 처리한다.

- **수정일(`updatedAt`) / 생성일(`createdAt`)**: ISO 문자열 날짜를 **시간순으로 비교**한다.
  - `asc`(오름차순) = 과거 → 최신, `desc`(내림차순) = 최신 → 과거.
  - ISO 8601 문자열은 사전식 비교가 곧 시간순 비교라 `Date` 변환 없이 문자열 비교로 충분하다. (가독성을 위해 `localeCompare` 또는 `<` 비교 중 하나로 일관 적용.)
- **제목(`title`)**: 문자열을 **`localeCompare('ko')`** 로 비교한다(한글·영문 자연 정렬).
  - `asc` = 가나다·ABC 순, `desc` = 역순.
  - 근거: 단순 `<` 비교는 한글·대소문자 정렬이 어색하다. `localeCompare('ko')`가 한·영 혼용에 자연스럽다.
  - **제목 빈 값 처리**: `title`이 빈 문자열인 노트는 빈 문자열 그대로 비교한다(빈 제목은 오름차순에서 맨 앞). 목록에서는 `(제목 없음)`으로 보이지만 **정렬 키는 원본 `title`** 을 쓴다(표시 문구를 정렬에 섞지 않음).
- **방향 적용**: 기준 비교 결과에 방향을 곱해 적용한다(`desc`면 비교 결과 부호 반전). 한 곳(`sortNotes`)에서만 방향을 처리해 분기를 단순화한다.
- **안정 정렬(stable)**: 비교 결과가 같을 때(예: 같은 수정일) 원본 순서를 유지한다. 모던 JS `Array.prototype.sort`는 stable이라 추가 처리 불필요.
- **원본 불변**: 반드시 `[...notes].sort(...)`로 **복사본을 정렬**해 반환한다. 원본 `notes`는 절대 변형하지 않는다.

## 4. 표시·핀 섹션

- 정렬 결과는 `NoteList`가 렌더하는 목록에 그대로 반영된다. 검색(`filterNotes`)이 함께 적용된 경우 **"먼저 거르고(filter) 그 결과를 정렬(sort)"** 순서로 합성한다.
  - 근거: 거르기는 "무엇을 보일지", 정렬은 "어떤 순서로 보일지"라 직교한다. filter→sort 순서면 걸러진 결과만 정렬해 비용도 작다.
- **핀(고정) 섹션 호환**: 목록이 "고정 노트 섹션 + 일반 노트 섹션"으로 나뉘는 경우, **각 섹션 내부에 동일한 `sortNotes`를 독립 적용**한다.
  - 섹션 경계(고정 ↔ 일반)는 정렬보다 우선한다 — 고정 노트가 항상 위, 그 안에서 선택한 기준·방향으로 정렬, 일반 노트도 같은 기준·방향으로 정렬.
  - **현재 코드 정합**: 지금 코드베이스에는 핀 기능이 아직 없다. 따라서 이번 구현은 `sortNotes`를 **목록 단위로 적용**하되, 핀 섹션이 도입되면 **섹션별로 같은 함수를 재호출**하기만 하면 되도록 순수 함수로 설계한다(섹션 분리 자체는 핀 기능의 몫이며 이번 범위가 만들지 않는다). 이로써 "핀 섹션 내부에도 동일 정렬"이라는 요구를 함수 설계 차원에서 미리 충족한다.

## 5. 저장·상태 흐름

데이터는 `db.json → JSON Server → api/notes.ts → NotesContext → 컴포넌트` 순으로 흐른다. 정렬은 이 흐름 **위에 얹히는 화면 표시**일 뿐, 서버 데이터를 바꾸지 않는다.

- **데이터 흐름**: `App`이 `sortBy`/`sortDir`을 소유하고 `SortControl`(value/onChange)과 `NoteList`(정렬에 쓸 기준·방향)에 내려준다. `NoteList`는 `useNotes()`의 원본 `notes`를 (검색이 있으면 `filterNotes` 후) `sortNotes(list, sortBy, sortDir)`로 정렬해 렌더한다.
- **선택/생성/검색 시 정렬 유지**: 노트를 열거나 새로 만들거나 검색해도 **정렬 기준·방향은 그대로 둔다**. 정렬은 `selectedNoteId`/`searchQuery`와 독립된 App 화면 상태다.
- **세션 한정**: 정렬 선택은 **이번 세션(메모리) 동안만** 유지한다. 새로고침 시 기본값(`updatedAt`/`desc`)으로 돌아간다 — `localStorage` 영속화는 이번 범위 제외(아래 6 참고).
- **변경 함수 미수정**: `addNote`/`editNote`/`removeNote`와 `api/notes.ts`는 정렬 기능 때문에 손대지 않는다. 정렬은 순수 읽기·표시 계층이다.

## 6. 범위 (포함·제외)

### 이번 범위에 포함

- 수정일·생성일·제목 정렬 기준 (드롭다운)
- 오름/내림차순 방향 토글
- 실시간 재정렬, 원본 불변(복사 후 정렬)
- `sortNotes` 순수 함수 분리 + `SortControl` 프레젠테이션 컴포넌트
- 검색과의 합성(filter → sort) 및 핀 섹션 재사용 가능한 함수 설계

### 이번 범위에서 제외 (다음 이슈)

- **사용자 수동(드래그 앤 드롭) 커스텀 순서** → 별도 데이터 모델·영속화 필요, 큰 슬라이스
- **서버 정렬**(JSON Server `_sort`/`_order` 쿼리 위임) → 클라이언트 메모리 정렬로 충분, 서버 왕복 불필요
- **정렬 설정 영속화**(`localStorage`로 새로고침 후 복원) → 후속 개선
- **추가 정렬 기준**(제목 길이·태그 수·본문 길이 등) → 현재 요구 밖
- **핀 섹션 자체 구현**(고정 토글·고정 데이터 모델) → 핀 기능의 별도 PRD/슬라이스

### 접근성·스타일 기준(이번 범위)

- **접근성 기본 수준**: `<select>`에 `aria-label`(또는 연결된 라벨)을 부여한다. 방향 토글 버튼에도 `aria-label`을 주고 키보드 포커스·활성화가 가능해야 한다.
- **디자인 토큰만 사용**: `bg-card`, `border-border`, `text-muted-foreground` 등 **시맨틱 토큰**만 쓴다. 임의 hex 색·인라인 style·임의 그림자값을 쓰지 않는다(프로젝트 규칙 준수).

## 7. 테스트

- 테스트 도구(Vitest + jsdom + `src/test-setup.ts`)는 갖춰져 있다. 정렬 로직은 입출력이 분명해 단위 테스트 예제로 적합하다(검색·태그 spec과 동일 철학).
- **`sortNotes` 순수 함수 단위 테스트** (핵심):
  - `updatedAt` 내림차순이면 가장 최근 수정 노트가 맨 앞에 온다
  - `updatedAt` 오름차순이면 가장 오래된 노트가 맨 앞에 온다
  - `createdAt` 기준 오름·내림이 생성일 순서대로 정렬한다
  - `title` 오름차순이면 가나다·ABC 순으로 정렬한다(`localeCompare`)
  - `title` 내림차순이면 그 역순으로 정렬한다
  - 빈 제목(`''`) 노트도 에러 없이 정렬된다
  - 같은 값일 때 원본 순서를 유지한다(안정 정렬)
  - **원본 배열을 변형하지 않는다**(원본 불변 — 입력 배열 참조·순서가 그대로)
- **`SortControl` 컴포넌트 입력 테스트**:
  - 드롭다운에서 옵션을 바꾸면 `onSortByChange`가 선택한 기준으로 호출된다
  - 방향 토글을 누르면 `onSortDirChange`가 반대 방향으로 호출된다
  - `sortBy`/`sortDir` prop이 컨트롤 상태에 반영된다
- (선택) **`NoteList` 통합 테스트**: 검색과 정렬이 함께 걸린 상태에서 "걸러진 뒤 정렬된" 순서로 렌더된다.
