# 목록 정렬 토글 기능 이슈 분해 (Vertical Slicing)

[`PRD.md`](PRD.md)를 기반으로 한 구현 이슈 목록입니다.

## 분해 원칙 — 수직 슬라이싱

각 이슈는 **데이터 → 순수 함수 → UI를 관통하는, 그 자체로 동작하고 사용자에게 보이는 한 조각**입니다.
PRD의 마일스톤(M1~M4)은 계층 예고라, "사용자가 끝까지 쓸 수 있는 단위"로 재구성했습니다.

- 먼저 **walking skeleton**(SORT-1: 수정일 정렬이 드롭다운으로 화면 끝까지 동작)으로 전 계층을 얇게 관통한 뒤,
- 정렬 기준 확장(SORT-2) → 방향 토글(SORT-3) → 검색과의 합성·핀 재사용(SORT-4) 순으로 가치를 덧붙입니다.

## 각 이슈의 구성 — AC와 DoD 구분

- **인수 기준(AC, Acceptance Criteria)** = "**올바르게 동작**하는가?" 기능별 행위 기준. **Given/When/Then**으로 표현.
- **완료 정의(DoD, Definition of Done)** = "**정말 끝났는가?**" 품질·프로세스 게이트. 행위를 다시 적지 않고 완료 조건만.

> 행위(무엇이 맞아야 하나)는 AC에서, 품질(무엇이 갖춰져야 끝인가)은 DoD에서 다룹니다. 둘은 겹치지 않습니다.

## 이슈 ↔ 요구사항(FR) 매핑

| 이슈                                     | 담는 FR                | 의존   |
| ---------------------------------------- | ---------------------- | ------ |
| SORT-1 수정일 정렬(walking skeleton)     | FR-1, FR-3, FR-6, FR-7 | —      |
| SORT-2 정렬 기준 확장(생성일·제목)       | FR-4, FR-5             | SORT-1 |
| SORT-3 오름/내림 방향 토글               | FR-2                   | SORT-1 |
| SORT-4 검색 합성 + 정렬 유지 + 핀 재사용 | FR-8, FR-9, FR-10      | SORT-1 |

> SORT-1이 모든 이슈의 선행입니다. SORT-2·3·4는 SORT-1 이후 병렬 진행 가능.

## SORT-1 — 수정일 정렬이 드롭다운으로 목록 끝까지 동작한다

**라벨**: `feature/sort` `P0` `slice:happy-path`
**의존**: 없음 (선행 이슈)

### 설명

정렬 기능의 **walking skeleton**. 목록 상단 드롭다운에서 "수정일"을 고르면 `App`의 화면 상태(`sortBy`)가 바뀌고, `NoteList`가 순수 함수 `sortNotes`로 원본을 복사해 정렬한 목록을 렌더한다. 이 한 슬라이스로 `App`(상태) → `SortControl`(UI) → `sortNotes`(순수 함수) → `NoteList`(렌더)까지 전 계층을 얇게 관통한다.

> **얇게 유지**: 이번엔 **기준은 "수정일" 하나, 방향은 기본값 `desc` 고정**이다. 방향을 바꾸는 컨트롤·로직은 SORT-3의 몫이므로 이 슬라이스의 AC는 "방향이 어떻게 동작하는가"가 아니라 **"고른 기준으로 정렬이 적용되어 화면에 반영되는가"** 에만 집중한다. (생성일·제목 기준은 SORT-2, 방향 토글은 SORT-3, 검색 합성은 SORT-4)

### 범위

- `src/utils/sortNotes.ts` (신규): `sortNotes(notes, sortBy, sortDir)` 순수 함수 골격 — 이번엔 `updatedAt` 기준 + 기본 방향만, **반드시 `[...notes].sort`로 원본 불변** (ADR-0002).
- `src/components/SortControl.tsx` (신규): 정렬 기준 `<select>` 드롭다운 프레젠테이션 컴포넌트(`sortBy`/`onSortByChange` props). 이번엔 "수정일" 옵션이 동작(다른 옵션 자리는 SORT-2에서 채움).
- `src/App.tsx`: `sortBy` 화면 상태 추가(기본 `updatedAt`), `SortControl`에 주입, `NoteList`에 정렬 기준 전달 (ADR-0001).
- `src/components/NoteList.tsx`: `sortNotes`로 정렬해 렌더.

### 인수 기준 (AC)

```
시나리오 A: 수정일 정렬이 목록에 반영
  Given 수정일이 서로 다른 노트가 여러 개이고, 목록이 db 순서(미정렬) 그대로 보인다
  When 상단 드롭다운에서 "수정일"을 선택한다
  Then 기본 정렬(최근 수정 먼저)로 목록이 재정렬되어, db 순서와 다른 순서로 보인다
  (방향 자체를 바꾸는 동작은 SORT-3 범위 — 이 슬라이스는 "정렬이 적용된다"까지)

시나리오 B: 선택 즉시 반영
  Given 노트 목록이 보인다
  When 정렬 드롭다운에서 기준을 고른다
  Then 확인 버튼 없이 선택과 동시에 목록 순서가 바뀐다

시나리오 C: 원본 불변
  Given 정렬을 적용한다
  When 정렬된 목록을 렌더한다
  Then NotesContext의 원본 notes 배열 내용·순서가 변형되지 않는다(sortNotes는 복사본을 정렬해 새 배열을 반환)

시나리오 D: 빈 목록·단일 노트
  Given 노트가 0개 또는 1개다
  When 정렬을 적용한다
  Then 에러 없이 그대로 표시된다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~D)가 모두 통과한다.
- [ ] `sortNotes`가 순수 함수로 분리되고 원본 불변(복사 후 정렬) 단위 테스트가 통과한다.
- [ ] 디자인 시스템 토큰 사용(임의 색/그림자/인라인 style 없음), lint·build·`tsc` 통과.
- [ ] 기존 노트 CRUD·태그·검색 동작에 회귀가 없다.
- [ ] 코드 리뷰 후 머지된다.

## SORT-2 — 생성일·제목으로도 정렬할 수 있다

**라벨**: `feature/sort` `P0` `slice:sort-keys`
**의존**: SORT-1

### 설명

SORT-1의 수정일 정렬에 **생성일(`createdAt`)·제목(`title`)** 기준을 더한다. 드롭다운에 세 옵션이 모두 동작하고, 제목은 `localeCompare('ko')`로 한·영 자연 정렬한다(ADR-0002).

### 범위

- `sortNotes`: `sortBy`에 `createdAt`·`title` 분기 추가. 날짜는 ISO 문자열 비교, 제목은 `localeCompare('ko')`. 빈 제목(`''`)은 빈 문자열 그대로 비교.
- `SortControl`: 드롭다운에 "생성일", "제목" 옵션 추가(총 3개).
- Vitest 단위 테스트 추가(생성일·제목·빈 제목·안정 정렬).

### 인수 기준 (AC)

```
시나리오 A: 생성일 정렬
  Given 생성일이 서로 다른 노트가 있다
  When 드롭다운에서 "생성일"을 선택한다
  Then createdAt 순서대로 목록이 재정렬된다

시나리오 B: 제목 가나다·ABC 정렬
  Given 제목이 "바나나", "apple", "가지"인 노트가 있다
  When "제목"을 선택한다(오름차순 기준)
  Then localeCompare('ko') 기준 가나다·ABC 순으로 정렬된다

시나리오 C: 빈 제목 처리
  Given 제목이 빈 문자열인 노트가 섞여 있다
  When "제목"으로 정렬한다
  Then 에러 없이 빈 제목 노트도 정렬에 포함된다

시나리오 D: 안정 정렬
  Given 같은 기준 값(예: 동일 생성일)을 가진 노트가 둘 있다
  When 그 기준으로 정렬한다
  Then 두 노트의 원래 상대 순서가 유지된다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~D)가 모두 통과한다.
- [ ] 생성일·제목·빈 제목·안정 정렬에 대한 단위 테스트가 통과한다.
- [ ] 디자인 토큰 사용, lint·build·`tsc` 통과.
- [ ] 코드 리뷰 후 머지된다.

## SORT-3 — 오름/내림차순 방향을 토글할 수 있다

**라벨**: `feature/sort` `P0` `slice:direction`
**의존**: SORT-1

### 설명

정렬 기준은 그대로 두고 **방향(오름·내림)** 만 뒤집는 토글 버튼을 드롭다운 옆에 둔다. `App`의 `sortDir` 화면 상태가 바뀌면 같은 기준에서 순서가 반전된다.

### 범위

- `src/App.tsx`: `sortDir` 화면 상태 추가(기본 `desc`), `SortControl`·`NoteList`에 전달.
- `SortControl`: 방향 토글 버튼(`↑`/`↓` 또는 오름/내림 라벨) 추가, `onSortDirChange`로 반전 통지. `aria-label` 부여.
- `sortNotes`: 이미 SORT-1에서 `sortDir`을 받으므로 비교 결과 부호 반전이 동작하도록 확정.

### 인수 기준 (AC)

```
시나리오 A: 방향 반전
  Given 수정일 내림차순(최신 먼저)으로 정렬돼 있다
  When 방향 토글을 누른다
  Then 같은 기준에서 오름차순(오래된 먼저)으로 순서가 뒤집힌다

시나리오 B: 기준과 독립
  Given 제목 오름차순으로 정렬돼 있다
  When 방향만 토글한다
  Then 정렬 기준은 "제목"으로 유지되고 방향만 내림차순으로 바뀐다

시나리오 C: 키보드 접근성
  Given 방향 토글 버튼에 포커스가 있다
  When 키보드로 버튼을 활성화한다
  Then 방향이 반전된다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~C)가 모두 통과한다.
- [ ] 방향 토글에 `aria-label`이 있고 키보드 포커스·활성화가 가능하다(접근성).
- [ ] 방향 반전 단위 테스트(`SortControl` onChange + `sortNotes` 부호 반전)가 통과한다.
- [ ] 디자인 토큰 사용, lint·build·`tsc` 통과.
- [ ] 코드 리뷰 후 머지된다.

## SORT-4 — 검색과 함께, 그리고 화면을 오가도 정렬이 유지된다

**라벨**: `feature/sort` `P1` `slice:compose`
**의존**: SORT-1

### 설명

정렬을 다른 표시 기능과 **충돌 없이 합성**한다. 검색이 걸리면 **거른 뒤 정렬**(filter → sort)하고, 노트를 열거나 새로 만들거나 검색해도 정렬 기준·방향을 **유지**한다. 또한 `sortNotes`가 **핀 섹션에도 재사용** 가능한 순수 함수임을 확인한다(섹션 분리 자체는 핀 기능의 몫 — 이번엔 함수 재사용성만 보장).

### 범위

- `src/components/NoteList.tsx`: 검색(`filterNotes`)이 적용된 경우 `filterNotes` 결과를 `sortNotes`로 정렬(filter → sort 순서).
- `App`의 `sortBy`/`sortDir`이 `selectedNoteId`/`searchQuery` 변경에 영향받지 않음을 확인(독립 상태).
- `sortNotes` 재사용성 테스트: 임의 부분 배열(예: 검색 결과·핀 후보)을 넣어도 동일 규칙으로 정렬됨을 확인.

### 인수 기준 (AC)

```
시나리오 A: 검색과 정렬 합성
  Given 검색어로 일부 노트만 걸러진 상태다
  When 정렬 기준·방향을 고른다
  Then 걸러진 결과만 그 기준·방향으로 정렬되어 보인다(전체가 아니라 검색 결과 내 정렬)

시나리오 B: 노트 선택 후 정렬 유지
  Given 제목 오름차순으로 정렬돼 있다
  When 목록에서 노트를 하나 열었다가 돌아온다
  Then 정렬 기준·방향이 그대로 유지된다

시나리오 C: 새 노트 생성 후 정렬 유지
  Given 특정 정렬이 적용돼 있다
  When 새 노트를 만들고 저장한다
  Then 정렬 기준·방향이 유지되고, 새 노트도 그 기준에 맞는 위치에 배치된다
```

> **핀 섹션 재사용성**(FR-10)은 사용자에게 보이는 화면 행위가 아니라 `sortNotes`의 함수 속성이므로 AC(행위)가 아닌 **DoD의 품질 게이트**로 검증한다(아래 참고). 핀 섹션 자체 구현은 핀 기능의 별도 슬라이스다.

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~C)가 모두 통과한다.
- [ ] 검색 합성(filter → sort) 및 정렬 유지에 대한 테스트가 통과한다.
- [ ] **`sortNotes`가 임의 노트 부분배열(핀 섹션 가정)에 동일 규칙으로 재사용 가능함**이 단위 테스트로 보장된다(섹션별 독립 적용 대비, FR-10).
- [ ] 기존 검색·태그·CRUD 동작에 회귀가 없다.
- [ ] 디자인 토큰 사용, lint·build·`tsc` 통과.
- [ ] 코드 리뷰 후 머지된다.
