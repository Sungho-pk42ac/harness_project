## 1. TL;DR

노트 목록을 **수정일·생성일·제목** 기준과 **오름/내림차순** 방향으로 정렬하는 드롭다운을 목록 상단에 도입한다. 정렬은 **서버 변경 없이** 클라이언트 메모리에서 순수 함수로 순서만 파생 계산한다. 이번 범위는 **기준·방향 선택과 실시간 재정렬**까지이며, **수동 커스텀 순서·서버 정렬·정렬 설정 영속화는 다음 릴리스로 분리**한다.

## 2. 배경 & 문제 (Background & Problem)

- 현재 노트 목록은 `db.json`이 반환한 **순서 그대로** 렌더된다. 노트가 쌓이면 "방금 고친 노트"나 "제목으로 찾는 노트"를 위치로 기억할 수 없어 탐색이 어렵다.
- 검색(v0.2)이 "거르기"를 해결했지만, 거른 결과 **안에서의 순서**를 사용자가 통제할 수단이 없다. 검색과 정렬은 직교한 두 축이다.
- 데이터는 이미 클라이언트 메모리(`NotesContext`의 `notes`)에 전부 있으므로, 서버 왕복 없이 **화면 정렬**만으로 즉시 가치를 줄 수 있다.
- **왜 지금**: 태그·검색 기반이 갖춰진 지금, 정렬은 "데이터를 읽어 순서만 바꾸기"라는 가장 단순한 표시 계층 기능이라 적은 비용으로 탐색성을 크게 높인다.

## 3. 목표 & 비목표 (Goals / Non-Goals)

### Goals

- G1. 목록 상단 드롭다운으로 정렬 기준(수정일·생성일·제목)을 고르면 즉시 목록 순서가 바뀐다.
- G2. 오름/내림차순 방향을 토글해 같은 기준의 정렬 방향을 뒤집을 수 있다.
- G3. 정렬은 원본 노트 데이터를 변형하지 않고 순서만 파생 계산한다(검색·핀과 충돌 없음).

### Non-Goals (이번 릴리스 제외)

- N1. **사용자 수동(드래그) 커스텀 순서** (별도 데이터 모델·영속화 — 별도 PRD).
- N2. **서버 정렬** (JSON Server `_sort`/`_order` 위임).
- N3. **정렬 설정 영속화** (`localStorage`로 새로고침 후 복원) — 후속 개선.
- N4. **추가 정렬 기준** (제목 길이·태그 수·본문 길이 등).
- N5. **핀(고정) 섹션 자체 구현** — 핀 기능의 별도 PRD. (단, 정렬 함수는 핀 섹션에 재사용 가능하도록 설계.)

## 4. 성공 지표 (Success Metrics)

| 지표        | 정의                                          | 목표                     |
| ----------- | --------------------------------------------- | ------------------------ |
| 정렬 정확성 | 선택한 기준·방향대로 순서가 정확히 정렬되는가 | 단위 테스트 100% 통과    |
| 즉시성      | 선택 후 목록 재정렬 체감 지연                 | 0 (동기 정렬, 데모 기준) |
| 원본 불변   | 정렬이 `NotesContext.notes`를 변형하지 않는가 | 단위 테스트로 검증       |
| 회귀        | 기존 노트 CRUD·태그·검색 동작 영향            | 회귀 0                   |

> 소규모 학습 프로젝트이므로 정량 트래킹 인프라는 없다. 지표는 **수용 기준(AC)과 단위 테스트·수동 점검**으로 대체한다.

## 5. 사용자 & 유스케이스

- **페르소나**: 노트를 다수 작성·관리하는 단일 사용자(로컬 앱).
- **핵심 유스케이스**
  - U1. 방금 고친 노트를 위로 올려 이어서 작업한다(수정일 내림차순 — 기본).
  - U2. 가장 먼저 만든 노트부터 훑어본다(생성일 오름차순).
  - U3. 제목 가나다 순으로 정렬해 이름으로 노트를 찾는다.
  - U4. 같은 기준에서 방향만 뒤집어 반대 끝의 노트를 본다.

| FR-2  | 오름/내림차순 방향을 토글한다              | P0       | 방향 토글을 누르면 같은 기준에서 순서가 뒤집힌다                                   |
| FR-3  | 수정일 기준으로 정렬한다                   | P0       | `updatedAt` 시간순으로 정렬(내림=최신 먼저, 오름=오래된 먼저)                      |

| FR-6  | 선택 즉시 실시간으로 재정렬한다            | P0       | 확인 버튼 없이 선택과 동시에 목록이 재정렬된다                                     |
| FR-7  | 원본 데이터를 변형하지 않는다              | P0       | 정렬은 파생 계산만 — `NotesContext.notes`는 불변(복사 후 정렬) (ADR-0002)          |
| FR-8  | 검색과 함께 동작한다                       | P0       | 검색 결과를 먼저 거른 뒤 정렬한다(filter → sort)                                   |
| FR-9  | 선택/생성/검색 중에도 정렬 설정을 유지한다 | P1       | 노트를 열거나 새로 만들거나 검색해도 정렬 기준·방향이 유지된다 (ADR-0001)          |
| FR-10 | 핀 섹션 내부에도 동일 정렬을 적용한다      | P1       | 핀 섹션이 있으면 각 섹션에 같은 `sortNotes`를 독립 적용(섹션 경계는 정렬보다 우선) |

> FR-10은 현재 핀 기능이 없으므로 "핀 섹션이 도입되면 재사용되는 함수 설계"로 충족한다. 섹션 분리 자체는 핀 기능의 몫(Non-Goal N5).

## 7. 비기능 요구사항 (NFR)

- **아키텍처 정합**: 정렬 기준·방향은 **화면 상태**이므로 `App.tsx`의 `useState`에 둔다(`searchQuery`와 같은 경계). 서버 데이터(`NotesContext`)와 섞지 않는다 (ADR-0001).
- **원본 불변**: `Array.prototype.sort`는 in-place라 반드시 `[...notes].sort(...)`로 복사 후 정렬한다 (ADR-0002).
- **호환성**: `tags`처럼 누락 가능한 필드는 없다 — `createdAt`/`updatedAt`/`title`은 모든 노트가 보유. 빈 `title`은 빈 문자열로 안전 비교.
- **성능**: 수십~수백 건 동기 `sort`는 체감 지연 0. 데이터가 커지면 메모이제이션(`useMemo`)·서버 정렬을 별도 도입.
- **접근성**: `<select>`와 방향 토글 버튼에 `aria-label`, 키보드 포커스·활성화 가능.
- **스타일**: 시맨틱 토큰(`bg-card`, `border-border`, `text-muted-foreground`)만 사용. 임의 색/그림자/인라인 style 금지.

## 8. UX / 인터랙션 흐름

```
[정렬 기준 변경]
드롭다운 선택(수정일|생성일|제목) → sortBy 갱신 → sortNotes(list, sortBy, sortDir) → 재정렬된 목록

[방향 토글]
↑/↓ 버튼 클릭 → sortDir(asc↔desc) 반전 → 재정렬된 목록

[검색과 합성]
원본 notes → filterNotes(검색어) → sortNotes(기준·방향) → 화면 목록
```

`NoteList` 렌더 파이프라인: `loading` → `error` → 원본 0건 → (검색 0건) → **filter → sort 결과 렌더**.

## 9. 데이터 / 상태 모델 (개요)

- 정렬 상태 위치: **[ADR-0001](adr/0001-sort-state-location.md)** — `App.tsx`의 `sortBy: SortBy`, `sortDir: SortDir`.
- 정렬 전략: **[ADR-0002](adr/0002-client-pure-sort.md)** — 순수 함수 `sortNotes`, 원본 불변(복사 후 정렬), 클라이언트 실시간.

```ts
// src/utils/sortNotes.ts
export type SortBy = 'updatedAt' | 'createdAt' | 'title';
export type SortDir = 'asc' | 'desc';
export function sortNotes(notes: Note[], sortBy: SortBy, sortDir: SortDir): Note[];

// App.tsx
const [sortBy, setSortBy] = useState<SortBy>('updatedAt');
const [sortDir, setSortDir] = useState<SortDir>('desc');
```

## 13. 의존성

- 검색 기능(v0.2): 정렬은 검색 결과 위에 얹힌다(filter → sort). 검색이 없어도 단독 동작.
- `NotesContext`의 `notes`(단일 출처)와 `App.tsx`의 화면 상태 경계.
- (향후) 핀 기능: 핀 섹션이 생기면 `sortNotes`를 섹션별로 재사용 — 현재는 함수 설계로만 대비.
