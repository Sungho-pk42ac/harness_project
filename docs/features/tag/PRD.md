## 1. TL;DR

노트에 **태그를 추가/삭제/표시**하는 기능을 도입한다. 사용자가 노트를 주제별로 분류·식별할 수 있게 하는 첫걸음이다. 이번 범위는 **태그 입력·표시·검증**까지이며, **태그 기반 필터·검색은 다음 릴리스로 분리**한다.

## 2. 배경 & 문제 (Background & Problem)

- 현재 노트는 `title`·`content`만 가진다. 노트가 늘어나면 **분류·식별 수단이 없어** 원하는 노트를 찾기 어렵다.
- `src/types/note.ts`에는 `tags` 자리가 의도적으로 비어 있다(강의용 미완성 지점). 이를 실제 기능으로 완성한다.
- **왜 지금**: 노트 CRUD 기본 흐름이 안정적이므로, 분류 체계의 최소 단위인 "태그"를 먼저 세워 이후 검색/필터/관련노트 기능의 토대를 만든다.

## 3. 목표 & 비목표 (Goals / Non-Goals)

### Goals

- G1. 노트 편집 화면에서 태그를 직관적으로 추가/삭제할 수 있다.
- G2. 노트 목록·상세에서 태그를 한눈에 확인할 수 있다.
- G3. 태그 데이터가 일관성 있게(중복 없이, 정규화되어) 저장된다.

### Non-Goals (이번 릴리스 제외)

- N1. 태그 기반 **필터/검색** (다음 릴리스 — 별도 PRD).
- N2. 태그 **자동완성·추천**.
- N3. 태그 **이름 변경·일괄 관리(태그 관리 페이지)**.
- N4. 태그 **색상 커스터마이즈**(태그별 사용자 지정 색).

## 4. 성공 지표 (Success Metrics)

| 지표                | 정의                                        | 목표                            |
| ------------------- | ------------------------------------------- | ------------------------------- |
| 태그 채택률         | 신규/수정 노트 중 1개 이상 태그를 가진 비율 | 정성 관찰(강의 데모 기준)       |
| 노트당 평균 태그 수 | 태그가 있는 노트의 평균 태그 개수           | 1~3개                           |
| 저장 오류율         | 태그 포함 저장 요청 중 실패 비율            | 기존 노트 저장과 동일 수준 유지 |
| 데이터 정합성       | 중복/공백 태그가 저장된 노트 수             | 0                               |

> 소규모 학습 프로젝트이므로 정량 트래킹 인프라는 없다. 지표는 **수용 기준(AC)과 수동 점검**으로 대체한다.

## 5. 사용자 & 유스케이스

- **페르소나**: 노트를 다수 작성·관리하는 단일 사용자(로컬 앱).
- **핵심 유스케이스**
  - U1. 새 노트를 작성하며 주제 태그를 단다. (예: `회의`, `아이디어`)
  - U2. 기존 노트를 열어 태그를 추가하거나 잘못 단 태그를 뺀다.
  - U3. 목록을 훑으며 각 노트의 태그로 성격을 빠르게 파악한다.

## 6. 기능 요구사항 (Functional Requirements)

우선순위: **P0**=이번 릴리스 필수, **P1**=가능하면 포함.

| ID    | 요구사항                         | 우선순위 | 수용 기준(AC)                                                     |
| ----- | -------------------------------- | -------- | ----------------------------------------------------------------- |
| FR-1  | 편집기에서 태그를 추가한다       | P0       | 입력칸에 텍스트 후 **Enter 또는 쉼표(,)** 입력 시 칩으로 추가된다 |
| FR-2  | 추가된 태그를 개별 삭제한다      | P0       | 각 칩의 **× 버튼** 클릭 시 해당 태그만 제거된다                   |
| FR-3  | 태그를 정규화한다                | P0       | 저장 전 **앞뒤 공백 제거 + 영문 소문자화** 적용 (ADR-0002)        |
| FR-4  | 빈 태그를 무시한다               | P0       | 정규화 후 빈 문자열이면 추가되지 않는다                           |
| FR-5  | 중복 태그를 막는다               | P0       | 이미 존재하는 태그 입력 시 추가 안 함 + **안내 alert** 노출       |
| FR-6  | 태그 개수를 제한한다             | P0       | 한 노트당 **최대 5개**. 초과 입력은 추가되지 않는다               |
| FR-7  | 목록(NoteItem)에 태그를 표시한다 | P0       | 태그를 칩으로 표시, 많으면 **줄바꿈**, 없으면 미표시              |
| FR-8  | 상세/편집에 태그를 표시한다      | P0       | 편집 화면에서 현재 태그 목록이 칩으로 보인다                      |
| FR-9  | 태그를 생성/수정 저장에 반영한다 | P0       | 저장 시 서버에 `tags`가 포함되어 영속화된다 (ADR-0003)            |
| FR-10 | 저장 실패를 사용자에게 알린다    | P0       | 실패 시 기존 패턴(`try/catch` + alert)으로 알림                   |

## 7. 비기능 요구사항 (NFR)

- **일관성**: 색·형태는 [디자인 시스템](../../design-system/README.md) 토큰을 따른다(칩은 `bg-primary-subtle text-primary` 또는 중립 `muted`). 임의 색/그림자 금지.
- **호환성**: 기존 `db.json` 노트는 `tags` 필드가 없다. 읽을 때 `note.tags ?? []`로 처리하여 마이그레이션 없이 동작한다 (ADR-0001).
- **성능**: 태그 수 상한(5)이 작아 렌더/검증 비용은 무시 가능.
- **접근성**: 칩 삭제 버튼은 키보드 포커스·클릭 가능해야 한다.

## 8. UX / 인터랙션 흐름

```
[태그 추가]
입력 → (Enter | ,) → 정규화(trim+소문자) → 빈값? ─yes→ 무시
                                          └no→ 중복? ─yes→ alert, 무시
                                                  └no→ 5개 초과? ─yes→ 무시
                                                          └no→ 칩 추가
[태그 삭제]
칩의 × 클릭 → 해당 태그 제거
[저장]
저장 버튼 → addNote/editNote에 tags 포함 → 서버 반영 → 실패 시 alert
```

## 9. 데이터 모델 (개요)

- `Note.tags: string[]` 추가, 기본값 `[]`. 상세 근거는 **[ADR-0001](adr/0001-tag-data-model.md)**.
- 정규화·중복·개수 정책: **[ADR-0002](adr/0002-tag-normalization-policy.md)**.
- API/Context 반영 방식: **[ADR-0003](adr/0003-addnote-api-extension.md)**.

```ts
export interface Note {
  id: string;
  title: string;
  content: string;
  createdAt: string;
  updatedAt: string;
  tags: string[]; // 신규
}
```
