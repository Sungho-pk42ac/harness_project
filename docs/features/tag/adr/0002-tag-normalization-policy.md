# ADR-0002 — 태그 정규화·중복·개수 정책

- **상태(Status)**: Accepted
- **날짜**: 2026-05-30
- **관련**: [PRD §6 FR-3~FR-6](../PRD.md#6-기능-요구사항-functional-requirements), [ADR-0001](0001-tag-data-model.md)

## Context

태그는 사용자 자유 입력이다. 아무 가공 없이 저장하면 다음 문제가 생긴다.

- `회의 `, `회의`처럼 공백 차이로 **사실상 같은 태그가 다르게** 저장됨.
- `Work`, `work`가 **다른 태그로 취급**되어 중복 방지가 무력화됨.
- 빈 문자열·과도한 개수로 **데이터가 지저분**해짐.

PRD는 "중복 불허"를 요구한다. 중복 판정이 의미 있으려면 **판정 이전에 정규화 기준**이 있어야 한다.

## Decision

태그를 추가하기 직전, 아래 순서로 처리한다.

1. **정규화**: `trim()`으로 앞뒤 공백 제거 + 영문 **소문자화**(`toLowerCase()`). (한글은 영향 없음)
2. **빈 값 무시**: 정규화 결과가 빈 문자열이면 추가하지 않는다.
3. **중복 검사**: 정규화된 값이 이미 목록에 있으면 추가하지 않고 **안내 `alert`** 를 노출한다.
4. **개수 제한**: 노트당 태그는 **최대 5개**. 초과 시 추가하지 않는다.

확정 키는 **Enter 또는 쉼표(`,`)**, 삭제는 칩의 **× 버튼**.

## Consequences

**긍정**

- 소문자 정규화로 `Work`/`work`가 한 태그로 합쳐져 중복 방지가 실제로 동작한다.
- 빈 값·개수 상한으로 목록이 깔끔하게 유지된다.
- 규칙이 한 곳(추가 직전)에 모여 단위 테스트하기 쉽다 → PRD M4.

**부정 / 트레이드오프**

- 소문자 강제는 영문 **고유명사 대문자 표기**(예: `iOS`)를 보존하지 못한다. MVP에선 일관성을 우선해 수용.
- 중복 시 `alert`는 다소 거슬릴 수 있다(PRD R1). 사용성 이슈가 확인되면 인라인 메시지로 전환을 별도 ADR로 검토.

## Alternatives Considered

1. **정규화 없이 원문 저장 + 정확 일치 중복검사**
   - 기각: `Work`/`work`, 공백 차이를 못 걸러 중복 방지가 형식적이 됨.
2. **중복 시 조용히 무시(알림 없음)**
   - 기각 사유: 사용자가 "왜 안 들어가지?"를 모름. 명시적 피드백(alert)을 선택. (스펙 확정 단계에서 사용자가 alert 선호)
3. **개수 무제한**
   - 기각: UI 도배·가독성 저하 우려. 상한 5로 시작하고 필요 시 조정.
