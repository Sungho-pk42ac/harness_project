# ADR-0001 — 태그 데이터 모델: note 내 `string[]` 임베드

- **상태(Status)**: Accepted
- **날짜**: 2026-05-30
- **관련**: [PRD §9](../PRD.md#9-데이터-모델-개요), [ADR-0002](0002-tag-normalization-policy.md)

## Context

노트에 태그를 붙이려면 태그를 어디에 어떤 형태로 저장할지 결정해야 한다. 제약은 다음과 같다.

- 저장소는 JSON Server(`db.json`)로 스키마가 자유롭고, 관계형 조인이 없다.
- 기존 노트에는 `tags` 필드가 없다(마이그레이션 부담 최소화 필요).
- 현재 범위에는 태그 기반 필터/검색이 **없다**(Non-Goal). 즉, 태그를 "독립 엔티티"로 다룰 강한 이유가 아직 없다.

## Decision

태그를 **노트 문서 안에 문자열 배열로 임베드**한다.

```ts
interface Note {
  // ...
  tags: string[]; // 기본값 []
}
```

- 태그가 없는 노트의 기본값은 **빈 배열 `[]`** 로 통일한다.
- 기존 데이터 호환을 위해, 읽는 지점에서 **`note.tags ?? []`** 로 방어한다(서버/기존 문서 일괄 수정 없음).

## Consequences

**긍정**

- 구현이 단순하다(조인·별도 컬렉션 없음). 노트 한 건을 읽으면 태그가 함께 온다.
- 마이그레이션이 필요 없다 — 신규 저장부터 자연스럽게 `tags`가 채워진다.
- "태그 없음 = 빈 배열" 불변식 덕에 화면에서 `tags.length` 등을 안전하게 사용.

**부정 / 트레이드오프**

- 태그를 1급 엔티티로 다루는 기능(전역 태그 목록, 사용 횟수 집계, 태그 이름 일괄 변경)은 이 모델로는 비효율적이다 → 해당 요구가 생기면 **별도 ADR로 정규화 모델을 재검토**한다.
- 같은 태그 문자열이 여러 노트에 중복 저장된다(비정규화). 현재 규모에선 무시 가능.

## Alternatives Considered

1. **별도 태그 엔티티 + 노트-태그 관계** (`/tags`, `noteTags` 조인)
   - 장점: 전역 태그 관리·집계에 유리.
   - 기각 이유: 현재 범위(필터·관리 없음)에 과도한 복잡성. JSON Server에서 조인 비용·정합성 관리 부담. YAGNI.
2. **쉼표로 이은 단일 문자열**(`tags: "a,b,c"`)
   - 기각 이유: 개별 추가/삭제·중복검사 시 매번 파싱/조립 필요. 배열보다 다루기 나쁨.
