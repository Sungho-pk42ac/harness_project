# 태그 기능 이슈 분해 (Vertical Slicing)

[`PRD.md`](PRD.md)를 기반으로 한 구현 이슈 목록입니다.

## 분해 원칙 — 수직 슬라이싱

각 이슈는 **데이터 → API/Context → UI를 관통하는, 그 자체로 동작하고 사용자에게 보이는 한 조각**입니다.
PRD의 마일스톤(M1~M4)은 계층별 수평 분해라 그대로 쓰지 않고, "사용자가 끝까지 쓸 수 있는 단위"로 재구성했습니다.

- 먼저 **walking skeleton**(TAG-1: 태그를 달고 저장하면 다시 보인다)으로 전 계층을 얇게 관통한 뒤,
- 삭제(TAG-2) → 입력 견고화(TAG-3) → 목록 노출(TAG-4) 순으로 가치를 덧붙입니다.

## 각 이슈의 구성 — AC와 DoD 구분

- **인수 기준(AC, Acceptance Criteria)** = "**올바르게 동작**하는가?" 기능별 행위 기준. **Given/When/Then**으로 표현.
- **완료 정의(DoD, Definition of Done)** = "**정말 끝났는가?**" 품질·프로세스 게이트. 행위를 다시 적지 않고 완료 조건만.

> 행위(무엇이 맞아야 하나)는 AC에서, 품질(무엇이 갖춰져야 끝인가)은 DoD에서 다룹니다. 둘은 겹치지 않습니다.

## 이슈 ↔ 요구사항(FR) 매핑

| 이슈                        | 담는 FR                 | 의존  |
| --------------------------- | ----------------------- | ----- |
| TAG-1 태그 추가 & 영속화    | FR-1, FR-8, FR-9, FR-10 | —     |
| TAG-2 태그 삭제             | FR-2                    | TAG-1 |
| TAG-3 태그 입력 검증·정규화 | FR-3, FR-4, FR-5, FR-6  | TAG-1 |
| TAG-4 목록에 태그 표시      | FR-7                    | TAG-1 |

> TAG-1이 모든 이슈의 선행입니다. TAG-2·3·4는 TAG-1 이후 병렬 진행 가능.

## TAG-1 — 노트에 태그를 달고 저장하면 다시 열었을 때 보인다

**라벨**: `feature/tag` `P0` `slice:happy-path`
**의존**: 없음 (선행 이슈)

### 설명

태그 기능의 **walking skeleton**. 사용자가 편집기에서 태그를 입력해 칩으로 추가하고, 저장하면 서버에 영속화되며, 노트를 다시 열었을 때 그 태그가 그대로 보인다. 이 한 슬라이스로 `Note` 타입 → `api`/`Context` → `NoteEditor`까지 전 계층을 얇게 관통한다. (정규화·중복·개수 제한은 TAG-3, 삭제는 TAG-2, 목록 노출은 TAG-4)

### 범위

- `src/types/note.ts`: `tags: string[]` 추가.
- 읽기 호환: 기존 노트는 `note.tags ?? []`로 처리 (ADR-0001).
- `NotesContext.addNote(title, content, tags)` 시그니처 확장, `editNote`는 `tags` 포함 전달 (ADR-0003).
- `NoteEditor`: 태그 입력칸 + 현재 태그 칩 렌더 + **Enter/쉼표로 추가**, 저장 시 `tags` 반영, 실패 시 기존 `try/catch`+`alert`.

### 인수 기준 (AC)

```
시나리오 A: 새 노트에 태그를 달고 저장
  Given 새 노트를 작성 중이고 제목을 입력했다
  When 태그 입력칸에 "회의"를 적고 Enter를 누른 뒤 저장한다
  Then "회의" 칩이 편집기에 보이고, 저장 후 그 노트를 다시 열면 "회의" 태그가 그대로 보인다

시나리오 B: 쉼표로 태그 추가
  Given 노트를 편집 중이다
  When 태그 입력칸에 "아이디어,"를 입력한다
  Then "아이디어" 칩이 추가된다

시나리오 C: 기존(tags 없는) 노트 호환
  Given tags 필드가 없는 기존 노트가 있다
  When 그 노트를 연다
  Then 에러 없이 태그가 0개인 상태로 표시된다

시나리오 D: 기존 노트 수정 저장
  Given 기존 노트를 편집 중이다
  When 태그를 추가하고 저장한다
  Then 수정된 노트에 태그가 영속화된다

시나리오 E: 저장 실패 알림
  Given 서버가 응답하지 않는다
  When 태그를 포함해 저장을 시도한다
  Then 저장 실패 알림이 사용자에게 표시된다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~E)가 모두 통과한다.
- [ ] 디자인 시스템 토큰 사용, husky 디자인 검사·lint·build 통과(임의 색/그림자/인라인 style 없음).
- [ ] 기존 노트 CRUD(생성·조회·수정·삭제)에 회귀가 없다.
- [ ] 코드 리뷰 후 머지된다.

## TAG-2 — 추가한 태그를 개별 삭제할 수 있다

**라벨**: `feature/tag` `P0` `slice:delete`
**의존**: TAG-1

### 설명

편집기의 각 태그 칩 오른쪽 **× 버튼**으로 해당 태그만 제거하고, 저장 시 변경이 영속화된다.

### 범위

- `NoteEditor`: 각 칩에 `×` 버튼, 클릭 시 해당 태그를 로컬 태그 상태에서 제거.
- 저장 시 줄어든 `tags`가 서버에 반영.

### 인수 기준 (AC)

```
시나리오 A: 특정 태그 삭제
  Given 노트에 "회의", "아이디어" 태그가 있다
  When "회의" 칩의 × 버튼을 클릭한다
  Then "회의"만 사라지고 "아이디어"는 남는다

시나리오 B: 삭제의 영속화
  Given "회의" 태그를 삭제한 상태다
  When 저장하고 노트를 다시 연다
  Then "회의" 태그가 더 이상 보이지 않는다

시나리오 C: 키보드 접근성
  Given 태그 칩에 포커스가 있다
  When 키보드로 × 버튼을 활성화한다
  Then 해당 태그가 삭제된다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~C)가 모두 통과한다.
- [ ] × 버튼이 키보드 포커스·클릭 모두 가능하다(접근성).
- [ ] 디자인 토큰 사용, husky 검사·lint·build 통과.
- [ ] 코드 리뷰 후 머지된다.

## TAG-3 — 태그 입력이 정규화·검증된다 (빈값·중복·개수)

**라벨**: `feature/tag` `P0` `slice:validation`
**의존**: TAG-1

### 설명

TAG-1의 추가 흐름을 견고하게 만든다. 추가 직전 **정규화(공백 제거 + 영문 소문자)** 후, **빈 값 무시 / 중복 차단(alert) / 최대 5개 제한**을 적용한다. 검증 로직은 순수 함수로 분리해 단위 테스트한다(ADR-0002).

### 범위

- 정규화·검증 로직을 **순수 함수**로 분리(예: `normalizeTag`, `canAddTag`).
- `NoteEditor` 추가 흐름에 연결.
- Vitest 단위 테스트 추가(`src/**/*.test.ts(x)`).

### 인수 기준 (AC)

```
시나리오 A: 공백·대소문자 정규화
  Given 태그 입력칸에 "  Work  "를 입력한다
  When Enter를 누른다
  Then "work"로 정규화되어 추가된다

시나리오 B: 빈 값 무시
  Given 태그 입력칸에 공백만 입력했다
  When Enter를 누른다
  Then 아무 태그도 추가되지 않는다

시나리오 C: 중복 차단
  Given 노트에 "work" 태그가 이미 있다
  When "Work"를 입력하고 Enter를 누른다
  Then 추가되지 않고 "이미 있는 태그" 안내 알림이 뜬다

시나리오 D: 개수 제한
  Given 노트에 태그가 5개 있다
  When 6번째 태그를 추가하려 한다
  Then 추가되지 않는다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~D)가 모두 통과한다.
- [ ] 정규화·중복·개수 로직이 순수 함수로 분리되고, 그에 대한 **단위 테스트가 통과**한다.
- [ ] 디자인 토큰 사용, husky 검사·lint·build 통과.
- [ ] 코드 리뷰 후 머지된다.

## TAG-4 — 노트 목록에서 각 노트의 태그가 보인다

**라벨**: `feature/tag` `P0` `slice:list-display`
**의존**: TAG-1

### 설명

노트 목록(`NoteItem`)에서 각 노트의 태그를 칩으로 표시해, 목록만 훑어도 노트 성격을 파악할 수 있게 한다.

### 범위

- `NoteItem`: `note.tags`를 칩으로 렌더. 많으면 줄바꿈, 없으면 미표시.
- 디자인 토큰 사용, 칩은 작게(`text-xs`).

### 인수 기준 (AC)

```
시나리오 A: 목록에 태그 표시
  Given "회의" 태그가 달린 노트가 있다
  When 노트 목록을 본다
  Then 해당 노트 카드에 "회의" 칩이 보인다

시나리오 B: 태그 없는 노트
  Given 태그가 없는 노트가 있다
  When 노트 목록을 본다
  Then 해당 카드에는 태그 칩 영역이 표시되지 않는다

시나리오 C: 다수 태그 줄바꿈
  Given 태그가 5개 달린 노트가 있다
  When 노트 목록을 본다
  Then 태그 칩이 줄바꿈되어 모두 보인다
```

### 완료 정의 (DoD)

- [ ] 위 AC 시나리오(A~C)가 모두 통과한다.
- [ ] 칩 스타일이 디자인 시스템 토큰을 따른다, husky 검사·lint·build 통과.
- [ ] 목록 렌더 성능·레이아웃에 회귀가 없다.
- [ ] 코드 리뷰 후 머지된다.
