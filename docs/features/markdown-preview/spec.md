# 마크다운 미리보기 기능 정의서 (초안)

> 거친 아이디어를 최소한의 형태로 적은 ① 초안입니다. 모호함은 다음 단계(`spec-fixed.md`)의 인터뷰에서 결정합니다.

## 기능 개요

노트 본문(`content`)을 **마크다운으로 렌더링**해서 읽기 좋게 보여준다. 편집기에서 **편집(raw 텍스트)** 과 **미리보기(렌더 결과)** 를 토글로 전환할 수 있다.

## 한 줄 요약

본문을 마크다운으로 렌더하고 편집/미리보기를 토글한다.

## 가치

- 노트 앱의 핵심 가치인 **읽기 좋은 메모**를 완성한다.
- 저장 포맷은 **plain text 그대로** 두고 **렌더 레이어만 추가**하면 되므로 위험이 낮다.
- 헤딩·리스트·코드블록만 지원해도 체감 개선이 크다.

## 핵심 요구사항 (러프)

- 본문을 마크다운으로 렌더한다. (헤딩, 리스트, 강조, 코드)
- 편집기에 편집 / 미리보기 **토글**을 둔다.
- 저장되는 데이터 형식은 바꾸지 않는다(여전히 plain text).

## 비고 / 제약 (러프)

- **포함**: `react-markdown` 의존성, 편집/미리보기 토글 상태, 헤딩·리스트·강조·코드 렌더.
- **제외**: 분할 동시 뷰(에디터+프리뷰 나란히), 이미지 업로드, 저장 포맷 변경.
- 기존 아키텍처 경계 유지: 서버 데이터는 `NotesContext`, 화면 상태는 컴포넌트/`App`.
- 디자인은 임의 색/그림자 없이 시맨틱 토큰만 사용.

## 아직 결정 안 된 것 (→ 다음 단계 질문거리)

- 토글 상태를 어디에 둘까? (편집기 로컬 / `App` / Context)
- 미리보기에서도 제목·태그를 함께 보일까, 본문만 렌더할까?
- 어떤 마크다운 라이브러리/플러그인을 쓸까? (`react-markdown` 단독 vs `remark-gfm` 추가)
- 코드블록 문법 강조(syntax highlighting)까지 할까, 단순 `<pre><code>`까지만 할까?
- 새 노트 작성 중에도 미리보기를 허용할까?
- 마크다운 렌더 시 보안(raw HTML/XSS)은 어떻게 막을까?
