## 1. TL;DR

선택한 노트를 **한 번의 클릭으로 복제**하는 기능을 도입한다. 제목에 `(사본)`을 붙이고 본문·태그까지 복사해 **새 노트**(서버가 새 `id` 부여)를 만들고, 복제 직후 그 노트를 연다. 이번 범위는 **단일 노트 복제·즉시 선택**까지이며, **다중 선택 일괄 복제·템플릿 라이브러리는 범위에서 제외**한다. 신규 API 없이 기존 `addNote`를 재사용한다.

## 2. 배경 & 문제 (Background & Problem)

- 사용자는 회의록·일지·체크리스트처럼 **같은 양식을 반복 작성**한다. 현재는 기존 노트를 열어 제목·본문·태그를 **수동 복사·붙여넣기**해야 해 번거롭고 실수가 잦다.
- 데이터 흐름(`db.json → JSON Server → api/notes.ts → NotesContext → 컴포넌트`)과 생성 경로(`createNote`/`addNote`)가 이미 안정적이다. 복제는 **새 데이터 모델 없이** 기존 생성 경로 위에 얇게 얹을 수 있다.
- **왜 지금**: 태그(v0.1)·검색(v0.2)으로 노트를 분류·탐색하는 토대가 갖춰졌다. 이제 **작성 생산성**을 높이는 가장 저렴한 한 수가 "복제"다 — 기존 `addNote(title, content, tags)` 재사용으로 구현 비용이 최소다.

## 3. 목표 & 비목표 (Goals / Non-Goals)

### Goals

- G1. 노트 목록에서 한 번의 클릭으로 특정 노트를 복제할 수 있다.
- G2. 복제본은 제목에 `(사본)`이 붙고 본문·태그가 원본과 동일하다.
- G3. 복제본은 원본과 독립된 **새 노트**(서버 부여 `id`)이며, 복제 직후 열린다.

### Non-Goals (이번 릴리스 제외)

- N1. **다중 선택 일괄 복제** (여러 노트를 한 번에).
- N2. **템플릿 라이브러리** (양식을 별도 저장·관리하는 보관소).
- N3. **사본 연쇄 카운팅**(`(사본 2)`, `(사본 3)` 자동 증가).
- N4. **복제 확인 대화상자·되돌리기(undo)**.
- N5. **편집기 내부 "현재 노트 복제" 버튼** (복제는 목록 카드에서만).

## 4. 성공 지표 (Success Metrics)

| 지표        | 정의                                            | 목표                            |
| ----------- | ----------------------------------------------- | ------------------------------- |
| 복제 정확성 | 복제본의 제목·본문·태그가 규칙대로 만들어지는가 | 단위 테스트 100% 통과           |
| 독립성      | 복제본이 원본과 다른 `id`를 갖고 독립적인가     | AC로 검증 (원본 수정 영향 없음) |
| 즉시 선택   | 복제 직후 복제본이 선택(열림) 상태가 되는가     | AC로 검증                       |
| 회귀        | 기존 노트 CRUD·태그·검색 동작 영향              | 회귀 0                          |

> 소규모 학습 프로젝트이므로 정량 트래킹 인프라는 없다. 지표는 **수용 기준(AC)과 단위 테스트·수동 점검**으로 대체한다.

## 5. 사용자 & 유스케이스

- **페르소나**: 같은 양식의 노트를 반복 작성·관리하는 단일 사용자(로컬 앱).
- **핵심 유스케이스**
  - U1. 지난주 회의록을 복제해 이번 주 회의록의 골격(제목 형식·항목·태그)을 즉시 얻는다.
  - U2. 일지 템플릿 노트를 복제해 오늘 날짜 내용만 갈아끼운다.
  - U3. 복제 직후 열린 복제본에서 제목의 `(사본)`을 지우고 내용을 바로 편집한다.

## 6. 기능 요구사항 (Functional Requirements)

우선순위: **P0**=이번 릴리스 필수, **P1**=가능하면 포함.

| ID   | 요구사항                         | 우선순위 | 수용 기준(AC)                                                                   |
| ---- | -------------------------------- | -------- | ------------------------------------------------------------------------------- |
| FR-1 | 목록 카드에서 노트를 복제한다    | P0       | 카드의 **복제 버튼** 클릭 시 그 노트의 복제본이 생성된다 (ADR-0002)             |
| FR-2 | 복제본 제목에 `(사본)`을 붙인다  | P0       | 복제본 제목 = `원본 제목 + " (사본)"`(원본 빈 제목이면 `"(사본)"`)              |
| FR-3 | 본문·태그를 복사한다             | P0       | 복제본의 `content`·`tags`가 원본과 동일(태그는 얕은 복사) (ADR-0001)            |
| FR-4 | 복제본에 새 `id`를 부여한다      | P0       | `id`·타임스탬프는 복사하지 않고 **서버가 새로 부여**한다 (ADR-0001)             |
| FR-5 | 복제는 신규 API 없이 처리한다    | P0       | 기존 `createNote`/`addNote` 경로 재사용, 새 fetch 함수 없음 (ADR-0001)          |
| FR-6 | 복제 직후 복제본을 선택한다      | P0       | 생성 성공 후 `selectedNoteId`가 복제본 `id`로 바뀌어 편집기에 열린다 (ADR-0002) |
| FR-7 | 카드 선택과 복제 클릭을 분리한다 | P0       | 복제 버튼 클릭이 카드 선택을 트리거하지 않는다(`stopPropagation`)               |
| FR-8 | 복제 실패를 사용자에게 알린다    | P0       | 실패 시 기존 패턴(`try/catch` + alert)으로 알리고 선택을 옮기지 않는다          |
| FR-9 | 원본 데이터를 변형하지 않는다    | P0       | 복제는 원본 `Note`를 읽기만 하고 변경하지 않는다(값 복사) (ADR-0001)            |

## 7. 비기능 요구사항 (NFR)

- **아키텍처 정합**: 노트 생성(서버 데이터)은 `NotesContext`, 선택(화면 상태)은 `App.tsx`. 복제는 이 경계를 그대로 따른다 — Context가 만들고 App이 선택을 옮긴다 (ADR-0002). 컴포넌트는 직접 `fetch`하지 않는다.
- **호환성**: 구버전 노트는 `tags`가 없을 수 있으므로 `note.tags ?? []`로 방어(태그·검색 기능과 동일 철학).
- **단순성**: 신규 API·신규 데이터 모델 없음. `addNote` 반환을 `void → Note`로 넓히는 **하위호환 변경**만 한다(기존 호출부 무영향).
- **성능**: 복제는 POST 한 번 + 로컬 배열 append 한 번. 체감 지연 없음.
- **접근성**: 복제 버튼은 키보드 포커스·클릭 가능, 명확한 라벨(`복제`).
- **스타일**: 시맨틱 토큰(`text-muted-foreground` 등)만 사용. 임의 색/그림자/인라인 style 금지.

## 8. UX / 인터랙션 흐름

```
[복제]
목록 카드의 [복제] 클릭
  → stopPropagation(카드 선택 분리)
  → buildDuplicatePayload(note): { title: title+" (사본)", content, tags: [...tags] }
  → addNote(payload)  ── 서버 POST → 새 id 부여 → notes에 append
  → 성공? ─yes→ setSelectedNoteId(생성노트.id)  (복제본 열림)
          └no → alert("복제에 실패했습니다"), 선택 그대로
```

빈 제목 처리: 원본 `title === ""` → 복제본 제목 `"(사본)"`(앞 공백 없음). 그 외엔 `"제목 (사본)"`.

## 9. 데이터 / 상태 모델 (개요)

- 복제 페이로드 생성: **[ADR-0001](adr/0001-reuse-addnote-return-created.md)** — 순수 함수 `buildDuplicatePayload`, `id`/타임스탬프 제외, `addNote` 재사용 + 반환을 `Promise<Note>`로 확장.
- 복제 후 선택 책임 분리: **[ADR-0002](adr/0002-duplicate-context-app-selection.md)** — Context의 `duplicateNote(id): Promise<Note>`가 생성하고, `App.tsx`가 반환 `id`로 선택 이동.

```ts
// src/lib/duplicate.ts
export function buildDuplicatePayload(note: Note): {
  title: string;
  content: string;
  tags: string[];
};

// src/context/NotesContext.tsx
addNote: (title: string, content: string, tags: string[]) => Promise<Note>; // void → Note
duplicateNote: (id: string) => Promise<Note>;

// src/App.tsx
const handleDuplicate = async (id: string) => {
  const created = await duplicateNote(id);
  setSelectedNoteId(created.id);
};
```

## 13. 의존성

- 태그 기능(v0.1): 복제는 `Note.tags`를 복사. 없으면 `?? []`로 방어.
- 생성 경로: `api/notes.ts`의 `createNote`(서버가 `id`, 클라이언트가 타임스탬프) — 변경 없이 재사용.
- 상태 경계: `NotesContext`(서버 데이터)와 `App.tsx`의 `selectedNoteId`(화면 상태).
