# 노트 복제 기능 정의서 (초안)

> 거친 아이디어를 최소한의 형태로 적은 ① 초안입니다. 결정되지 않은 부분은 ②(`spec-fixed.md`)의 인터뷰에서 채웁니다.

## 기능 개요

선택한 노트를 **한 번의 클릭으로 복사**해 새 노트를 만든다. 제목에는 `(사본)`을 붙이고, 본문과 태그까지 그대로 복사한다.

## 핵심 요구사항

- 노트 목록에서 노트마다 **복제 버튼**을 눌러 그 노트를 복사할 수 있다.
- 복제본의 제목은 원본 제목 + `(사본)`.
- 본문(`content`)과 태그(`tags`)는 원본 그대로 복사한다.
- 복제본은 **새 노트**다 — 새 `id`는 서버가 부여하고, 원본과 독립적이다(이후 한쪽을 고쳐도 다른 쪽은 안 바뀐다).
- 복제 직후 그 새 노트가 **선택(열림)** 상태가 된다.

## 가치

회의록·일지·템플릿처럼 **같은 양식을 반복 작성**하는 패턴을 한 번 클릭으로 해결한다. 기존 `addNote(title, content, tags)` 시그니처를 그대로 재사용하면 신규 API 없이 구현 비용이 최소다.

## 비고 / 제약 (러프)

- 데이터는 `db.json` → JSON Server → `api/notes.ts` → `NotesContext` → 컴포넌트로 단방향으로 흐른다. 복제도 이 경로를 따른다.
- `id`/타임스탬프는 **새로 부여**되어야 한다(원본 것 재사용 금지). 생성은 기존 `createNote` 규칙(서버가 `id`, 클라이언트가 타임스탬프)을 그대로 탄다.
- 선택 상태(`selectedNoteId`)는 화면 상태로 `App.tsx`가 소유한다. 복제 후 선택은 이 경계 안에서 처리해야 한다.

## 아직 안 정한 것 (②에서 결정)

- 복제를 **새 Context 함수(`duplicateNote`)** 로 둘지, 컴포넌트에서 **기존 `addNote` 재사용**으로 끝낼지.
- "복제 직후 선택"을 어떻게 구현할지(`addNote`가 현재 `void`라 새 `id`를 못 돌려줌).
- 복제 버튼을 어디에 둘지(목록 카드 / 편집기 / 둘 다).
- 제목이 비었거나 이미 `(사본)`이 붙은 노트를 복제할 때의 처리.
- 복제 실패 시 사용자 알림 방식.
  </content>
  </invoke>
