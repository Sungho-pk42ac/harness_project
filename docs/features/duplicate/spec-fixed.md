# 노트 복제 기능 정의서 (확정본)

> 원본 `spec.md`를 바탕으로, 설계 질문의 답변을 반영해 구현 가능한 수준으로 구체화한 문서입니다.

## 기능 개요

선택한 노트를 **한 번의 클릭으로 복제**한다. 제목에 `(사본)`을 붙이고 본문·태그까지 복사해 **새 노트**를 만든 뒤, 그 새 노트를 선택(열림) 상태로 둔다.

## 기능 요구사항

- 노트 목록 카드에서 복제 버튼으로 그 노트를 복제할 수 있다.
- 복제본 제목은 `원본 제목 + " (사본)"`.
- 본문·태그는 원본 그대로 복사한다.
- 복제본은 새 `id`(서버 부여)를 갖는 독립 노트다.
- 복제 직후 그 새 노트가 선택된다.

---

## 1. 데이터 / 상태 모델

- 복제본은 **`Note` 한 건을 새로 생성**하는 것이다. 데이터 형태는 기존과 동일(`title/content/createdAt/updatedAt/tags`).
- `id`와 타임스탬프(`createdAt/updatedAt`)는 **복사하지 않는다.** `id`는 **서버가 부여**하고(POST 응답의 `id`), 타임스탬프는 `createNote`가 클라이언트에서 새로 채운다(기존 규칙 유지).
- 태그는 `string[]`을 **얕은 복사**(`[...note.tags]`)로 넘긴다. 빈 배열·`undefined`(구버전 노트)는 `note.tags ?? []`로 방어한다.
- **상태 경계 준수**: 서버 데이터(노트 배열)는 `NotesContext`가, 화면 선택 상태(`selectedNoteId`)는 `App.tsx`가 소유한다. 복제는 **Context가 노트를 만들고**, **App이 새 id로 선택을 옮긴다**. (ADR-0002)

---

## 2. 입력 / 트리거

- **위치**: 노트 목록 카드(`NoteItem`)에 기존 `삭제` 버튼 옆에 **`복제` 버튼**을 둔다. (편집기·다중선택은 범위 외)
- 카드 클릭은 선택이므로, 복제 버튼 클릭은 `e.stopPropagation()`으로 **카드 선택과 분리**한다(삭제 버튼과 동일 패턴).
- 복제는 **확인 대화상자 없이 즉시** 실행한다(되돌리기 쉬운 비파괴 작업이라 마찰 최소화).

---

## 3. 복제 규칙 / 동작

복제 버튼을 누르면 아래 순서로 처리한다.

1. **제목 생성**: `복제본 제목 = (원본 title) + " (사본)"`.
   - 원본 제목이 비어 있으면(`""`) → `"(사본)"`만 단다(앞 공백 없이).
   - 이미 `(사본)`이 붙은 노트를 또 복제하면 `"제목 (사본) (사본)"`이 된다. **연쇄 카운팅(사본 2 등)은 하지 않는다**(범위 외, 단순성 우선).
2. **본문 복사**: `content`를 그대로 복사한다.
3. **태그 복사**: `note.tags ?? []`를 얕은 복사해 넘긴다(원본 배열과 참조 분리).
4. **생성**: `addNote(title, content, tags)` 경로로 **새 노트를 생성**(서버 POST → 새 `id`).
5. **선택 이동**: 생성된 노트의 `id`로 `selectedNoteId`를 바꿔 **복제본을 연다**. (생성 성공 후에만)

> 원본과 복제본은 완전히 독립이다. 생성 후 한쪽을 수정/삭제해도 다른 쪽에 영향이 없다(임베드 데이터를 값으로 복사하므로).

---

## 4. 표시

- 복제 버튼은 텍스트 라벨 **`복제`** 로, 삭제 버튼과 같은 작은 스타일(`text-xs`, `text-muted-foreground`, hover 시 강조)을 따른다.
- 색·형태는 **디자인 시스템 시맨틱 토큰만** 사용한다(`text-muted-foreground`, hover는 기존 톤). 임의 색/그림자/인라인 style 금지.
- 복제가 끝나면 목록에 복제본 카드가 즉시 나타나고(서버 응답 기반 갱신), 그 카드가 선택 표시(테두리 강조)된다.

---

## 5. 저장 흐름 (Context / API)

데이터는 `목록 카드 → App/Context → api/notes.ts → 서버` 순으로 흐른다.

- **API**: **신규 API 없음.** 기존 `createNote`(→`addNote`)를 그대로 재사용한다. `addNote`의 인자 시그니처 `(title, content, tags)`도 그대로 쓴다.
- **반환값 확장(핵심 결정)**: 현재 `addNote`는 `Promise<void>`라 복제 직후 선택할 새 `id`를 알 수 없다. 따라서 **`addNote`가 생성된 `Note`를 반환**하도록 `Promise<Note>`로 넓힌다. 내부는 이미 `created`를 갖고 있으므로 `return created` 한 줄만 추가하면 되고, 반환을 무시하는 기존 호출부(`NoteEditor`)는 영향이 없다. (ADR-0001)
- **Context 함수**: 얇은 `duplicateNote(id)`를 Context에 추가한다. 원본을 `notes`에서 찾아 위 복제 규칙으로 `addNote`를 호출하고, **생성된 `Note`를 반환**한다. (ADR-0002)
  ```ts
  duplicateNote(id: string) => Promise<Note>
  ```
- **선택 이동**: `App.tsx`가 `duplicateNote`의 반환 노트 `id`로 `setSelectedNoteId`를 호출한다. (Context는 선택 상태를 모른다 — 경계 유지)
- **실패 처리**: 기존 패턴 그대로 **호출부에서 `try/catch` + `alert`**. 실패 시 선택을 옮기지 않는다(생성 성공 후에만 선택). 원본을 찾을 수 없으면(이미 삭제 등) 조용히 무시하거나 알림한다 — AC에서 명시.

## 6. 범위

### 이번 범위에 포함

- 목록 카드의 `복제` 버튼으로 단일 노트 복제(제목 `(사본)` + 본문·태그 복사).
- 새 `id`(서버 부여)로 독립 노트 생성, 복제 직후 그 노트 선택.
- `addNote` 반환값을 `Promise<Note>`로 확장, `duplicateNote(id)` Context 함수 추가.
- 복제 실패 시 알림.

### 이번 범위에서 제외 (다음 단계)

- **다중 선택 일괄 복제** (여러 노트를 한 번에).
- **템플릿 라이브러리** (양식을 따로 저장·관리).
- **사본 연쇄 카운팅**(`(사본 2)`, `(사본 3)` …) — 이번엔 단순히 `(사본)`만 덧붙인다.
- 편집기 내부의 "현재 노트 복제" 버튼(목록 카드에만 둔다).
- 복제 확인 대화상자·되돌리기(undo).

---

## 7. 테스트

- 테스트 도구(Vitest)는 갖춰져 있다.
- **순수 로직(복제 페이로드 생성: 제목 `(사본)` 규칙, 본문·태그 복사, id/타임스탬프 제외)** 을 순수 함수로 분리해 단위 테스트한다(입력·출력이 분명).
- Context의 `duplicateNote`가 **원본을 찾아 `addNote`를 올바른 인자로 호출하고 생성 노트를 반환**하는지, 선택 이동이 새 `id`로 일어나는지 검증한다(컴포넌트·통합 테스트).
