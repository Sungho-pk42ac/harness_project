# ADR-0002 — 클라이언트 순수 함수 필터 + 원본 불변

- **상태(Status)**: Accepted
- **날짜**: 2026-05-31
- **관련**: [PRD §9 데이터/상태 모델](../PRD.md#9-데이터--상태-모델-개요), [ADR-0001](0001-search-state-location.md)

## Context

검색을 "어떻게" 거를지 정해야 한다. 제약·맥락은 다음과 같다.

- 데이터는 이미 `NotesContext.notes`에 **전부 클라이언트 메모리**에 있다(소규모, 수십~수백 건).
- `NotesContext`는 서버 데이터의 **단일 출처**다 — 변경 함수는 응답 기반으로 `append/map/filter` 한다.
- 검색은 **읽기·표시**일 뿐 서버 데이터를 바꾸지 않는다.

## Decision

검색을 **순수 함수 `filterNotes(notes, query)`** 로 분리하고, **렌더 시점에 파생 목록만 계산**한다(원본 `notes` 불변). 입력은 **타이핑 즉시 실시간** 필터링하며, **디바운스·서버 검색은 도입하지 않는다**.

```ts
// src/utils/filterNotes.ts
export function filterNotes(notes: Note[], query: string): Note[];
//  - 제목·본문·태그 부분 포함, 대소문자 무시(양쪽 toLowerCase)
//  - 검색어 앞뒤 trim, 빈 검색어면 notes 원본 반환
//  - note.tags ?? [] 로 태그 없는 노트 안전 처리
//  - 원본 배열을 변형하지 않음(filter로 새 배열 반환)
```

## Consequences

**긍정**

- 입출력이 명확한 순수 함수라 **단위 테스트가 쉽고** 재사용 가능하다(첫 테스트 예제로 적합).
- 원본 불변이라 **검색어를 지우면 전체가 즉시 복구**되고, Context의 갱신 로직이 걸러진 배열로 오염되지 않는다.
- 서버 왕복·로딩 상태가 없어 체감 지연 0, 구현이 가장 단순하다.

**부정 / 트레이드오프**

- 데이터가 매우 커지면 매 키 입력마다 전체 `filter`가 부담될 수 있다 → 그때 **디바운스** 또는 **서버 검색**을 별도 ADR로 도입한다(YAGNI: 지금은 불필요).
- 관련도 정렬·하이라이트는 이 함수의 책임이 아니다(범위 제외, 별도 슬라이스).

## Alternatives Considered

1. **서버 검색**(JSON Server 쿼리 파라미터로 위임)
   - 기각 이유: 네트워크 왕복·로딩 처리·`api/notes.ts` 확장까지 끌어와 범위가 과도하다. 데이터가 메모리에 이미 있어 이득이 없다.
2. **`NotesContext.notes`를 걸러진 결과로 덮어쓰기**
   - 기각 이유: 원본 복구 불가, `add/edit/remove`의 응답 기반 갱신이 걸러진 배열을 망가뜨린다(단일 출처 훼손).
3. **디바운스 적용**
   - 기각 이유: 타이머 정리·fake timer 테스트라는 숨은 복잡도. 소규모 동기 필터엔 체감 이득이 없다. 규모가 커지면 그때 도입.
