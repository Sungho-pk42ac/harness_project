## 1. TL;DR

노트 목록에서 **검색어로 노트를 걸러 찾는** 기능을 도입한다. 제목·본문·태그를 **한 검색창**으로 통합 검색하고, 타이핑하면 **실시간으로** 목록이 걸러진다. 이번 범위는 **거르기·빈 결과 안내**까지이며, **하이라이트·관련도 정렬·고급 필터·서버 검색은 다음 릴리스로 분리**한다.

## 2. 배경 & 문제 (Background & Problem)

- 노트가 늘어나면서 목록을 끝까지 스크롤하지 않으면 원하는 노트를 찾기 어렵다. 태그(v0.1)로 분류는 가능해졌지만 **직접 찾는 수단**이 없다.
- 데이터는 이미 클라이언트 메모리(`NotesContext`의 `notes`)에 전부 있으므로, 서버 왕복 없이 **화면 필터**만으로 즉시 검색이 가능하다.
- **왜 지금**: 태그·로그인 기반이 갖춰진 지금, 검색은 "데이터를 읽어 거르기"라는 가장 단순한 읽기 계층 기능이라 빠르게 가치(탐색성)를 더할 수 있다.

## 3. 목표 & 비목표 (Goals / Non-Goals)

### Goals

- G1. 검색창에 입력하면 일치하는 노트만 목록에 보인다(제목·본문·태그 통합).
- G2. 검색어를 지우면 다시 전체 노트가 보인다.
- G3. 일치하는 노트가 없으면 "데이터 없음"과 구분되는 전용 안내를 보여준다.

### Non-Goals (이번 릴리스 제외)

- N1. **매치 글자 하이라이트** (매치 인덱스 계산·XSS 위험 → 별도 슬라이스).
- N2. **관련도 정렬** (제목 우선·최신순 등 가중치).
- N3. **고급 필터** (태그 클릭 필터, 날짜 범위, 필드 지정 검색).
- N4. **서버 검색** (JSON Server 쿼리 위임), **디바운스**.

## 4. 성공 지표 (Success Metrics)

| 지표        | 정의                                        | 목표                     |
| ----------- | ------------------------------------------- | ------------------------ |
| 검색 정확성 | 검색어 포함 노트가 빠짐없이 걸러지는가      | 단위 테스트 100% 통과    |
| 즉시성      | 입력 후 목록 갱신 체감 지연                 | 0 (동기 필터, 데모 기준) |
| 상태 구분   | "검색 0건"과 "노트 0건"을 다르게 안내하는가 | AC로 검증                |
| 회귀        | 기존 노트 CRUD·태그·로그인 동작 영향        | 회귀 0                   |

> 소규모 학습 프로젝트이므로 정량 트래킹 인프라는 없다. 지표는 **수용 기준(AC)과 단위 테스트·수동 점검**으로 대체한다.

## 5. 사용자 & 유스케이스

- **페르소나**: 노트를 다수 작성·관리하는 단일 사용자(로컬 앱).
- **핵심 유스케이스**
  - U1. 기억나는 단어(제목 일부)를 쳐서 노트를 바로 찾는다.
  - U2. 본문에만 등장하는 키워드로 노트를 찾는다.
  - U3. 태그명을 쳐서 그 태그가 달린 노트를 찾는다.
  - U4. 검색어를 지워 전체 목록으로 돌아온다.

## 6. 기능 요구사항 (Functional Requirements)

우선순위: **P0**=이번 릴리스 필수, **P1**=가능하면 포함.

| ID   | 요구사항                             | 우선순위 | 수용 기준(AC)                                                                 |
| ---- | ------------------------------------ | -------- | ----------------------------------------------------------------------------- |
| FR-1 | 검색창으로 노트를 거른다             | P0       | 검색창에 입력하면 일치 노트만 목록에 보인다(실시간) (ADR-0001, ADR-0002)      |
| FR-2 | 제목·본문·태그를 통합 검색한다       | P0       | 검색어가 제목·본문·태그 **어디에 들어 있어도** 매치된다                       |
| FR-3 | 부분 포함·대소문자 무시로 매칭한다   | P0       | 검색어가 일부만 들어 있어도, 대소문자가 달라도 매치된다                       |
| FR-4 | 앞뒤 공백을 무시한다                 | P0       | 검색어 앞뒤 공백은 `trim` 후 비교(가운데 공백은 유지)                         |
| FR-5 | 빈 검색어는 전체를 보여준다          | P0       | 미입력 또는 공백만이면 전체 노트를 그대로 표시                                |
| FR-6 | 검색어를 한 번에 지운다              | P0       | 입력칸의 **× 버튼** 클릭 시 검색어가 비워지고 전체가 보인다                   |
| FR-7 | 검색 결과 0건을 전용 문구로 안내한다 | P0       | 결과가 없으면 **"검색 결과가 없습니다"**(원본 0건의 "노트가 없습니다"와 구분) |
| FR-8 | 선택/생성 중에도 검색어를 유지한다   | P1       | 검색 중 노트를 열거나 새로 만들어도 검색어·필터가 유지된다 (ADR-0001)         |
| FR-9 | 원본 데이터를 변형하지 않는다        | P0       | 검색은 파생 계산만 — `NotesContext`의 `notes`는 불변 (ADR-0002)               |

## 7. 비기능 요구사항 (NFR)

- **아키텍처 정합**: 검색어는 **화면 상태**이므로 `App.tsx`의 `useState`에 둔다(`selectedNoteId`와 같은 경계). 서버 데이터(`NotesContext`)와 섞지 않는다 (ADR-0001).
- **호환성**: `tags`가 없는 기존 노트는 `note.tags ?? []`로 안전 처리(태그 기능과 동일 철학).
- **성능**: 수십~수백 건 동기 `filter`는 체감 지연 0. 디바운스는 데이터가 커지면 별도 도입.
- **접근성**: 입력칸에 `type="search"` + `aria-label`, × 버튼에도 `aria-label`.
- **스타일**: 시맨틱 토큰(`bg-card`, `border-border`, `text-muted-foreground`)만 사용. 임의 색/그림자/인라인 style 금지.

## 8. UX / 인터랙션 흐름

```
[검색]
검색창 입력 → trim → 빈값? ─yes→ 전체 목록 표시
                     └no→ filterNotes(notes, query) → 결과>0? ─yes→ 걸러진 목록
                                                              └no→ "검색 결과가 없습니다"
[지우기]
× 버튼 → 검색어 "" → 전체 목록 (포커스 유지)
```

빈 상태 우선순위(`NoteList`): `loading` → `error` → 원본 0건("노트가 없습니다") → 검색 0건("검색 결과가 없습니다") → 목록.

## 9. 데이터 / 상태 모델 (개요)

- 검색어 상태 위치: **[ADR-0001](adr/0001-search-state-location.md)** — `App.tsx`의 `searchQuery: string`.
- 필터링 전략: **[ADR-0002](adr/0002-client-pure-filter.md)** — 순수 함수 `filterNotes`, 원본 불변, 클라이언트 실시간.

```ts
// App.tsx

```
