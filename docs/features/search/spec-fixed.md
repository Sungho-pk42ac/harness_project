# 검색 기능 정의서 (확정본)

> 원본 spec.md를 바탕으로 멀티 에이전트 패널 토론(20개 설계 질문)을 거쳐 확정한 문서입니다.

## 기능 개요

노트 목록에서 검색어로 원하는 노트를 빠르게 걸러 찾을 수 있다.

## 기능 요구사항

- 검색창에 글자를 입력하면 일치하는 노트만 목록에 보인다
- 제목·본문·태그 중 어디에 들어 있어도 한 검색창으로 찾을 수 있다
- 대소문자를 구분하지 않고 찾는다
- 검색어를 지우면 다시 전체 노트가 보인다
- 일치하는 노트가 하나도 없으면 "검색 결과가 없습니다"를 안내한다

---

## 1. 데이터·검색 상태 모델

- 검색어는 **서버 데이터가 아니라 화면 상태**다. `selectedNoteId`/`isCreating`과 같은 경계에 두어 **`App.tsx`의 `useState`** 로 보관한다 → `searchQuery: string`.
  - 근거(의견 일치): 기존 규칙(서버 데이터만 Context, 선택·생성 같은 화면 상태는 App)을 그대로 따라야 상태 책임 경계가 깨지지 않는다. Context에 두면 노트 전환·재진입 시 잔여 상태가 남을 위험이 있다.
- 검색은 **원본 `notes` 배열을 바꾸지 않고**, 렌더링 시점에 **걸러진 목록만 계산**한다(원본 불변).
  - 근거: `NotesContext`는 "서버 데이터의 단일 출처"다. Context의 `notes`를 걸러진 결과로 덮어쓰면 검색어를 지웠을 때 원본 복구가 불가능하고, `addNote`(append)·`editNote`(map)·`removeNote`(filter)의 응답 기반 갱신이 걸러진 배열을 망가뜨린다.
- 걸러내는 로직은 **순수 함수 `filterNotes(notes, query)`** 로 분리한다. 입력(노트 배열 + 검색어) → 출력(걸러진 노트 배열)이 명확해 단위 테스트와 재사용이 쉽다.

```ts
// src/utils/filterNotes.ts
/**
 * 검색어로 노트 목록을 거른다 (원본 불변, 제목·본문·태그 부분 포함, 대소문자 무시).
 * @param notes 원본 노트 목록
 * @param query 사용자가 입력한 검색어
 * @returns 걸러진 노트 목록 (빈 검색어면 원본을 그대로 반환)
 */
export function filterNotes(notes: Note[], query: string): Note[];
```

```ts
// App.tsx — 화면 상태로 보관 (selectedNoteId와 같은 자리)
const [searchQuery, setSearchQuery] = useState('');
```

## 2. 입력 동작 (SearchBar)

- 검색창은 **사이드바의 노트 목록 바로 위(NoteList 영역 상단)** 에 둔다.
  - 근거: spec의 "노트 목록 화면 위에서 동작"에 정확히 부합하고, 검색창과 결과 목록이 시선상 이어져 "걸러진다"는 인과가 즉시 이해된다. 헤더/Layout 슬롯에 두면 합성 구조까지 건드려 범위가 샌다.
- 검색창은 **`SearchBar` 프레젠테이션 컴포넌트로 분리**하고 `value`/`onChange`만 props로 받는다(stateless).
  - 근거: 프로젝트의 "프레젠테이션 컴포넌트는 콜백 props(`onXxx`)로 받는다" 컨벤션과 일치하고, 입력 UI를 독립시켜야 placeholder·`×` 버튼·접근성 같은 디테일을 단위 테스트로 검증하기 쉽다.
- 결과는 **타이핑할 때마다 실시간으로 필터링**한다.
  - 근거: 클라이언트 메모리 필터에 소규모 데이터라 즉시 반응에 지연이 없고 구현·테스트가 가장 단순하다. Enter 방식은 "지우면 전체"를 깨는 추가 분기를 부른다.
- **디바운스는 이번 범위에서 제외**한다(단순 실시간으로 시작).
  - 근거: 수십~수백 건의 동기 `filter`는 체감 지연이 0이다. 디바운스는 타이머 정리·fake timer 테스트라는 숨은 복잡도만 추가하므로, 데이터 규모가 커지면 별도 이슈로 도입한다.
- 검색어를 한 번에 비우는 **`×`(지우기) 버튼**을 입력칸 안에 제공한다(직접 글자 지우기도 가능).
  - 근거: "검색어를 지우면 전체가 보인다"를 원클릭으로 실행하는 표준 UX다. 버튼에도 `aria-label`을 부여하고, 클릭 후에도 입력 포커스를 유지한다(ESC 단축키는 이번 범위 과잉이라 제외).

```ts
// src/components/SearchBar.tsx
interface SearchBarProps {
  value: string;
  onChange: (value: string) => void;
}
```

## 3. 매칭 규칙

검색어가 노트의 어떤 항목과 일치할 때 "맞다"고 볼지 정한다. 모든 규칙은 순수 함수 `filterNotes` 안에서 처리한다.

- **검색 대상**: **제목 + 본문 + 태그**를 한 검색창으로 통합 검색한다(spec대로).
  - 근거: spec 요구를 한 슬라이스로 충족하고, 사용자는 어디에 있든 "한 칸"에 치면 찾기를 기대한다. 분리하면 입력 위치를 고민시켜 직관성이 떨어진다.
- **부분 포함(부분 일치)**: 검색어가 일부만 들어 있어도 매치한다(예: `회의` → `주간회의록`).
  - 근거: 노트 검색의 표준 멘탈모델이며 spec의 "걸러진다" 취지에 맞다. 정확/접두 일치는 빈 결과로 좌절을 유발한다.
- **대소문자 무시**: 검색어와 비교 대상을 **양쪽 다 소문자로 바꿔 비교**한다(`toLowerCase`). 한글·숫자는 영향 없다.
  - 근거: spec의 "대소문자 구분 안 함"을 최소 비용으로 만족하고, 태그 정규화의 소문자 통일 규칙과 일관된다. (튀르키예 `i` 등 로케일 함정은 강의용 한·영 범위에선 충분히 무시 가능.)
- **공백 처리**: 검색어의 **앞뒤 공백만 제거(`trim`)** 하고 비교한다. 가운데 공백은 그대로 둔다.
  - 근거: 흔한 입력 실수(앞뒤 공백)만 막아 동작이 예측 가능하다. 가운데 공백까지 지우면 `주간 회의`와 `주간회의`가 섞여 의도와 다른 매치를 부른다.
- **빈 검색어**: 아무것도 안 쳤거나 **공백만 입력**한 경우(`trim` 후 빈 문자열) **전체 노트를 그대로** 보여준다.
  - 근거: spec의 "검색어를 지우면 전체가 보인다"와 동일 동작이고, 빈 화면은 사용자에게 손해다.
- **태그 매칭**: 태그는 `string[]`이므로 **태그 중 하나라도 검색어를 부분 포함하면 매치**한다(예: `회의` → `['주간회의']`).
  - 근거: 제목·본문의 부분 포함 규칙과 일관되어 `filterNotes` 한 함수로 동일 로직을 재사용한다. 모든 태그를 한 문자열로 join해 검사하면 태그 경계가 뭉개져 오매치를 부른다.
  - **엣지 처리**: `tags`가 없는 기존 노트는 `note.tags ?? []`로 빈 배열 취급해 안전하게 건너뛴다(태그 spec과 동일 철학).

## 4. 표시·빈 결과

- 걸러진 결과는 **기존 노트 목록 순서를 그대로 유지**한다(거르기만, 순서 안 바꿈).
  - 근거: 이번 슬라이스는 "거르기"까지다. 관련도(제목 매치 우선 등) 재정렬은 가중치·동점 처리라는 별개 복잡도이고, 순서가 갑자기 바뀌면 위치 기억이 깨져 혼란스럽다 → 별도 이슈(한 번에 한 기능).
- 검색했는데 **맞는 노트가 하나도 없을 때**는 **`검색 결과가 없습니다`** 라는 **검색 전용 안내문**을 표시한다.
  - 근거: "노트가 0개"(데이터 없음)와 "검색에 안 걸림"(데이터는 있지만 미일치)은 다른 상태다. 같은 문구를 재사용하면 사용자가 데이터가 사라진 줄 안다. `NoteList`의 early-return 패턴에 분기 하나만 추가한다.
- **빈 상태 분기 우선순위**(`NoteList`): `loading` → `error` → **원본 0개("노트가 없습니다")** → **검색 결과 0개("검색 결과가 없습니다")** → 목록 렌더.

## 5. 저장·상태 흐름

데이터는 `db.json → JSON Server → api/notes.ts → NotesContext → 컴포넌트` 순으로 흐른다. 검색은 이 흐름 **위에 얹히는 화면 필터**일 뿐, 서버 데이터를 바꾸지 않는다.

- **데이터 흐름**: `App`이 `searchQuery`를 소유하고 `SearchBar`(value/onChange)와 `NoteList`(걸러진 목록)에 내려준다. `NoteList`는 `useNotes()`의 원본 `notes`를 `filterNotes(notes, searchQuery)`로 걸러 렌더한다. (`NoteList`는 "걸러진 목록만 받는" 형태로 단순화해도 무방하며, 결합도를 낮추는 쪽을 택한다.)
- **선택/생성 시 검색어 유지**: 검색 중 노트를 클릭해 열거나 새 노트를 만들어도 **검색어는 그대로 둔다**(목록은 계속 걸러진 상태).
  - 근거: 검색은 `selectedNoteId`와 독립된 App 화면 상태다. 선택마다 초기화하면 여러 결과를 오가며 보는 흐름이 끊기고 상태 경계가 흐려진다.
- **새 노트 저장 후 노출**: 검색어가 걸린 상태에서 새 노트를 저장했는데 그 노트가 검색어에 안 맞으면 **그대로 둔다**(검색 필터 우선 — 안 보여도 정상).
  - 근거: 규칙이 단순·예측 가능하고, 안 보이면 검색어를 지우면 해결된다. "저장 직후 자동 초기화"는 "검색어 유지"(위 항목)와 정면 충돌한다.
- **변경 함수 미수정**: `addNote`/`editNote`/`removeNote`와 `api/notes.ts`는 검색 기능 때문에 손대지 않는다. 검색은 순수 읽기·표시 계층이다.

## 6. 범위 (포함·제외)

### 이번 범위에 포함

- 제목·본문·태그 통합 검색 (부분 포함, 대소문자 무시, 앞뒤 공백 trim)
- 빈 검색어 = 전체 표시, 검색 결과 0개 전용 안내문
- 실시간 필터링, `×`(지우기) 버튼
- `filterNotes` 순수 함수 분리 + `SearchBar` 프레젠테이션 컴포넌트

### 이번 범위에서 제외 (다음 이슈)

- **디바운스**(입력 멈춤 후 지연 필터) → 데이터 규모가 커지면 도입
- **관련도 정렬**(제목 매치 우선·최신순 등) → 가중치·동점 처리 별개 슬라이스
- **매치 글자 하이라이트** → 매치 인덱스 계산·`dangerouslySetInnerHTML` XSS 위험까지 끌어와 범위 크리프
- **고급 필터**(태그 클릭 필터·날짜 범위·필드 지정 검색)
- **서버 검색**(쿼리 파라미터로 JSON Server에 검색 위임)
- **결과 개수 음성 안내**(`aria-live`로 "N개 검색됨" 낭독) → 실시간 필터와 결합 시 과도한 낭독 유발, 후속 개선

### 접근성·스타일 기준(이번 범위)

- **접근성 기본 수준**: input에 `aria-label`(또는 라벨)과 `type="search"`를 부여한다. `×` 버튼에도 `aria-label`을 준다. (`aria-live` 결과 안내는 제외 — 위 참고.)
- **디자인 토큰만 사용**: `bg-card`, `border-border`, `text-muted-foreground` 등 **시맨틱 토큰**만 쓴다. 임의 hex 색·인라인 style·임의 그림자값을 쓰지 않는다(프로젝트 규칙 준수).

## 7. 테스트

- 테스트 도구(Vitest + jsdom + `src/test-setup.ts`)는 갖춰져 있다. 검색 로직은 입출력이 분명해 첫 단위 테스트 예제로 적합하다(태그 spec과 동일 철학).
- **`filterNotes` 순수 함수 단위 테스트** (핵심):
  - 제목에 부분 포함되면 매치된다
  - 본문에 부분 포함되면 매치된다
  - 태그 중 하나라도 부분 포함되면 매치된다
  - 대소문자가 달라도 매치된다(소문자화 비교)
  - 검색어가 빈 문자열이면 전체를 반환한다
  - 검색어가 공백만이면(`trim` 후 빈 값) 전체를 반환한다
  - 일치하는 노트가 없으면 빈 배열을 반환한다
  - `tags`가 없는(undefined) 노트도 에러 없이 건너뛴다
  - 원본 배열을 변형하지 않는다(원본 불변)
- **`SearchBar` 컴포넌트 입력 테스트**:
  - 입력값을 바꾸면 `onChange`가 입력 문자열로 호출된다
  - `×` 버튼을 누르면 `onChange`가 빈 문자열로 호출된다
  - `value` prop이 입력칸에 반영된다
- (선택) **`NoteList` 빈 결과 분기 테스트**: 검색 결과가 0개면 "검색 결과가 없습니다"가, 원본이 0개면 "노트가 없습니다"가 렌더된다.
