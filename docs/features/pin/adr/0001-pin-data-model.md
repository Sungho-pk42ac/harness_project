# ADR-0001 — 핀 데이터 모델: note 내 `isPinned` 불리언 임베드

- **상태(Status)**: Accepted
- **날짜**: 2026-05-31
- **관련**: [PRD §9](../PRD.md#9-데이터--상태-모델-개요), [ADR-0002](0002-pin-state-ownership.md)

## Context

노트에 핀(고정)을 붙이려면 핀 상태를 어디에 어떤 형태로 저장할지 결정해야 한다. 제약은 다음과 같다.

- 저장소는 JSON Server(`db.json`)로 스키마가 자유롭고, 관계형 조인이 없다.
- 기존 노트에는 `isPinned` 필드가 없다(마이그레이션 부담 최소화 필요).
- 핀은 "노트당 고정/해제"라는 **이진 상태** 하나뿐이다. 핀 그룹·핀 순서·핀 메모 같은 부가 정보가 현재 범위에 없다.
- 태그(`tags: string[]`)가 이미 노트 문서에 임베드되어 있고, 읽기 시 `note.tags ?? []`로 구버전을 방어하는 선례가 있다.

## Decision

핀 상태를 **노트 문서 안에 불리언 필드로 임베드**한다.

```ts
interface Note {
  // ...
  isPinned: boolean; // 기본값 false
}
```

- 고정되지 않은 노트의 기본값은 **`false`** 로 통일한다.
- 기존 데이터 호환을 위해, 읽는 지점에서 **`note.isPinned ?? false`** 로 방어한다(서버/기존 문서 일괄 수정 없음).
- 새 노트 생성 시 호출부에서 `isPinned: false`를 채워 보낸다(`api/notes.ts`는 받은 걸 그대로 보냄).

## Consequences

**긍정**

- 구현이 단순하다(별도 컬렉션·조인 없음). 노트 한 건을 읽으면 핀 상태가 함께 온다.
- 마이그레이션이 필요 없다 — 신규 저장·토글부터 자연스럽게 `isPinned`가 채워진다.
- "핀 없음 = false" 불변식 덕에 정렬 분기(`note.isPinned`)를 안전하게 사용한다.
- 태그(`string[]` 임베드)와 동일한 모델·방어 철학이라 코드베이스 일관성이 높다.

**부정 / 트레이드오프**

- 핀을 1급 엔티티로 다루는 기능(전역 핀 목록 집계, 핀 순서 영속화, 핀 그룹)은 이 모델로는 부족하다 → 해당 요구가 생기면 **별도 ADR로 모델을 재검토**한다(예: `pinnedOrder: number`).
- 핀 순서를 사용자가 바꾸는 기능은 불리언만으로 표현 불가 → 드래그 순서는 Non-Goal로 분리(다음 릴리스).

## Alternatives Considered

1. **별도 `pinnedNoteIds: string[]` 목록을 앱 설정/단일 문서로 관리**
   - 장점: 핀 순서를 배열 인덱스로 자연히 표현 가능.
   - 기각 이유: 노트와 별개 자료구조라 정합성(삭제된 노트 ID 잔존 등) 관리 부담. 현재 "순서 없는 이진 고정"에는 과도. YAGNI.
2. **`pinnedAt: string | null`(고정 시각) 필드**
   - 장점: "최근 고정순" 정렬에 유리.
   - 기각 이유: 이번 범위는 원본 순서 유지(재정렬 없음). 시각 정보는 지금 쓰이지 않아 불리언이 더 단순. 정렬 요구가 생기면 그때 도입.
3. **숫자 `pinRank: number`(0=비고정, 1+=순위)**
   - 기각 이유: 순서 변경(Non-Goal)을 위한 표현인데 현재 불필요. 이진 상태에 불리언이 의미상 정확하다.
